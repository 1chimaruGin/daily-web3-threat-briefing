## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 AI-Assisted Decompilation of Unverified Contracts — Chainalysis Research (June 10, 2026)
- **Pattern:** Attackers are using AI-powered decompilation pipelines to reverse-engineer the bytecode of *unverified* smart contracts (contracts whose source is not published on Etherscan/Blockscout). The AI tooling dramatically reduces the time to identify exploitable logic.
- **Confirmed victims:** Truebit ($26.2 M, Jan 8, 2026 — integer overflow in bonding curve, compiled with Solidity v0.5.3 which lacks automatic overflow checks), Aperture Finance, Ekubo, Trusted Volumes
- **Cumulative losses H1 2026:** $36.7 M from contracts with no verified source
- **Where it applies:** Any protocol that deploys periphery/helper contracts without source verification. Even if the core protocol is verified and audited, unverified satellite contracts with admin powers are targeted first.
- **Sources:**
  - [Chainalysis — Attackers Exploiting Unverified Smart Contracts](https://www.chainalysis.com/blog/attackers-exploiting-unverified-smart-contracts/)
  - [CryptoTimes — AI-Assisted Hackers Drain $36.7M](https://www.cryptotimes.io/2026/06/10/ai-assisted-hackers-drain-36-7m-from-hidden-smart-contracts-in-2026/)
  - [Metaverse Post — Chainalysis Warning](https://mpost.io/unverified-smart-contracts-increasingly-targeted-in-36-7m-wave-of-crypto-exploits-chainalysis-warns/)

### 2.2 Arithmetic Overflow in Sui/Move AMM Math — Cetus Protocol ($223 M, May 2026)
- **Pattern:** A flawed overflow check (`checked_shlw`) in the `inter_mate` open-source Move library validated inputs against a 256-bit bound instead of the correct 192-bit bound. The `get_delta_a` liquidity function accepted the malformed value, allowing the attacker to deposit 1 token and receive millions of dollars worth of liquidity.
- **Attack mechanic:** Flash-loan + tightly bounded price range + single-token deposit → AMM mints massively inflated LP position → attacker redeems.
- **Where it applies:** All Move/Sui AMMs that import shared math libraries without independent overflow-boundary audits. Also applicable conceptually to any AMM using fixed-point math where intermediate bit-width assumptions are not validated at *each function boundary*.
- **Sources:**
  - [Cyfrin — Inside The $223M Cetus Exploit (root cause)](https://www.cyfrin.io/blog/inside-the-223m-cetus-exploit-root-cause-and-impact-analysis)
  - [Dedaub — Flawed Overflow Check Analysis](https://dedaub.com/blog/the-cetus-amm-200m-hack-how-a-flawed-overflow-check-led-to-catastrophic-loss/)
  - [The Block — Cetus restarts after recovery](https://www.theblock.co/post/357386/sui-dex-cetus-protocol-restarts-platform-after-recovering-from-223-million-exploit)

### 2.3 Single-Point-of-Failure DVN Configuration in LayerZero Bridges — KelpDAO ($292 M, April 2026)
- **Pattern:** Kelp DAO's cross-chain bridge was configured with a **1-of-1 Decentralized Verifier Network (DVN)** — only one node had to confirm a cross-chain message for funds to be released. Attackers (attributed to Lazarus Group / North Korea) compromised Kelp's internal RPC nodes and simultaneously DDoS'd external fallback nodes, forcing all queries through attacker-controlled infrastructure. The poisoned nodes reported false burn events on the source chain; the single DVN confirmed the message; the destination chain released 116,500 rsETH (~$292 M).
- **Where it applies:** Any LayerZero-connected bridge or omnichain token. The 1-of-1 DVN pattern is the critical anti-pattern. Protocols using default LayerZero configs without explicitly hardening the DVN quorum are at risk.
- **Sources:**
  - [CoinDesk — Kelp DAO exploit breakdown](https://www.coindesk.com/tech/2026/04/19/2026-s-biggest-crypto-exploit-kelp-dao-hit-for-usd292-million-with-wrapped-ether-stranded-across-20-chains)
  - [Chainalysis — KelpDAO Bridge Exploit deep dive](https://www.chainalysis.com/blog/kelpdao-bridge-exploit-april-2026/)
  - [Halborn — Explained: The Kelp DAO Hack](https://www.halborn.com/blog/post/explained-the-kelp-dao-hack-april-2026)
  - [CoinDesk — LayerZero blames Kelp setup, attributes to Lazarus](https://www.coindesk.com/tech/2026/04/20/layerzero-blames-kelp-s-setup-for-usd290-million-exploit-attributes-it-to-north-korea-s-lazarus)

### 2.4 Access Control Remains #1 Vector in 2026 — OWASP Smart Contract Top 10 Update
- **Pattern:** Access control vulnerabilities now account for $953.2 M in 2026 losses vs $35.7 M for reentrancy. The 2026 data reveals a consistent pattern: contracts programmatically correct but missing `onlyOwner` / `onlyRole` guards on privileged state-changing functions (minting, pausing, fund withdrawal).
- **Source:** [OWASP Smart Contract Security Top 10](https://scs.owasp.org/sctop10/SC07-FlashLoanAttacks/)

## 3. 🟢 Advisories / CVEs

### 3.1 Chainalysis Advisory — Unverified Contract Risk (June 10, 2026)
- **Affected:** All EVM protocols deploying contracts without Etherscan/Blockscout source verification, especially those compiled with Solidity <0.8.0 (pre-SafeMath built-in)
- **Severity:** High — AI-assisted attackers have reduced time-to-exploit for unverified contracts significantly
- **Recommendation:** Immediately verify all deployed contracts on block explorers; upgrade compiler to ≥0.8.0; migrate any bonding curves or custom math using old overflow patterns
- **Source:** [Chainalysis Research](https://www.chainalysis.com/blog/attackers-exploiting-unverified-smart-contracts/)

### 3.2 Solidity <0.8.0 Integer Overflow — Truebit Bonding Curve ($26.2 M)
- **Affected library/version:** Solidity ≤ 0.7.x (no built-in overflow revert); specifically Solidity v0.5.3 confirmed in Truebit exploit
- **Severity:** Critical when combined with bonding curves, price math, or liquidity calculations
- **Detail:** Truebit's bonding curve integer overflow (Jan 8, 2026) is the single largest loss in the unverified-contract dataset. The root cause — relying on unchecked arithmetic in an old compiler — is preventable with a compiler bump or explicit `require` bounds.
- **Source:** [blockchain.news — $36.7M from unverified contracts](https://blockchain.news/news/attackers-stole-36m-unverified-defi-contracts)

## 4. 🎯 Hunt Hooks

**Hook 1 — Verify every deployed contract on your target, including periphery.**
Per the June 10 Chainalysis report, AI decompilation makes unverified bytecode *not* a hiding strategy anymore. On any target in scope: run `cast etherscan-source <address>` (or check Blockscout) for every deployed address — including proxy implementations, fee collectors, and helper contracts. Any unverified contract with admin-level functions (mint, withdraw, pause) is an immediate high-priority target for manual bytecode review.

**Hook 2 — Audit LayerZero DVN configuration on any bridge or omnichain token.**
Pull the `oApp.getConfig()` call on the source chain for every LayerZero-connected protocol in your scope. Check the `dvnConfig` field: the `requiredDVNs` array must have ≥2 entries, and each must point to independent infrastructure. A single DVN (`requiredDVNs.length == 1`) or multiple DVNs sharing the same operator key is a critical misconfiguration replicating the Kelp DAO failure condition. Worth filing as critical if the TVL justifies it.

**Hook 3 — Hunt for single-key minting authority behind proxy admins.**
Humanity Protocol's attacker escalated from a stolen dev key to minting 100 M tokens by exploiting a single EOA with proxy admin + minter roles. On any protocol with upgradeable tokens: check `proxyAdmin.owner()` and `token.hasRole(MINTER_ROLE, x)` on-chain. If either resolves to an EOA (not a multi-sig), or if the multi-sig threshold is 1-of-N, that's a critical access control issue. Cross-reference with the 2026 OWASP top finding: access control bugs drove $953 M in losses this year.
