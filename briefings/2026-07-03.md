### 1.1 ResupplyFi — $9.6M | ERC-4626 Vault Inflation Attack + Live Tornado Cash Laundering

| Field | Detail |
|---|---|
| **Protocol** | ResupplyFi (Convex/Yearn-integrated stablecoin lending) |
| **Chain** | Ethereum |
| **Exploit date** | June 25–26, 2026 |
| **$ Lost** | ~$9.6M (wstUSR market, cvcrvUSD vault) |
| **Active now (July 2–3)** | Attacker depositing 1,607 ETH (~$6.5M) into Tornado Cash; full batch ~2,280 ETH (~$9.8M) expected |
| **Bug class** | flash-loan, price-manipulation (ERC-4626 vault inflation / donation attack) |

**Root cause:** ResupplyFi deployed a new `cvcrvUSD` ERC-4626 vault that had been live for approximately **two hours** before the exploit. Freshly-deployed ERC-4626 vaults with near-zero TVL are vulnerable to a "donation attack" that inflates the share price and poisons any lending protocol that uses `convertToAssets()` to value the vault share as collateral:

1. **Inflate share price:** Attacker directly transferred 2,000 crvUSD to the vault address *without minting shares* — a "donation" that immediately inflates `convertToAssets(1 wei)` by orders of magnitude.
2. **Dust collateral deposit:** Minted exactly **1 wei** of vault shares. Thanks to the inflated exchange rate, this registered as ~$10M in apparent collateral value in ResupplyFi's lending engine.
3. **Flash-loan-funded borrow:** Borrowed 10M reUSD (ResupplyFi's native stablecoin) against the phantom collateral.
4. **Exit:** Converted reUSD → ETH → dispersed. Now laundering via Tornado Cash (~$6.5M deposited to date).

The initial flash loan was a modest **$4,000 USDC** from Morpho, swapped to crvUSD on Curve. Both audit firms that reviewed ResupplyFi's code found functionally correct logic — the exploit emerged from **deployment sequencing**, not contract code. The `cvcrvUSD` vault was exploitable during the narrow window between first deployment block and first meaningful TVL.

**Why this is fresh today:** The exploit itself occurred June 25–26. Post-mortems are being published now (Ackee, Veridise, Guardrail all released analyses this week). The Tornado Cash fund movements are happening in real-time (July 2–3).

### 2.2 ERC-4626 Vault Inflation: "Deployment-Window" Exploitability Formalized

Multiple post-mortems published this week (Ackee, Veridise, Guardrail — all linked in §1.1) formalize the ERC-4626 donation attack as a **deployment-window vulnerability class**, not just a theoretical edge case. This deserves its own note:

**Vulnerable preconditions (all must hold simultaneously):**
1. A freshly deployed ERC-4626 vault with near-zero TVL and no initial liquidity lock
2. A lending protocol that accepts this vault's shares as collateral within the same or near-same blocks
3. The vault's `convertToAssets()` (or equivalent) is used as the collateral price oracle without TWAP, TVL floor, or virtual-share offset

**Known mitigations (check whether your targets implement any):**
| Mitigation | Status in ResupplyFi? | Effectiveness |
|---|---|---|
| Virtual shares (`_decimalsOffset > 0`) | Absent | Partial — raises cost of attack but doesn't prevent it |
| Dead-address share burn at deploy | Absent | Effective — locks minimum liquidity permanently |
| Lending protocol: min TVL before accept | Absent | Effective — prevents zero-state exploitation |
| Lending protocol: min time-since-deploy | Absent | Effective — but requires oracle/timing infrastructure |
| Audit of deployment script (not just contracts) | Absent | **Critical gap** — both auditors missed the ordering issue |

This class has now produced confirmed losses of at least **$9.6M (ResupplyFi)** plus historic losses across prior ERC-4626-related incidents. If your target lending protocol accepts ERC-4626 vault shares without these mitigations, it is an active target.

## 4. 🎯 Hunt Hooks

### Hook 1: Check Every Lending Protocol for ERC-4626 Vaults Missing Zero-State Protection
*Tied to ResupplyFi $9.6M (§1.1, §2.2)*

This is the highest-signal finding to run today. The pattern is concrete, repeatable, and confirmed in production.

**Quick triage process:**
1. **List ERC-4626 vaults accepted as collateral** in your target lending protocol — look for `addAsset()`, `setCollateral()`, `registerVault()`, or equivalent calls in recent txs.
2. **Check TVL and age:** Call `totalAssets()` on the vault. If it's low (< $100K) and the vault was deployed recently (check contract creation block), you have a potential target.
3. **Check virtual shares:** Call `convertToAssets(1)` on the vault. If it returns `1` (no offset), virtual share protection is absent.
4. **Simulate the inflation:** Fork the chain, send 1,000 underlying tokens directly to the vault address (no `deposit()` call), then call `convertToAssets(1)` again. If it now returns a large number (the inflated rate), the vault is donation-vulnerable.
5. **Check if the lending oracle uses `convertToAssets()`:** Read the collateral pricing function in the lending protocol. If it calls `convertToAssets()` or `previewRedeem()` without TWAP-wrapping, you have a complete attack path.

**Bug class:** flash-loan, price-manipulation | **Bounty ceiling:** Critical (funds directly at risk)

### Hook 2: Enumerate Proxy Deployment Scripts for Unatomic Admin Role Assignment
*Tied to Kinto Protocol $1.55M (§1.2) and USPD December 2025*

Two confirmed $1M+ incidents now share the same root cause: admin roles not atomically secured at proxy deployment.

**What to look for:**
1. **Target:** Any protocol deployed in the past 90 days using a transparent or UUPS proxy with a governance token. Priority: DeFi protocols on Arbitrum, Base, or Optimism (where gas is cheap enough to make front-running deployment txs viable).
2. **Flag:** Is `_grantRole(DEFAULT_ADMIN_ROLE, ...)` or `initialize(owner_address)` called in a **separate transaction** from the proxy deployment? If yes, there's a window.
3. **Check implementation contract:** Call `initialize()` directly on the logic contract (not the proxy). If it hasn't been called and doesn't revert, you can claim ownership of the logic contract — enabling `delegatecall` poisoning.
4. **Check for Multicall3 deployment:** If the team used Multicall3 to batch deployment + init, that batch is frontrunnable if any step emits a predictable `CREATE2` address that an attacker can race.

**Bug class:** upgradeability, access-control | **Bounty ceiling:** Critical
