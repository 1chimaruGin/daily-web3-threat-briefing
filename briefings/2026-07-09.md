### 2.1 CertiK H1 2026 Report: Wallet Compromise Overtakes Code Exploits as #1 $ Vector

**Classification:** Industry threat report (CertiK Hack3D, released July 6, 2026)
**Relevance window:** H1 2026 (January–June), 344 incidents, $1.31B total losses
**Bug class:** `key-management` (primary), code vulnerability (secondary by count)

**Key findings for bounty hunters:**

| Metric | H1 2026 | Change YoY |
|---|---|---|
| Total losses | $1.31B (344 incidents) | −46.8% vs H1 2025 (Bybit effect); +28% excluding Bybit outlier |
| Wallet compromise losses | **$444.5M across 33 incidents** | #1 vector by $ impact |
| Code vulnerability incidents | 204 / 344 (59%) | #1 by incident count |
| Phishing volume | −50% YoY | Losses down only 11% (4 incidents = 85% of phishing losses) |
| Oldest targeted contracts | Growing share >1 year old | Automated tools revisiting legacy code |

**What this means for hunters:**

The biggest financial payoff vector in H1 2026 was not a Solidity logic bug — it was **key theft**: compromised RPC infrastructure (Kelp DAO $291M), compromised admin wallets (Drift Protocol $285M), and admin key exfiltration (Wasabi Protocol $4.5M, Coindesk April 30). Yet most bug bounties only cover smart contract code. This creates an asymmetry: the highest-impact attack class is often **out of scope** on Immunefi, while medium-severity logic bugs are heavily competed over.

**Tactical implication:** When scoping a new protocol, explicitly read the bounty policy for "admin key compromise" and "off-chain infrastructure." Several programs (Kelp DAO-style) now include RPC/node security in scope. If they do, check:
1. Whether `eth_sendRawTransaction` endpoints are permissioned or public
2. Whether any oracle/price-feed upkeeper key is stored in a hot environment
3. Whether contract upgrade paths rely on a single EOA

### 2.2 Aptos Move VM Type-Confusion Bug Post-Mortem (Hexens / July 4 Disclosure)

**Classification:** Critical VM-level vulnerability — patched February 27, publicly disclosed July 4
**Chain:** Aptos (Move VM)
**Bug class:** `logic-error` (type-confusion / stale cache)
**Systemic risk assessed:** ~$70B (stablecoins, bridges, cross-chain messaging)
**Researcher:** Hexens security firm
**Status:** Patched; no funds lost

**Technical summary:**

Hexens identified a **stale-cache bug** in Aptos's Move virtual machine that produced a **type-confusion condition** — the VM could be tricked into treating one on-chain resource type as another. The exploit allowed attacker-controlled Move code to:
1. Write into storage belonging to **other contracts** (breaking the type-system isolation guarantee)
2. Potentially bypass all type-safety invariants that prevent cross-module storage corruption

The proof-of-concept achieved >90% success rate in simulated network conditions using a commodity server setup (~$3,000). The attack required no insider access or special permissions.

**Why this matters for auditors beyond Aptos:**

The stale-cache type-confusion class appears in multiple Move VM implementations (Aptos, Sui, and movements built on Move language tooling). Auditors reviewing **Move-language contracts** on any chain should examine:
- Any pattern where a **module upgrades a resource type** and older cached representations of that type persist in VM memory
- Cross-module function calls where the **caller's type assumptions** about a resource differ from the **callee's current definition**
- Patterns relying on `move_from<T>` / `borrow_global<T>` across module boundaries after upgrades

**Aptos's position:** Aptos Labs disputes the $70B systemic risk figure, contending that practical exploitability on live mainnet was low. The patch was applied proactively without live exploitation.

## 4. 🎯 Hunt Hooks

### Hook 1: Audit Hot-Key and RPC Infrastructure in Your Target
*Tied to §2.1 CertiK H1 2026 report — wallet compromise = #1 $ loss vector*

The top-grossing attack class in H1 2026 was not a Solidity bug — it was infrastructure key theft. Before diving into contract code for a new target, run this checklist:

```bash
# 1. Check for exposed admin key patterns in any public repo or docs
grep -rE "PRIVATE_KEY|MNEMONIC|DEPLOYER_KEY" .env.example docs/ scripts/
# If found, check the bounty scope — "off-chain infrastructure" key theft
# may be in scope for high-severity reports

# 2. Find all Ownable/AccessControl privileged addresses on-chain
cast call <CONTRACT> "owner()(address)" --rpc-url $RPC
cast call <CONTRACT> "getRoleMember(bytes32,uint256)(address)" \
  $(cast keccak "DEFAULT_ADMIN_ROLE") 0 --rpc-url $RPC
# Then check: are these EOAs or multisigs?
# Single EOA admin on a >$10M TVL protocol = critical finding

# 3. Check oracle/keeper addresses — are they hot wallets?
cast call <ORACLE_CONTRACT> "getKeepers()(address[])" --rpc-url $RPC
```

**What to report:** Single EOA controlling upgrade, pause, or fee functions on a protocol with significant TVL — especially if discoverable via `Ownable.owner()` and traceable to a hot/exchange wallet. Many bounty programs now accept this as a High even without PoC exploitation.

**Bug class:** `key-management`, `access-control` | **Ceiling:** Critical to High

### Hook 2: ERC-4626 Vault Share Price Manipulation via Flash Loan
*Tied to Summer.fi $6M (§1), first disclosed 2026-07-07*

The Summer.fi exploit used a large flash loan to distort the share-price accounting of an ERC-4626 vault. This pattern is generalized and repeatable. Any ERC-4626 vault that:
- Calculates share price using `totalAssets()` that reads from a pool **susceptible to flash loan inflation**, AND
- Does not have an effective per-block deposit/withdrawal limit or TWAP-based share pricing

…is exploitable with this approach.

**In code, look for:**
```solidity
// Red flag: totalAssets() reads live balance manipulable by flash loan
function totalAssets() public view override returns (uint256) {
    return token.balanceOf(address(this));  // ← susceptible if flash-loanable
}

// Red flag: no per-block or per-tx deposit cap
function deposit(uint256 assets, address receiver)
    public override returns (uint256 shares) {
    // No flash-loan guard, no deposit cooldown
    shares = previewDeposit(assets);
    ...
}
```

**Check:** Does the vault use `previewDeposit`/`convertToShares` based on live `balanceOf`? Is there a `maxDeposit` limit? Is there a timelock between deposit and redemption? Missing any two of these on a vault with external yield sources = exploitable.

**Bug class:** `flash-loan`, `logic-error` | **Ceiling:** Critical

### Hook 3: Move VM Cross-Module Resource Type Safety
*Tied to §2.2 Aptos Move VM disclosure*

If you audit Move-language protocols (Aptos, Sui, Initia, or any Move-derivative chain), the Hexens finding points to a class of bugs that may appear in application-layer Move modules — not just the VM itself:

```move
// Red flag: a module upgrade changes a struct definition,
// but callers in other modules still operate on the OLD struct layout
// via cached references
module attacker_module::exploit {
    use victim::resource_module;

    public fun drain_stale(account: &signer) {
        // If resource_module::Resource was recently upgraded
        // and the VM hasn't flushed the cache, this borrow
        // may succeed against the OLD type while operating on
        // the NEW memory layout — causing type confusion
        let res = borrow_global_mut<resource_module::Resource>(
            signer::address_of(account)
        );
        // ... perform privileged operations on mistyped resource
    }
}
```

**Hunt for:** Move contracts that recently upgraded a `struct` definition while keeping the same resource name. Check module version history via the chain's explorer. Any `borrow_global<T>` call across module boundaries post-upgrade is worth manual review.

**Bug class:** `logic-error`, `upgradeability` | **Ceiling:** Critical (chain-level if in core modules)
