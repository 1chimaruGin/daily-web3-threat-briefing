## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 CVE-2026-48558 — SimpleHelp RMM CVSS 10.0 Actively Exploited; Djinn Stealer Targets Crypto Wallets

**Disclosed:** June 30, 2026  
**CVSS:** 10.0 (Critical)  
**CISA KEV added:** June 30, 2026 — federal agency patch deadline **July 2, 2026**  
**Technique class:** access-control (authentication bypass) → credential stealer payload  
**Web3 relevance:** CRITICAL — payload explicitly targets cryptocurrency wallets and web3 dev tooling

CVE-2026-48558 is an **authentication bypass in SimpleHelp RMM's OpenID Connect (OIDC) flow**. A remote, unauthenticated attacker sends a crafted OIDC request that SimpleHelp's session handler treats as a fully authenticated technician session — no credentials required. Once inside, attackers deploy two payloads:

**TaskWeaver** (persistence + lateral movement dropper) and **Djinn Stealer** (cross-platform infostealer, Windows/macOS/Linux).

**Djinn Stealer's crypto/web3 targeting scope:**

| Target category | Specifics |
|---|---|
| Cryptocurrency wallet files | Bitcoin Core, Litecoin, Dogecoin, Dash, Ethereum keystore JSON, Monero, Zcash, Exodus, Atomic Wallet, Electrum |
| Web3 dev credentials | `~/.foundry`, `~/.hardhat`, `.env` with `PRIVATE_KEY`/`MNEMONIC`, `cast` wallet keystores |
| Cloud & infra keys | AWS, GCP, Azure credentials, `~/.ssh` keys |
| AI dev tool tokens | GitHub Copilot, Cursor, Windsurf, Claude API keys (persistent access to code/repos) |
| Package registries | npm tokens (`~/.npmrc`), PyPI tokens |
| Browser credentials | All major browsers (cold wallet extension seeds, MetaMask vault files, WalletConnect sessions) |

**Why SimpleHelp RMM matters in web3:** RMM tools are widely deployed in crypto companies to manage developer machines, node servers, and cloud infra. A single technician session can reach every endpoint in the organization's RMM inventory. Blackpoint Cyber notes: *"The damage outlasts the breached server — a single bypass became a path into cloud platforms, code repositories, AI tools and customer environments, with stolen credentials keeping that access alive after the endpoint is isolated."*

**Attack chain for web3 targets:** `Unauthenticated SimpleHelp session` → `TaskWeaver establishes persistence` → `Djinn Stealer exfils all wallet keystores, .env, hardhat/foundry configs, npm tokens` → `attacker gains signing key material and deployment keys` → `drains protocol treasury at own timing`.

**Affected:** SimpleHelp `< 5.5.8` (all prior versions). Patch: `>= 5.5.8`.

### 2.2 GlassWorm macOS Wave — New Infrastructure, OpenVSX Delivery (Post-Takedown Resurgence)

**Published:** ~June 29–30, 2026  
**Technique class:** supply-chain (trojanized VSCode/OpenVSX extension)  
**Web3 relevance:** HIGH — crypto wallet extensions and Solidity/Rust dev tooling explicitly targeted

CrowdStrike disrupted the GlassWorm botnet on **May 26, 2026** (simultaneous hit on all four C2 channels: Solana memo dead-drops, BitTorrent DHT, Google Calendar titles, and direct VPS). However, security firm Koi Security has now documented **new GlassWorm infrastructure targeting macOS developers** via the OpenVSX marketplace — the primary extension source for open-source VSCode forks (Cursor, VSCodium, Windsurf).

**New wave characteristics:**
- Delivery via trojanized **Solidity language server** and **Hardhat-helper** extensions on OpenVSX
- The worm component self-propagates by injecting into other installed VSCode extensions at runtime (bypasses manual review of extension .vsix files post-install)
- C2 now uses **IPFS content IDs** as dead-drops (replaces the disrupted Solana memo channel)
- Persistence via LaunchAgent plist on macOS; replaces MetaMask keyring decryption key in browser extension storage on first run

**Solana dead-drop replacement — IPFS:** The shift to IPFS C2 is significant: unlike Solana memo fields (pinnable and monitorable), IPFS CIDs can be unpinned but the content remains accessible from any gateway that cached it. Takedown is practically harder.

### 3.2 CVE-2026-48054 — OpenZeppelin Wizard Code Injection (HIGH, Day 20 Unpatched)

| Field | Detail |
|---|---|
| **CVE** | CVE-2026-48054 |
| **Severity** | HIGH |
| **Library** | `@openzeppelin/wizard` (npm) |
| **Root cause** | Code Injection via unsanitized `opts.name` and `opts.uri` in generated Hardhat and Foundry test scaffolds |
| **Patch** | Upgrade to `@openzeppelin/wizard >= <patched version>` |
| **First disclosed** | ~June 11, 2026 (Day 20 unpatched across many CI pipelines) |

Any web3 project that used OZ Wizard to generate test scaffolds with attacker-controllable `name` or `uri` parameters (e.g., dynamic token names, collection URIs from user input) may have injected executable code into their Foundry/Hardhat test harness. This is a CI/CD supply chain risk: if the generated test file is committed and a CI runner executes it, the injected code runs with the runner's cloud credentials.

**Bug bounty angle:** The most interesting targets are protocols that offer "build your own token" UX flows where the wizard parameters are derived from user input.

## 4. 🎯 Hunt Hooks

**Three concrete checks for today's targets:**

### Hook 1: Scan Bug Bounty Targets for SimpleHelp RMM Exposure
*Tied to CVE-2026-48558, CVSS 10.0, CISA KEV July 2 deadline*

1. **Shodan/Censys scan:** Search for `product:"SimpleHelp"` — SimpleHelp RMM servers expose a distinct HTTP fingerprint on port 5950. If your target protocol's infrastructure runs a pre-5.5.8 instance exposed to the internet, this is a confirmed critical finding.
2. **Incident indicator check:** Look for `.djinn` file artifacts, new LaunchAgent/systemd service entries, or unusual `~/.foundry/keystores/` access timestamps on developer machines in your scope.
3. **Scope note:** Under most bug bounty programs, demonstrating that a CVE-10 in a team's RMM could expose `PRIVATE_KEY` from `.env` or Foundry keystores qualifies as a critical infrastructure finding — even if direct exploitation requires MiTM or local access.

**Bug class:** access-control, key-management | **CVSS base:** 10.0 | **Bounty floor:** Critical

---

### Hook 2: OpenVSX / Cursor Extension Audit for GlassWorm Signatures
*Tied to GlassWorm macOS resurgence (post-CrowdStrike takedown)*

For any protocol whose team uses Cursor, VSCodium, or any VSCode fork with OpenVSX as the extension marketplace:

1. **Check installed extensions:** `ls ~/.vscode-oss/extensions/` (Linux) or `~/Library/Application Support/Cursor/User/extensions/` (macOS) for unexpected `solidity-*`, `hardhat-*`, or `foundry-*` packages from unrecognized publishers.
2. **Worm propagation indicator:** Open any installed extension's `dist/extension.js` and grep for `require('child_process')` or `require('net')` — legitimate language server extensions don't make raw TCP connections.
3. **IPFS dead-drop indicator:** Monitor outbound connections from VSCode processes to `*.ipfs.io`, `*.dweb.link`, or gateway.pinata.cloud — legitimate extensions don't phone home to IPFS.

**Bug class:** supply-chain | **Bounty context:** infrastructure/opsec finding; confirm in-scope

---

### Hook 3: OZ Wizard CI/CD Code Injection in "Build Your Own Token" Flows
*Tied to CVE-2026-48054, Day 20 unpatched*

For any DeFi protocol that lets users create tokens, collections, or vaults with custom names/URIs via an OZ Wizard-backed backend:

1. Find the endpoint/function that accepts token `name` or `uri` parameters and passes them to `@openzeppelin/wizard`.
2. Submit a name containing: `` `; require('child_process').execSync('id > /tmp/pwned'); ` `` — if the generated test file is committed to a repo and the CI runner executes it, the command runs.
3. Check if the generated file lands in `test/` or `script/` directories with CI execute permissions. Even if injection doesn't reach production contracts, RCE in CI = access to deployment keys and protocol-level secrets.

**Bug class:** supply-chain | **CVSS base:** HIGH | **Bounty sweet spot:** protocols with user-defined token creation UI and automated test generation
