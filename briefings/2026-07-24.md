## 1. 🔴 Active Exploits (last 24h)

### 1.1 AFX Trade — Arbitrum Bridge Validator Key Compromise — $24.15M | Key-Management / Bridge-DVN | Arbitrum → Ethereum

- **Date:** July 23, 2026 (UTC)
- **Chain:** Arbitrum (bridge) → Ethereum (laundering)
- **Bug class:** `key-management`, `bridge-dvn`
- **Amount lost:** $24,150,000 USDC → swapped to ~12,467 ETH

AFX Trade, an Arbitrum-based perpetual DEX operating its own cross-chain bridge, was drained of all USDC in its bridge vault when an attacker — or attacker group — obtained **five of the bridge's validator signing keys**, accumulating 7,142 out of 10,000 total validator-power units. This cleared the protocol's two-thirds (≥6,667 power units) withdrawal approval threshold.

With the forged quorum assembled, the attacker issued a single authorized withdrawal of 24,150,000 USDC from the Arbitrum-side bridge vault, then bridged those funds to Ethereum and swapped them for approximately 12,467 ETH within minutes. The Arbitrum **native** bridge and rollup infrastructure were not involved — the issue was entirely contained to AFX Trade's self-operated bridge system.

**Attack anatomy:**
1. Exfiltrate or social-engineer 5 validator private keys (exact method undisclosed)
2. Construct withdrawal message signed by those 5 validators (7,142 / 10,000 power — clears 2/3 threshold)
3. Submit multi-sig withdrawal: 24,150,000 USDC exits Arbitrum vault to attacker's Ethereum address
4. Swap USDC → ETH (~12,467 ETH) on Ethereum DEX aggregators
5. Funds remain in attacker wallet; no mixer use detected as of this briefing

**Response:** AFX Trade paused bridge operations immediately after detection. The head of growth published a public on-chain and social message offering the attacker a **30% white-hat bounty** (≈$7.2M) to return the remaining 70%. No response from the attacker as of July 24 UTC.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Third-Party Bridge Validator Key Concentration — The "Quorum-Clearance" Attack

**Pattern surfaced:** July 23, 2026 (AFX Trade $24.15M)
**Bug class:** `key-management`, `bridge-dvn`
**Chains affected:** Any EVM chain using a permissioned multi-validator bridge architecture

The AFX Trade exploit introduces a new variant to the now-familiar "bridge key compromise" kill chain. Unlike the **Wanchain** case (Jul 22, where a single signature was *replayed* via encoding ambiguity) or the **Altura** case (Jul 4, where a DVN verifier was misconfigured), the AFX Trade attacker targeted **voting-power aggregation**: they didn't need to control all validators, only enough to clear the protocol's withdrawal quorum.

**Why this matters architecturally:**

Most bridge protocols use a weighted quorum system (e.g., ≥51% or ≥67% of total voting power) to authorize withdrawals. If validator keys are held by a small number of hot wallets — especially if some validators share infrastructure (co-hosted, same cloud provider, same DevOps team) — the practical attack surface is much smaller than the nominal validator count implies.

The AFX Trade bridge had 10,000 power units across its validators, but the attacker only needed to compromise keys totaling 6,668 power units. With 5 keys and 7,142 power units, the attacker had a comfortable margin above threshold.

**Structural red flags hunters should look for in any bridge protocol:**
- Total validator count < 10, with hot validator keys (vs. HSM/MPC)
- Unequal power distribution where a minority can clear quorum (e.g., 3 "super-validators" holding 70% of power)
- Bridge operator also controls key rotation — no independent oversight
- No on-chain circuit breaker pausing large withdrawals for a time-delay

**Disclosed:** July 23, 2026 (CertiK Report — H1 2026)
**Bug class:** `key-management`
**Chains affected:** All — asset-holder level, not protocol-level

CertiK's H1 2026 threat report published July 23 tracks **52 verified physical coercion incidents** (commonly called "$5 wrench attacks") in the first six months of 2026, a 33% increase year-over-year, with $124M in total exposure. These are attacks where criminals physically coerce holders to authorize on-chain transfers — no smart contract bug required.

**Relevance for hunters:** While this isn't a protocol vulnerability, it underscores a growing pattern that sophisticated on-chain attackers are using *off-chain* coercion to bypass smart contract security. If you're hunting for vulnerabilities in custodial bridges or institutional staking protocols, consider whether the protocol's key signing process has any protection against insider coercion (duress signals, time-delayed withdrawals, geo-fenced signing, secondary approval paths). These are valid report-worthy gaps in protocol design even without a cryptographic exploit.

## 4. 🎯 Hunt Hooks

### Hook 1: Validator Voting-Power Distribution in Permissioned Bridges
*Tied to §1.1 — AFX Trade $24.15M, quorum cleared with 5 compromised keys*

Before hunting for Solidity bugs in bridge contracts, map the validator/DVN architecture. Many cross-chain bridge contracts look secure on paper (proper access control, multi-sig withdrawal functions) but are operationally broken if a subset of hot validator keys can clear the withdrawal threshold unilaterally.

**What to check:**

```solidity
// Look for the validator power registry and withdrawal quorum check
mapping(address => uint256) public validatorPower;
uint256 public totalPower;
uint256 public constant QUORUM_BPS = 6667; // 66.67%

function isQuorumMet(address[] calldata signers) public view returns (bool) {
    uint256 signedPower;
    for (uint i = 0; i < signers.length; i++) {
        signedPower += validatorPower[signers[i]];
    }
    // ⚠️ If 3-5 validators control >6667 BPS of total power,
    // compromising those 3-5 keys = full bridge control
    return signedPower * 10000 / totalPower >= QUORUM_BPS;
}
```

**Check on-chain:**
- Run `validatorPower(addr)` for each validator; compute cumulative power of the smallest subset that clears quorum
- If `min_colluding_validators < 5`, it's a key management risk worth reporting
- If any validator address is an EOA (not a multisig), that's an additional escalation factor

**Report condition:** Smallest validator subset clearing quorum ≤ 3 EOA validators → High/Critical depending on TVL.

### Hook 2: Bridge Withdrawal Time-Lock and Circuit Breakers
*Tied to §1.1 AFX Trade — full $24.15M withdrawn in a single transaction with no delay*

The AFX Trade bridge allowed a single-transaction USDC withdrawal of $24.15M with no cooling-off period, no withdrawal cap, and no anomaly detection. This is the simplest possible circuit-breaker gap to report.

**What to check:**

```solidity
// Does the bridge have a withdrawal delay for large amounts?
function initiateWithdrawal(uint256 amount, address recipient) external {
    require(isValidatorQuorumMet(msg.sender), "insufficient quorum");
    
    // ✅ GOOD: time-lock large withdrawals
    if (amount > LARGE_WITHDRAWAL_THRESHOLD) {
        pendingWithdrawals[withdrawalNonce] = PendingWithdrawal({
            amount: amount,
            recipient: recipient,
            unlockTime: block.timestamp + WITHDRAWAL_DELAY // e.g. 24h
        });
        emit WithdrawalQueued(withdrawalNonce++, amount, recipient);
    } else {
        // Small withdrawals can be instant
        _executeWithdrawal(amount, recipient);
    }
    
    // ❌ VULNERABLE (AFX Trade pattern): no delay, no threshold check
    // _executeWithdrawal(amount, recipient); // entire vault drained in 1 tx
}
```

**Ask in any bridge report:** Is there a per-transaction withdrawal cap? Is there a daily outflow limit? Is there a time-delay on large withdrawals that would allow the protocol to pause the bridge? If all three are missing on a bridge with >$1M TVL → Medium/High.

### Hook 3: Validator Key Rotation Controls — Who Can Add/Remove Validators?
*Tied to §2.1 — the "quorum-clearance" pattern requires the attacker to know which keys control enough power*

Even if today's validator set is appropriately distributed, look at who controls validator addition and removal. If the bridge owner can add a malicious validator with 67% power unilaterally, the initial distribution is irrelevant.

**What to check:**

```solidity
// RED FLAG: single owner can add validators with arbitrary power
function addValidator(address validator, uint256 power) external onlyOwner {
    validatorPower[validator] = power;
    totalPower += power;
    // ⚠️ Owner could add attacker address with >67% of totalPower
    // in a single tx — instant quorum clearance without needing existing keys
}

// BETTER: validator changes require existing quorum approval + time-lock
function proposeValidatorChange(
    address validator, 
    uint256 newPower
) external {
    require(isValidatorQuorumMet(msg.sender), "need quorum to change validator set");
    // Queue change with 48h time-lock
    validatorChanges[changeNonce++] = ValidatorChange({
        validator: validator,
        newPower: newPower,
        executableAfter: block.timestamp + 48 hours
    });
}
```

**Report condition:** Bridge owner can add/remove validators or change power unilaterally without quorum approval or time-lock → Critical (owner key compromise or rug = instant full bridge drain).
