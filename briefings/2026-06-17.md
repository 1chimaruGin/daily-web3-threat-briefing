## 1. 🔴 Active Exploits (last 24–48h)

### 1.1 Aztec Connect — $2.19M | ZK Proof Settlement Boundary Bypass | Ethereum *(June 14, 2026 — catch-up)*

- **Chain:** Ethereum (Mainnet — deprecated `RollupProcessorV3`)
- **Lost:** ~$2.19M (909 ETH, 270K DAI, 167 wstETH + other tokens across 7 attacker transactions)
- **Root Cause:** Structural gap between the ZK-circuit's commitment scope and the L1 settlement loop's traversal range.
  - `computeRootHashes()` only validated the **start** of the submitted proof blob. The remaining 31/32 public-input slots were committed into the L2 state root by the ZK proof but **never traversed or enforced** by the L1 contract's settlement logic.
  - Concretely: the loop ran over `numRealTxs` while the ZK circuit committed to `_numTxs` — any public inputs in the gap between those two values were accepted on-chain without verification.
  - Attacker embedded seven malicious withdrawal entries in the unverified slots; the contract credited unbacked balances and released real funds.
- **Why unpatched:** Protocol deprecated March 2023; admin keys renounced and contracts immutable. No upgrade path existed.
- **Bug class:** `logic-error` (ZK settlement boundary mismatch)
- **Attribution / Response:** CertiK flagged on-chain; BlockSec published `numRealTxs/_numTxs` analysis; SlowMist released full L1/L2 state-discrepancy write-up. Aztec Foundation confirmed current Aztec Network and AZTEC ERC-20 are unaffected.
- **Sources:** [SlowMist analysis](https://slowmist.medium.com/analysis-of-the-2-19-million-asset-theft-from-aztec-connect-d867c59b1fc6) · [CoinInsider](https://www.coininsider.com/news/attacker-drains-2-1-million-from-deprecated-aztec-connect-in-proof-verification-exploit/) · [Aztec Foundation tweet](https://x.com/aztecFND/status/2066175938887619055) · [CryptoBriefing](https://cryptobriefing.com/aztec-connect-exploit-deprecated-contract/)

### 1.2 Syscoin Bridge — ~$8M | SPV Proof Forgery / NEVM–UTXO Burn-Mint Bypass | Syscoin *(June 7–8, 2026 — catch-up)*

- **Chain:** Syscoin (NEVM ↔ UTXO native bridge)
- **Lost:** ~$8M equivalent (5 billion SYS minted on UTXO side without a corresponding NEVM burn)
- **Root Cause:** The bridge relay's SPV-proof parsing code failed to reject a deliberately malformed proof.
  - Normal flow: user calls `freezeBurnERC20` on the `SyscoinERC20Manager` contract (NEVM side) → SPV proof of burn is submitted to the UTXO relay → UTXO-side SYS is unlocked.
  - Attack: attacker submitted a fake SPV proof crafted to *look* structurally valid to the parser without a corresponding on-chain burn call ever being made. The relay accepted it, minting ~5B SYS on the UTXO side.
  - No NEVM burn event existed for the proof; the relay's verification was incomplete for malformed inputs.
- **Response:** Bridge paused immediately; exchanges requested to freeze/blacklist tainted SYS trail.
- **Bug class:** `bridge-dvn` (SPV proof parsing flaw), `logic-error`
- **Sources:** [Halborn explained](https://www.halborn.com/blog/post/explained-the-syscoin-bridge-hack-june-2026) · [CryptoTimes](https://www.cryptotimes.io/2026/06/08/syscoin-halts-bridge-after-exploit-mints-5-billion-sys-tokens/) · [PricePredictions](https://pricepredictions.com/news/syscoin-bridge-exploit-5-billion-sys-tokens-minted-proof-validation-flaw-br7jyfk9) · [Rekt.news](https://rekt.news/syscoin-rekt)

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Aztec Connect — ZK Public-Input Slot Coverage Pattern

The SlowMist and BlockSec write-ups formalize a new attack pattern to watch:

**"ZK public-input under-coverage"** — A ZK circuit commits to `N` public inputs; the on-chain verifier or settlement loop only iterates over `M < N`. Any inputs in the gap [M, N) are proven but never enforced at settlement time, enabling state forgery on exactly those slots.

**Where this applies:** Any ZK-rollup or ZK-bridge where:
1. The ZK circuit and L1 settlement code have separate, independently-maintained loop bounds.
2. The settlement contract does not assert `numSettled == proofCommittedCount` before releasing funds.

Protocols known to have similar dual-path architectures: Polygon Hermez, older Loopring rings, early-version Scroll batches (all have since been audited — but **forks and derivatives** of older ZK stack code may still carry this). Worth checking in any ZK circuit codebase where `numTxs` or equivalent appears in both a Solidity loop and as a public input.

- **Source:** [SlowMist analysis](https://slowmist.medium.com/analysis-of-the-2-19-million-asset-theft-from-aztec-connect-d867c59b1fc6) · [DEV.to breakdown](https://dev.to/cryip/how-a-single-validation-mismatch-can-drain-millions-lessons-from-the-aztec-connect-exploit-2598)

### 2.2 Gnosis Pay Zodiac Delay Module — Missing Return-Value Check in Static Call *(June 1, 2026 — catch-up)*

- **Lost:** ~$265K (EURe + GNO drained from dozens of Safes)
- **Root Cause:** A missing **status check on a static call** in the Zodiac Delay Module. The module accepted the *return data* of the call regardless of whether the call succeeded or reverted, allowing an attacker to pass off crafted revert messages as valid approved signatures.
  - The exact bug had been **silently fixed** in the development repository before the exploit, but the older deployed contract was not upgraded and users were not warned.
- **Lesson:** "Fix it in the new version" is not disclosure. Bug hunters should look for this pattern on protocols using Safe/Zodiac modules: search for `staticcall` invocations where the `success` boolean is ignored or not validated.
- **Response:** GnosisPay co-founder Martin Köppelmann committed to covering all user losses.
- **Bug class:** `logic-error` (missing return-value check)
- **Sources:** [Gnosis Pay Hack Explained (Bleap)](https://www.bleap.finance/en-us/blog/gnosis-pay-hack-explained-jun-2026) · [Verichains write-up](https://blog.verichains.io/p/gnosis-pay-exploit-the-devs-discovered) · [CryptoTimes](https://www.cryptotimes.io/2026/06/05/delay-module-trick-costs-gnosispay-265k-reports-certik/) · [TheBlock](https://www.theblock.co/post/403147/gnosis-will-cover-all-user-losses-amid-exploit-related-to-gnosis-pay-co-founder-koppelmann-says)

## 4. 🎯 Hunt Hooks

**Hook A — ZK circuit / L1 settlement count mismatch (from Aztec Connect)**
Search any ZK-rollup or ZK-bridge codebase for places where the ZK proof commits to a count variable (e.g., `numTxs`, `batchSize`, `txCount`) AND the L1 Solidity settlement loop uses a *different* variable to bound its iteration. If `verifier.verify(proof, publicInputs)` passes but `settlement_loop(0..numRealTxs)` uses a smaller bound, funds in the gap are provably but unenforcedly committed — exact root cause of this $2.1M drain. Target: any fork of Aztec Noir stack, legacy Hermez v1, any bridge with a dual-count architecture.

**Hook B — Safe/Zodiac module staticcall return-value checks (from Gnosis Pay)**
In contracts that use Gnosis Safe + Zodiac modules: grep for `staticcall` (or `delegatecall` used for read-only checks) where the `bool success` return value is not explicitly validated with `require(success, ...)`. The Gnosis Pay attacker passed forge error-data as valid approval because the check was missing. Scope: any protocol using `DelayModule`, `RolesModule`, `ScopeGuard`, or custom Zodiac-based guards.

**Hook C — Bridge relay SPV/proof acceptance without on-chain event cross-check (from Syscoin Bridge)**
For any lock-mint or burn-unlock bridge that relies on an off-chain relay submitting proofs: verify that the relay **independently reads on-chain state** (a burn event, a lock receipt, a Merkle inclusion proof tied to a confirmed tx) before minting. If the relay validates the *format* of a proof without confirming the *underlying on-chain event actually occurred*, it is vulnerable to the exact Syscoin attack pattern. Target: any Cosmos IBC-adjacent relay, UTXO-EVM bridges, any bridge where the relay runs custom SPV parsing code rather than using a well-audited library.
