## 1. 🔴 Active Exploits (last 24h)

### 🆕 GMX V1 GLP Pool — $42M Reentrancy + AUM Accounting Desync (Arbitrum)

| Field | Detail |
|---|---|
| **Protocol** | GMX V1 — GLP liquidity pool |
| **Chain** | Arbitrum (GLP V1); Avalanche trading halted as precaution |
| **Date** | July 9, 2026 |
| **Amount lost** | ~$42M in GLP pool tokens |
| **Amount recovered** | ~$37M — attacker accepted 10% white-hat bounty (~$5M); remainder to GMX Security Multisig |
| **Bug class** | `reentrancy` + `logic-error` (AUM/share-price desync) |
| **Status** | Trading, minting, and GLP redemption **halted** on Arbitrum and Avalanche. $44M compensation plan finalized for GLP holders. |

**Attack mechanism:**

The exploit combined two weaknesses in GMX V1's `PositionManager`:

1. **Keeper-controlled reentrancy.** The attacker submitted a crafted decrease-order transaction where the `_callbackTarget` parameter pointed to an **attacker-controlled malicious contract** rather than a normal EOA. When the keeper executed `executeDecreaseOrder()`, it triggered the callback — re-entering the vault mid-execution before position state was finalized.

2. **GLP AUM accounting desync.** While total short open interest was updated on position open, the **average short price** (used in AUM calculation) remained stale at historical lows. The attacker exploited this by: (a) minting GLP, (b) opening short positions while prices were high, (c) the AUM calculation used the stale historical price — inflating the calculated AUM and therefore GLP value, (d) redeeming GLP at the inflated rate. The reentrancy in step (b) let the attacker chain these within a single transaction before the state was corrected.

PeckShield confirmed ~$9.6M of the initial drain was bridged to Ethereum before the negotiation began. GMX offered and paid a $5M white-hat bounty; remaining funds moved to the GMX Security Multisig.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Keeper-Callback Reentrancy Pattern (GMX Post-Mortem Class)

**Classification:** Newly confirmed exploitation of an established pattern
**Bug class:** `reentrancy`
**Affected pattern:** Any perpetual DEX / options protocol that allows **user-supplied callback addresses** passed through to a trusted keeper's execution loop

The GMX exploit reveals an underappreciated reentrancy surface: **keeper-mediated callback execution**. In many perpetual DEX designs, order keepers call into the protocol and pass a `callbackTarget` or `receiver` that was specified by the original order creator. If:
- The protocol does not validate that the callback target is a plain EOA (not a contract), **and**
- The keeper executes the callback **before** finalizing position state or updating accounting,

then an attacker can re-enter the vault from the callback and exploit any transient state inconsistency.

This pattern is distinct from the classic "check-effects-interactions" reentrancy — here the **caller is a trusted keeper**, not the victim contract calling out to an untrusted token. Standard reentrancy guards that operate at the vault level may not protect keeper callback paths if the guard is not inherited/applied to the callback execution step.

**Where to look:** GMX V2, Synthetix Perps (v2/v3 callback flow), dYdX (callback-on-fill orders), and any bespoke keeper system that executes order-creator-supplied addresses. Check whether `ReentrancyGuard` is applied to **both** the keeper entry-point and any callback step.

**Source:** Analysis derived from [QuillAudits GMX V1 hack analysis](https://www.quillaudits.com/blog/hack-analysis/how-gmx-lost-42m) and [Sherlock GMX hack explained](https://sherlock.xyz/post/gmx-exchange-hack-explained)

## 4. 🎯 Hunt Hooks

### Hook 1: Audit Keeper-Executed Callback / Receiver Addresses for Reentrancy
*Tied to §1 GMX V1 $42M exploit*

The GMX attack is a template for a class of perpetual-DEX / structured-product bugs. Go find it in your targets today:

```solidity
// Red flag pattern: keeper executes an address supplied by the order creator
function executeDecreaseOrder(
    address account,
    uint256 orderIndex,
    address payable feeReceiver  // ← user-controlled address
) external onlyOrderKeeper {
    // ... read order from storage ...

    // ❌ Danger: callback/transfer to user-supplied address BEFORE
    //    position state is fully written
    feeReceiver.call{value: executionFee}("");
    // ← At this point, if feeReceiver is a contract, it re-enters here
    //    and position accounting is still in a transient mid-state

    // state finalized only AFTER the call above
    _updatePositions(...);
}
```

**Check for:**
1. Any function where a keeper passes `address` parameters sourced from user-created orders
2. Verify `ReentrancyGuard` is applied to **the callback path**, not just the keeper entry point
3. Confirm that all accounting writes (position size, collateral, AUM) happen **before** any external call/transfer

**Bug class:** `reentrancy` | **Ceiling:** Critical

### Hook 2: GLP / LP Share AUM Staleness — Short Position Accounting Desync
*Tied to §1 GMX V1 $42M exploit*

The second component of the GMX attack is a logic flaw in liquidity pool AUM calculation: **average short entry price remains stale while total short size updates**. This inflates the perceived AUM and therefore the LP token redemption value. Generalized pattern:

```solidity
// In any pool that tracks "virtual P&L" of short positions:
// totalShortSize is updated on each new position
// BUT averageShortPrice may not update proportionally

uint256 profitLoss = getShortProfitLoss(
    totalShortSize,
    averageShortPrice,  // ← if this lags behind reality, AUM is wrong
    currentPrice
);
uint256 aum = poolBalance + profitLoss;
uint256 glpPrice = aum / totalGlpSupply;  // ← manipulable through stale averageShortPrice
```

**Hunt in:** Any protocol with a synthetic index token (GLP-style pooled LP), options vaults that track aggregate delta, or lending protocols that compute "virtual collateral value" using tracked short/hedge positions. Ask: *when a position is opened, do both `totalSize` and `averagePrice` update atomically and correctly?*

**Bug class:** `logic-error`, `price-manipulation` | **Ceiling:** Critical

### Hook 3: DAO Governance Quorum Threshold Analysis
*Tied to BonkDAO $20M attack (ongoing from 2026-07-08)*

The BonkDAO attack required only ~$4M in vote-buying to drain $20M, because the quorum threshold was 1% of circulating supply and vote delegation was not timelocked. For any DAO/governance protocol in your target list:

```bash
# 1. Find the quorum parameter
cast call <GOVERNOR> "quorumVotes()(uint256)" --rpc-url $RPC
# Compare to circulating supply — anything <5% of supply is a red flag

# 2. Check for proposal execution timelock
cast call <GOVERNOR> "timelockDelay()(uint256)" --rpc-url $RPC
# Zero or <48h = governance proposals can execute faster than community can react

# 3. Check if flash-loan vote buying is possible
#    (can voting power be acquired and used within the same block as the proposal execution?)
cast call <TOKEN> "getPastVotes(address,uint256)(uint256)" \
    <ATTACKER> <SNAPSHOT_BLOCK> --rpc-url $RPC
# If snapshot = block.number (not block.number - 1), flash voting is possible
```

**What to report:** A governance system where an attacker could accumulate quorum-threshold voting power from open markets, DEX liquidity, or flash loans at a cost well below the treasury value — especially where execution timelock is < 48h. Many programs list this as Medium-Critical depending on treasury size.

**Bug class:** `access-control`, `logic-error` | **Ceiling:** Critical
