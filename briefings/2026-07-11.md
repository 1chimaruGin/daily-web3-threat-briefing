## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 ERC-4626 Vault Donation-Inflation Attack: Summer.fi Anatomy

**Classification:** Confirmed exploitation of a known attack class, newly detailed
**Bug class:** `flash-loan`, `logic-error`
**Affected pattern:** Any ERC-4626 (or ERC-4626-compatible) vault that passes through to an underlying strategy contract whose `totalAssets()` can be inflated via direct token donation before a share redemption

The Summer.fi Fleet Commander exploit is now sufficiently documented to extract a precise attack anatomy. This is worth understanding for anyone auditing yield aggregator vaults:

**Attack sequence (generalized from Summer.fi):**

1. **Pre-position:** Accumulate shares in a target vault by depositing normally over multiple blocks (avoids detection via single large deposit).
2. **Flash loan large liquidity** (~$65.4M USDC in this case) to execute the attack in a single transaction.
3. **Inflate the underlying strategy's `totalAssets()`** by donating tokens *directly* to the strategy contract (the "Ark"). This is not a deposit — no new shares are minted. The strategy's reported asset balance rises without a corresponding share mint.
4. **Trigger the vault's `rebalance()` or equivalent function** that reads from the inflated strategy balance. Fleet Commander re-reads `totalAssets()` at this point and now believes the vault pool is larger than it is.
5. **Redeem shares at the inflated rate.** The share price (computed as `totalAssets / totalSupply`) is now artificially elevated. The attacker redeems their pre-accumulated shares, extracting more underlying tokens than they deposited.
6. Repay flash loan and keep the delta (~$6M).

**The critical flaw:** Fleet Commander's accounting trusted the live `totalAssets()` call at redemption time without a time-weighted or committed value. Any protocol where `totalAssets()` can be manipulated between deposit and redeem — and share price is computed at redemption time from that value — is vulnerable.

**Where to look today:**
- Yearn V3 strategy vaults with external `totalAssets()` reads from third-party Arks/strategies
- Morpho Blue / Euler ERC-4626 wrapper vaults
- Beefy vaults with external strategy adapters
- Any vault that aggregates yield from sub-strategies via a live `totalAssets()` call at redemption

**Key check:** Does the vault use a *snapshot* or *committed* totalAssets at the time of deposit (and use that same value at redemption), or does it re-query live at redemption? The latter is the vulnerability.

### 2.2 Governance Quorum Gaming — BonkDAO Post-Mortem Pattern

**Classification:** Confirmed exploitation of governance parameter weakness
**Bug class:** `access-control`, `logic-error`

The BonkDAO attack ($20M, July 7) has been attributed to a governance quorum set at 1% of circulating supply with no vote-purchase timelock. The attacker spent ~$4.4M acquiring sufficient voting power and submitted a malicious proposal that passed with 7 wallets voting (99.9% yes), draining $20M in BONK.

The key vectors:
- **No minimum voting delay** between proposal submission and vote: the attacker controlled both the proposal creation and the winning vote in coordinated fashion.
- **No timelock before execution**: the proposal executed within the same epoch it passed.
- **Token-weighted governance with no cap on single-wallet voting power**: $4.4M of BONK = enough to clear 1% quorum threshold unilaterally.

This is the *exact* pattern to look for in any DAO your target protocol has. Check three parameters:
1. Quorum threshold as % of circulating supply (< 3% is a red flag)
2. Voting delay (blocks between proposal creation and vote start — zero allows manipulation)
3. Execution timelock (zero or < 48h = treasury drainable in hours)

## 3. 🟢 Advisories / CVEs

### CVE-2026-48054 — OpenZeppelin Contracts Wizard: Code Injection (Day 28) [HIGH — CVSS 8.8]

**Package:** `@openzeppelin/wizard` (NPM)
**Vulnerability:** Unsanitized interpolation of user-supplied `opts.name` and `opts.uri` parameters into generated Hardhat/Foundry test source code. An attacker who controls these values (e.g., via a web form or CI pipeline that invokes the wizard) can inject arbitrary JavaScript/Shell into the generated test files, achieving **arbitrary code execution on the developer's machine** at test-run time.
**Impact:** Source code exfiltration, `.env` credential theft, Foundry keystore compromise, CI runner pivoting.
**Fix:** Upgrade to `@openzeppelin/wizard >= [patched version]`. The hosted `wizard.openzeppelin.com` is already patched; the risk is **local/self-hosted invocations** of the NPM package.
**Status:** Now in its 28th day post-disclosure. Many downstream toolchains have not yet pinned the patched version.

**Action for hunters:** If any target protocol uses wizard-generated test files and the wizard invocation accepts user-supplied token names, this is a direct code execution path in their CI/CD pipeline. This is a bug bounty-eligible supply chain issue on Immunefi programs that cover developer tooling.

**Source:** [CVE-2026-48054 — DailyCVE](https://dailycve.com/openzeppelin-contracts-wizard-code-injection-cve-2026-48054-high-dc-jun2026-487/)

## 4. 🎯 Hunt Hooks

### Hook 1: ERC-4626 Vault — Can `totalAssets()` Be Inflated Via Direct Donation?
*Tied to §2.1 Summer.fi Fleet Commander anatomy*

The most directly actionable finding from the Summer.fi post-mortem. Check any ERC-4626 vault in your target list for this pattern:

```solidity
// In the vault's share price / redemption logic:
function previewRedeem(uint256 shares) public view returns (uint256) {
    // ❌ Red flag: totalAssets() is queried LIVE at redemption time
    //    and reflects the current balance of underlying strategy contracts.
    //    If any strategy balance can be inflated without minting new shares,
    //    this calculation can be manipulated.
    return shares.mulDivDown(totalAssets(), totalSupply);
}

function totalAssets() public view returns (uint256) {
    uint256 total = 0;
    for (uint i = 0; i < strategies.length; i++) {
        // ← If this reads a live token balance (e.g., IERC20.balanceOf()),
        //   a direct donation to the strategy contract before this call
        //   inflates the result without minting new shares.
        total += IStrategy(strategies[i]).totalAssets();
    }
    return total;
}
```

**Test for exploitability:**
1. Find all strategy/Ark addresses underlying the vault
2. Check if direct ERC20 transfer to a strategy (without calling `deposit()`) increases `totalAssets()` return value
3. Check if this increase flows through to `previewRedeem()` / share price
4. If yes: calculate potential profit for a flash-loan funded donation given current vault TVL

**Bug class:** `flash-loan`, `logic-error` | **Ceiling:** Critical | **Bounty potential:** $500K–$2M on large yield aggregators

### Hook 2: Governance — Validate Three Parameters Before Calling a DAO Secure
*Tied to §2.2 BonkDAO $20M*

Run this diagnostic on any DAO treasury in your target scope before saying "governance is fine":

```bash
# 1. Check quorum threshold
cast call <GOVERNOR> "quorumNumerator()(uint256)" --rpc-url $RPC
# If quorumNumerator / quorumDenominator < 0.03 (3%), flag it.

# 2. Check voting delay (blocks between proposal creation and vote-start)
cast call <GOVERNOR> "votingDelay()(uint256)" --rpc-url $RPC
# Zero delay: proposal creator can vote in the same block they create.

# 3. Check execution timelock delay
cast call <TIMELOCK> "getMinDelay()(uint256)" --rpc-url $RPC
# < 172800 (48h in seconds) = treasury can be drained before community reacts.

# 4. Flash-vote check: can an attacker buy quorum from DEX + vote in same tx?
cast call <TOKEN> "clock()(uint48)" --rpc-url $RPC
# If clock() uses block.number (not a timestamp), compare to snapshot block.
# If proposalSnapshot(id) == block.number (not block.number - 1), flash-loan
# voting may be possible.
```

**Report condition:** An attacker can acquire quorum-threshold voting power from DEX liquidity at a cost-to-drain ratio < 10:1 (spend $X to drain 10X treasury) **and** execution timelock < 48h. This is commonly Medium–Critical depending on treasury value.

**Bug class:** `access-control`, `logic-error` | **Ceiling:** Critical
