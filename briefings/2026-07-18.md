### 1.2 Across Protocol — Solana Bridge Relayer Exploit (July 17)

**Date:** July 17, 2026, ~05:30 UTC
**Chain:** Solana (funds bridged to EVM — Solana + EVM addresses flagged by SEAL 911)
**Amount:** Not yet publicly disclosed (only Risk Labs relayer capital at risk; user bridge transactions completed)
**Bug class:** `bridge-dvn` / possible `logic-error` (event spoofing / deposit verification flaw in relayer)
**Status:** Solana deposits disabled. Investigation ongoing with SEAL 911 whitehat group. Post-mortem pending.

**Attack mechanism (preliminary):**
Across Protocol uses a **relayer-fronts-capital** architecture: relayers pre-fund user withdrawals, and the bridge's hub reimburses them. On Solana, because there is no canonical event system and events are reconstructed from transaction traces, a class of bug could allow an attacker to **spoof deposit events** — tricking a relayer into filling orders with no real deposit behind them. Whether today's attack is exactly this, a relayer infrastructure key compromise, or another variant is unconfirmed pending post-mortem.

**Containment:** Risk Labs' own capital absorbed the loss; no user funds affected. Cross-chain fund movement to EVM chains already detected by SEAL 911.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Backend Authorization Bypass — "Keyless Drain" Pattern (CoinDCX + BigONE)

**Bug class:** `key-management` / `supply-chain`
**Sources:** [CCN — CoinDCX Keyless Drain Analysis](https://www.ccn.com/education/crypto/coindcx-hack-44m-india-crypto-security-crisis-explained/) | [Crypto Economy — BigONE Analysis](https://crypto-economy.com/bigone-hacked-27m-stolen-in-devastating-hot-wallet-attack/) | [Ampcus Cyber — CoinDCX](https://www.ampcuscyber.com/shadowopsintel/coindcx-suffers-usd-44-million-hack/)

Two major incidents this week (CoinDCX $44M, BigONE $27M) share a critical pattern: **funds were drained without the attacker ever obtaining wallet private keys.** This "keyless drain" architecture flaw is becoming the dominant vector against centralized and semi-centralized crypto platforms:

| Variant | Mechanism | Defense Gap |
|---|---|---|
| **Backend session theft** (CoinDCX) | Malware on employee laptop → session/API keys harvested → withdraw API authorized | MFA on withdrawal APIs; device-bound auth tokens |
| **Server-side logic compromise** (BigONE) | Third-party supply chain → backend risk-control bypass → unlimited withdrawal authorized | Third-party supply chain isolation; server-side withdrawal rate-limits independent of risk logic |

Both exchanges had "safe" private keys. Both lost tens of millions. The **authorization layer between "user requests withdrawal" and "private key signs transaction" is the new primary attack surface for exchange hacks** — not the key vault itself.

### 2.2 Bridge Relayer Event Spoofing — Across Protocol Solana Pattern

**Bug class:** `bridge-dvn`
**Sources:** [CryptoTimes — Across Attack](https://www.cryptotimes.io/2026/07/17/across-protocol-reports-first-attack-on-solana-after-34b-in-bridge-volume/) | [Helius — Solana Hacks History](https://www.helius.dev/blog/solana-hacks)

Across Protocol's Solana incident highlights a structural risk in cross-chain bridges porting EVM-native patterns to Solana:

**The core problem:** EVM bridges often verify user deposits via on-chain event logs — a canonical, tamper-evident data structure. Solana **has no canonical event system**; events are reconstructed from transaction trace data, and crucially, **failed transactions still emit trace data**. A bridge relayer that verifies deposits by listening for "deposit events" without confirming the transaction succeeded could be tricked into filling orders against failed or spoofed deposits.

**Pattern to check in bridge codebases:**
1. Does the relayer verify the deposit transaction **status** (success/failure), not just the emitted event?
2. On Solana, is event data sourced from confirmed transaction logs with slot finality, or from raw trace data that can include failed-tx outputs?
3. Is there a minimum-deposit amount check that an attacker cannot fulfill with dust from a failed transaction?

**Historical context:** This same deposit-event spoofing category was identified in EVM bridges in 2022–2023; it's now resurfacing as bridges add Solana support without porting the same verification rigor.

## 4. 🎯 Hunt Hooks

### Hook 1: Withdrawal Authorization Audit — Check the Layer BETWEEN User and Signer
*Tied to §1.1 CoinDCX $44M + §1.4 BigONE $27M + §2.1 Keyless Drain Pattern*

The week's dominant theme: attackers don't need the private key if they control the system that **authorizes the key to sign**. On-chain contracts have `onlyOwner` / timelocks / multisig. The equivalent for CEX hot-wallet pipelines is the **withdrawal authorization service**.

**What to probe in CEX/semi-custodial targets:**

```
1. Is the withdrawal API endpoint rate-limited INDEPENDENTLY of the risk-engine?
   (If the risk-engine is bypassed, can the API still submit 10,000 withdrawals/minute?)

2. Does the withdrawal service accept session tokens, or device-bound hardware tokens?
   - Session tokens: rotate on employee offboarding, can be replayed cross-device
   - Device-bound: non-exportable, malware cannot steal the credential

3. Is there an anomaly detection step BETWEEN "withdrawal approved by risk engine" and
   "transaction signed by HSM/MPC"?
   - Both CoinDCX and BigONE had the signer/HSM safe — but no final gate before signing

4. For on-chain exchange contracts (hybrid CEX):
   - Is withdrawTo(address, uint256) callable by any backend server key, or only by a
     specific multi-signature hot-wallet controller?
   - Can a single compromised backend API key authorize the full treasury balance withdrawal?
```

**Bug class:** `key-management` | **Ceiling:** Critical

### Hook 2: Bridge Deposit Verification on Non-EVM Chains
*Tied to §1.2 Across Protocol Solana relayer exploit + §2.2 event-spoofing pattern*

Bridges porting their EVM verification logic to Solana, Cosmos, or Move-based chains are a major hunting ground right now. The specific question: **does the relayer/verifier confirm deposit transaction SUCCESS, not just deposit event emission?**

```python
# Pseudocode: what a VULNERABLE Solana relayer might do
def on_deposit_event(event_data):
    # BUG: verifies the event existed, not that the tx succeeded
    if event_data.deposit_id not in seen_deposits:
        fill_order(event_data.recipient, event_data.amount)  # front capital immediately

# What it SHOULD do:
def on_deposit_event(event_data, tx_signature):
    tx = rpc.getTransaction(tx_signature, commitment="finalized")
    if tx.meta.err is not None:
        return  # FAILED tx — do not fill
    if tx.slot < current_finalized_slot:
        return  # not yet finalized
    fill_order(event_data.recipient, event_data.amount)
```

**Specific code patterns to grep in bridge relayer repos:**
```bash
# In Solana/TypeScript bridge code — dangerous patterns:
grep -r "getLogs\|parseLogs\|onEvent\|addEventListener" relayer/ \
  --include="*.ts" --include="*.js" \
  | grep -v "isSuccess\|err === null\|meta.err\|finalized"

# Check if relayer waits for finality before filling:
grep -r "commitment" relayer/ --include="*.ts" \
  | grep -E "confirmed|processed"  # WARN: should be "finalized" for security
```

**Report condition:** Relayer fills orders based on unconfirmed or failed-tx event data → Critical.

### Hook 3: Check Your Targets' Employee Endpoint + API Withdrawal Controls
*Tied to §1.1 CoinDCX social-engineering entry point + §2.3 Indian CEX cluster*

With two major CEX hacks this month both tracing back to employee-endpoint compromise (CoinDCX explicitly via fake job offer + malware), the **"malware-as-an-entry-point to exchange backend"** vector is actively being weaponized. Bug bounty programs that include social engineering or insider-threat scenarios in their scope should look for:

1. **Withdrawal API calls that accept plain session cookies** — not FIDO2/WebAuthn hardware keys — as the final authorization signal
2. **Admin panels on non-standard ports reachable from the internet** without device certificate pinning
3. **CI/CD pipeline environment variables** that contain hot-wallet API keys or AWS credentials that could authorize withdrawals (any `PRIVATE_KEY`, `MNEMONIC`, or `AWS_SECRET` in GitHub Actions secrets reachable via a compromised runner)
4. **Batch-withdrawal endpoints** without per-transaction confirmation step that could drain the full balance in a single API call

**Bug class:** `key-management`, `supply-chain` | **Ceiling:** Critical
