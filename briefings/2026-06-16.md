## 1. 🔴 Active Exploits (last 24h)

### 1.1 Thetanuts Finance — $2.1M | Flash Loan + Low-Supply Math Flaw | Ethereum *(June 15, 2026)*

- **Chain:** Ethereum
- **Protocol:** Thetanuts Finance — a decentralized structured-product / options vault platform
- **Root cause:** A **low-supply accounting flaw** in the protocol's **index-token mint and claim logic**. The vulnerability lives in a deprecated vault that had been migrated away from but remained deployed and callable. When `totalSupply` in the vault drops to near-zero (as it does in a dormant/deprecated pool), the standard formula `amount * backing / totalSupply` becomes arithmetically unsafe — integer division can produce massively inflated claim amounts. An attacker used a **flash loan** to enter the deprecated vault, exploit the broken math, and exit with far more than they deposited. Blockaid characterised the root cause as a *"low-supply accounting flaw in the index-token mint and claim logic."*
- **Kill chain (simplified):**
  1. Attacker takes a flash loan of a large token amount.
  2. Attacker deposits into a dormant, low-`totalSupply` Thetanuts vault.
  3. `totalSupply ≈ 0` → mint formula overflows in attacker's favour → attacker credited a vastly inflated index-token balance.
  4. Attacker redeems the inflated balance for real backing assets, repays flash loan, keeps profit.
- **Amount drained:** ~$2.1M gross
- **Outcome:** Most funds (~$2M in option tokens) were **whitehatted and recovered**. The attacker retained approximately $105K USDC + ~60 ETH + $34K USDC in option tokens (~$139K net profit). Thetanuts confirmed the exploited vault was **deprecated** and completely separate from current live contracts and products.
- **Bug class:** `flash-loan`, `rounding`
- **Detection:** PeckShieldAlert and Blockaid flagged the exploit in near-real-time via on-chain activity monitoring.
- **Status:** Deprecated vault; current products unaffected per team. Whitehat recovery complete.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Deprecated-Vault Math Collapse — The "Forgotten Pool" Attack Surface

- **Pattern:** Thetanuts highlights a recurring but underweighted vulnerability class: **protocols that migrate users away from v1/legacy vaults but leave the contracts deployed and accessible.** Deprecated vaults typically have near-zero `totalSupply` and tiny backing balances — conditions that cause `amount * backing / totalSupply`-style formulas to produce wildly incorrect results. An attacker needs only to be the first to interact with the dormant pool after supply drains to near zero.
- **Why it's dangerous:** Deprecated contracts rarely appear in current audit scope. Security teams monitor live TVL, not legacy deployments. Bug bounty scopes often explicitly exclude "deprecated" contracts — but if they remain callable and hold any real tokens (or if a flash-loan deposit can revive them), they are live attack surface.
- **Applies to:** Any DeFi protocol that has released a v2 migration while leaving v1 contracts deployed — options vaults (structured products), AMM pool versions (Uniswap v2→v3 stragglers), staking contract iterations, lending markets with retired asset pools.
- **Formula to audit:** Look for any `(amount * backing) / totalSupply` or `shares * price / totalSupply` in `mint()`, `deposit()`, or `claim()` functions. Confirm that a `require(totalSupply > MIN_SUPPLY)` or equivalent zero-division guard exists AND is tested with `totalSupply == 0` and `totalSupply == 1` in the test suite.
- **Sources:**
  - [CryptoTimes — $2.1M Exploit Hits Thetanuts (Jun 15, 2026)](https://www.cryptotimes.io/2026/06/15/2-1m-exploit-hits-thetanuts-inside-the-latest-defi-flash-loan/)
  - [Crypto Economy — ThetanutsFi $2.1M Exploit](https://crypto-economy.com/thetanutsfi-suffers-2-1-million-exploit-white-hat-recovers-most-funds/)

### 2.2 Flash-Loan + Reward-Before-Balance Pattern — Confirmed Active (NovaBox, June 10)

- **Pattern (re-emphasis):** The NovaBox exploit (covered in the June 12 briefing, amplified here for hunter hooks) is the textbook version of a flash-loan-amplified **balance-before-snapshot ordering error**: dividends/rewards are calculated using a snapshot of balances taken *before* deposits/withdrawals update the state, allowing an attacker to borrow a huge sum, deposit it, collect inflated rewards, then repay — all in one transaction. NovaBox lost 99.86% of its ETH reward pool (~$107K) in a single Aave V3 flash loan.
- **Why both patterns matter today:** Thetanuts (low-supply) and NovaBox (reward-ordering) were both flash-loan attacks against math/accounting, both in June 2026, both on Ethereum. The underlying class is identical: **economic invariant violation under adversarial transaction ordering.** If a protocol's arithmetic relies on an internal state that an attacker can temporarily manipulate via flash-borrowed capital, the invariant is broken.
- **Sources:**
  - [CryptoTimes — NovaBox Loses Nearly 99.86% ETH Reward Pool in Flash Loan Exploit (Jun 10, 2026)](https://www.cryptotimes.io/2026/06/10/novabox-loses-nearly-99-86-eth-reward-pool-in-flash-loan-exploit/)

## 3. 🟢 Advisories / CVEs

### 3.1 Node.js HIGH Severity Security Releases — Scheduled June 17, 2026 *(Pre-Announcement)*

- **Affected versions:** Node.js **26.x**, **24.x** (LTS), and **22.x** (LTS)
- **Severity:** HIGH (highest severity class, confirmed by Node.js security team in pre-announcement)
- **CVE details:** Not yet published (pre-disclosure window). Specifics will appear in release notes for v26.3.0, v24.16.0, and v22.x.x upon publication June 17.
- **Web3 / blockchain relevance:** Node.js is the runtime for Hardhat, ethers.js, web3.js, many keeper bots, backend RPC services, and signing infrastructure. A HIGH-severity Node.js CVE in crypto/network layers (the most common category for HIGH-rated Node.js advisories) can directly affect:
  - Key management scripts and signing services running on Node.js
  - Hardhat test pipelines and CI/CD deployments
  - Backend RPC proxies that relay transactions for DeFi protocols
  - Keeper/liquidation bots that call on-chain functions
- **Action:** Update to patched versions immediately on June 17 when released. Do **not** run signing services or keeper bots on unpatched 24.x/22.x after that date.
- **Sources:**
  - [Node.js — Wednesday June 17, 2026 Security Releases (Pre-Announcement)](https://nodejs.org/en/blog/vulnerability/june-2026-security-releases)

### 3.2 CVE-2026-48054 — OpenZeppelin Contracts Wizard Code Injection *(Carried Forward)*

- Still outstanding from the [June 13 briefing](2026-06-13.md). **`@openzeppelin/contracts-wizard`** injects attacker-controlled Solidity into generated test scaffolds. ## 4. 🎯 Hunt Hooks

**Hook 1 — Audit deprecated / migrated vaults for zero-supply math collapse.**
For every protocol in your scope that has shipped a v2 migration: (a) enumerate *all* deployed contracts from v1 — look for `OldVault`, `LegacyPool`, `DeprecatedStaking`, or version-stamped contract names in the Etherscan deployment history; (b) check whether v1 contracts still hold non-zero token balances (Etherscan token balance tab); (c) call the v1 `totalSupply()` — if it is `0` or near-zero, the `shares * backing / totalSupply` formula will produce a divide-by-zero revert *or*, if no guard exists, an integer overflow/underflow. Write a PoC that deposits `1 wei` via flash loan and reads the minted share amount — if it exceeds the deposited amount, you have a critical. This is the exact Thetanuts kill chain, generalized.

**Hook 2 — Check reward/dividend snapshot ordering in pools with flash-loan-accessible entry.**
In any liquidity or staking pool where rewards are distributed proportionally: (a) trace the execution order in `deposit()` / `withdraw()` — does the reward snapshot (`earned`, `rewardPerToken`, `dividendPerShare`) read `balanceOf(msg.sender)` *before* or *after* updating it? If *before*, an attacker can flash-loan-deposit, collect a reward snapshot based on the inflated balance, withdraw, and repay — all in one tx; (b) confirm `nonReentrant` is applied to both `deposit()` and `claim()` and that they share the same lock (some protocols lock each function independently, missing cross-function reentrancy); (c) the NovaBox pattern specifically: check whether a *new* address with no prior deposit can immediately collect rewards from earlier allocations. A brand-new address should have `rewardDebt` initialized to the *current* `accRewardPerShare` to prevent harvest of historical rewards.

**Hook 3 — Node.js supply-chain surface in keeper bots / signing infra (pre-patch June 17).**
Before the Node.js HIGH-severity patch drops June 17: (a) if a protocol in your scope runs public keeper bots, liquidation scripts, or API endpoints built on Node.js 22.x / 24.x / 26.x, document the version and flag it as a pre-patch supply-chain risk; (b) check whether the keeper bot signs transactions with a hot wallet loaded into Node.js runtime memory — a HIGH-severity Node.js RCE would mean an attacker who can exploit the CVE can steal the signing key; (c) after June 17 releases, if a protocol is slow to patch its keeper infrastructure, this becomes a valid bug bounty finding (supply-chain / key-management severity). Watch the release notes for CVE specifics — if the HIGH is in the `crypto` or `net` module, the DeFi blast radius is severe.
