## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 CVE-2026-40072 Characterized — web3.py SSRF via EIP-3668 CCIP Read

**Published to NVD:** July 10, 2026 (indexed) → **fully described by July 14 (Day 4)**
**CWE:** CWE-918 — Server-Side Request Forgery (SSRF)
**Package:** `web3.py` (PyPI)
**Affected versions:** 6.0.0b3 ≤ version < 7.15.0 and < 8.0.0b2
**Fix:** Upgrade to `web3.py >= 7.15.0` or `>= 8.0.0b2`
**Source:** [CVE-2026-40072 — OffSeq Threat Radar](https://radar.offseq.com/threat/cve-2026-40072-cwe-918-server-side-request-forgery-61669d9c) | [NVD](https://nvd.nist.gov/vuln/detail/CVE-2026-40072)
**Bug class:** `supply-chain` (tool-chain SSRF), `logic-error`

#### What it is

EIP-3668 ("OffchainLookup" / CCIP Read) allows smart contracts to redirect `.call()` return values to off-chain HTTP endpoints. `web3.py` implements this by automatically following contract-supplied URLs in `offchain_lookup_payload["urls"]` — **without any destination validation**. CCIP Read is **enabled by default**.

The result: any Python backend that calls `.call()` against a contract address it does not fully control can be forced by that contract to issue HTTP requests to **arbitrary destinations** — including:

- Cloud metadata services (`169.254.169.254` on AWS, GCP, Azure) → credential theft
- Internal network services (admin panels, databases, key management APIs)
- Any HTTP endpoint accessible from the server's network

#### Attack scenario

```
Attacker deploys contract with OffchainLookup revert:
  revert OffchainLookup(
    address(this),
    ["http://169.254.169.254/latest/meta-data/iam/security-credentials/"],
    callData,
    this.callback.selector,
    callData
  );

Victim backend calls:
  contract.functions.balanceOf(user).call()  # Triggers SSRF

web3.py follows the attacker-supplied URL → leaks AWS IAM token to attacker's server
```

#### Real-world exposure

This is not theoretical. Any protocol with a backend that:
- Monitors on-chain state by calling smart contract view functions
- Uses `eth_call` via web3.py against user-supplied or third-party contract addresses
- Runs indexers, price feed watchers, or API services using web3.py

…is potentially vulnerable. A single malicious token contract on any EVM chain could trigger SSRF against any API backend querying it.

#### Mitigation

```python
# Option 1 (recommended): Upgrade
pip install "web3>=7.15.0"

# Option 2: Disable CCIP Read per-call (unpatched versions)
contract.functions.balanceOf(user).call(ccip_read_enabled=False)

# Option 3: Restrict allowable URL destinations in network egress rules
```

## 4. 🎯 Hunt Hooks

### Hook 1: web3.py SSRF — Find Unpatched Backends Calling Untrusted Contracts
*Tied to §2.1 / §3 CVE-2026-40072*

**The opportunity:** Any bug bounty program that includes backend API / indexer / infrastructure scope is now potentially vulnerable to Critical SSRF if they run an unpatched `web3.py` backend. The attack requires only a malicious contract — not a flaw in the target's own code.

```bash
# 1. Check if the target backend uses web3.py
# Look for requirements.txt / pyproject.toml / setup.py in repos
grep -rn "web3" requirements*.txt pyproject.toml setup.py 2>/dev/null
pip show web3 2>/dev/null | grep Version
# If version < 7.15.0, it's vulnerable

# 2. Find .call() invocations against non-hardcoded addresses
grep -rn "\.call(" . --include="*.py" | grep -v "#"
# Any call against a variable address (not a hardcoded, trusted contract) is the surface

# 3. Confirm CCIP_READ is not explicitly disabled
grep -rn "ccip_read_enabled=False" . --include="*.py"
# If this appears, that call is protected. Calls without it are exposed.

# 4. Test SSRF: deploy a contract with OffchainLookup pointing to your Burp Collaborator
# If the backend queries your Collaborator URL, it's SSRF-confirmed
```

**Report condition:** Backend uses unpatched web3.py + calls user-influenced or third-party contract addresses + CCIP Read not disabled + bug bounty scope includes backend/infrastructure = **Critical SSRF**, potential AWS credential leak.

**Bug class:** `supply-chain`, `logic-error` | **Ceiling:** Critical | **Bounty potential:** $100K–$500K

### Hook 2: BLS/Pairing Verifier — Reject the Identity Point (Carry-forward, still active)
*Tied to Bonzo Lend $9M — Supra oracle BLS bypass (first covered 2026-07-12)*

The Bonzo Lend post-exploit pattern continues to be highly relevant: any protocol using BLS aggregated signatures (oracle verifiers, bridge DVN verifiers, ZK proof verifiers on BN254/BLS12-381) that does **not** explicitly reject identity-point inputs before calling the pairing precompile.

```solidity
// Flag: any verifier that calls pairing() without this guard
require(!pubkey.isZero(), "Identity point rejected");
require(!sig.isZero(), "Zero signature rejected");
// Then call pairing...
```

Target: oracle verifiers (Supra-adjacent), LayerZero DVN verifiers, EIP-2537 BLS wallet implementations, ZK proof verifier contracts on Ethereum mainnet. One missing `isZero()` check = Critical finding.

**Bug class:** `oracle-manipulation` | **Ceiling:** Critical | **Bounty potential:** $500K–$2M+
