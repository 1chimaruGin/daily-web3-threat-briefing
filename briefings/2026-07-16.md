## 1. 🔴 Active Exploits (last 24h)

### 1.1 Ostium Perp DEX — $18–22M Oracle Signer Key Compromise (Arbitrum)

**Date:** July 15, 2026  
**Chain:** Arbitrum  
**Amount:** ~$18M USDC (Blockaid estimate) — ~$22M (CertiK) — up to ~$23.7M (CryptoBriefing). Best-estimate consensus: **$18–22M**  
**Bug class:** `key-management` + `oracle-manipulation` + `price-manipulation`  
**Status:** Trading halted. No post-mortem published. No recovery announced. Funds in attacker wallet.  
**Sources:** [Decrypt](https://decrypt.co/373566/defi-exploit-ostium-oracle-hack) | [CoinDesk](https://www.coindesk.com/business/2026/07/15/ostium-suffers-usd18-million-exploit-as-oracle-attack-wave-continues-to-hit-defi) | [The Defiant](https://thedefiant.io/news/hacks/ostium-halts-trading-after-oracle-exploit-drains-up-to-usd18m-from-vault) | [CryptoTimes](https://www.cryptotimes.io/2026/07/15/ostium-pauses-trading-after-alleged-18m-arbitrum-vault-exploit/) | [Protos](https://protos.com/more-oracle-exploits-as-ostium-loses-over-20m/)

**Protocol profile:** Ostium is an Arbitrum-based perpetuals DEX focused on real-world assets — gold, forex, equity indices. Had raised $27.8M and processed $50B+ in lifetime volume; $63M TVL at time of exploit. The attack drained roughly one-third of all liquidity.

**Root cause:** The attacker obtained a cryptographic private key used to sign and authorize Ostium's off-chain oracle price updates. Crucially, Ostium's on-chain verifier did not reject **future-dated** oracle reports — allowing an authorized signer to feed the system prices that had not yet occurred and which reflected massive fictitious profits.

**Attack chain (single `executeBatch` transaction, ~10 loops):**
1. Attacker controls the oracle signer key AND holds access to a registered `PriceUpKeep` forwarder address (an authorized Chainlink Keeper-style automation address)
2. Opens a trade at the current market price (legitimate, small)
3. Calls `performUpkeep` via the forwarder, submitting a **future-dated, attacker-signed oracle report** showing massively favorable price movement
4. Immediately closes the trade, locking in the fictitious profit
5. Compounds — margin snowballs from ~$1K → $80K → $700K → approximately **+900% per round**
6. Repeats ~10 times in the same atomic batch; total payout ~$18–22M USDC from OLP vault

**Why it worked:**
- Oracle report signature was cryptographically valid (attacker held the key)
- The verifier accepted future-dated timestamps without bounding them against `block.timestamp`
- The keeper forwarder was a registered address — not flagged as unauthorized
- The compound-in-batch design meant on-chain monitoring had zero blocks to react

**Incomplete mitigations that failed:** Chainlink-style keeper registration does not prevent a compromised key holder from submitting malicious data. Access controls were present; the key exposure bypassed them entirely.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Oracle Keeper Compound Attack — A Crystallizing Playbook

**Sources:** [Wu Blockchain / Blockaid X thread](https://x.com/WuBlockchain/status/2077410420877557767) | [CoinDesk Ostium](https://www.coindesk.com/business/2026/07/15/ostium-suffers-usd18-million-exploit-as-oracle-attack-wave-continues-to-hit-defi) | [Blockonomi — Bonzo Lend](https://blockonomi.com/bonzo-lend-loses-9-05m-in-hedera-oracle-exploit-linked-to-supra-flaw) | [CryptoBriefing — Ostium](https://cryptobriefing.com/ostium-suspends-trading-olp-vault-exploit/)  
**Bug class:** `oracle-manipulation` + `key-management`

**Three oracle attacks in five days (July 11–15):**

| Date | Protocol | Mechanism | $ Lost |
|---|---|---|---|
| Jul 11 | Bonzo Lend (Hedera) | Zeroed BLS signature accepted as valid by Supra verifier | $9M |
| Jul 14 | (no new attack; Bonzo still halted) | — | — |
| Jul 15 | Ostium (Arbitrum) | Compromised oracle signer key + future-dated report accepted | $18–22M |

The Ostium attack crystallizes a **distinct sub-pattern** from Bonzo: instead of exploiting signature verification math (zeroed key → pairing returns true), Ostium's attacker controlled a **legitimately obtained (or stolen) signer key** and exploited the absence of timestamp bounds on oracle data. Both attacks share a root: the on-chain contract trusted off-chain oracle credentials without defensive bounding.

**The compound trade loop is novel.** Prior oracle manipulation (e.g., Bonzo) used a single flash-loan-style manipulation → borrow sequence. Ostium's attacker ran a **compounding loop inside a single batch transaction**: each round reinvested proceeds, achieving ~900% return per iteration. This pattern likely works on any perp DEX where:
- Oracle price is updatable by an authorized signer within the same transaction
- The protocol does not enforce a minimum time-delta between price update and trade execution
- Keeper and trading functions can be called atomically

Any protocol with those three properties is vulnerable regardless of whether the signer key is compromised — the batch-compound mechanic may also be exploitable via flash loan if `performUpkeep` can be called without a registered keeper address.

### 2.2 Pattern: Registered Automation Addresses as Exploit Amplifiers

The Ostium attack leveraged a **registered `PriceUpKeep` forwarder** — a Chainlink Keeper-pattern address that was authorized to call `performUpkeep` on-chain. The registration gave the address elevated trust; the protocol did not apply additional validation to upkeep data payloads submitted by registered forwarders.

**This is a generalizable risk across any protocol that:**
- Uses Chainlink Automation (or a similar keeper registry)
- Passes oracle or price data as calldata inside the upkeep payload
- Applies validation only at the signer level (not at the data-content level)

A compromised registered-keeper address becomes an on-chain amplifier — it can submit attacker-controlled payloads with protocol-level trust. Audit target: any `performUpkeep(bytes calldata performData)` implementation that decodes and acts on `performData` without bounding its economic parameters (e.g., price deviation limits, staleness windows, timestamp upper bounds).

## 4. 🎯 Hunt Hooks

### Hook 1: Oracle Report Timestamp Bounding on Perp DEXes
*Tied to §1.1 Ostium $18–22M + §2.1 oracle keeper compound pattern*

The Ostium attack required no flash loan and no contract flaw — just a compromised key and an absent timestamp upper-bound check. This makes the underlying class exploitable even without key theft if the protocol also allows any EOA to submit oracle updates (check the `onlyOracle` modifier scope).

**What to look for in your perp DEX targets:**

```solidity
// RED FLAG — oracle price accepted with no timestamp bounding:
function updatePrice(uint256 price, uint256 timestamp, bytes calldata sig) external {
    require(_verify(price, timestamp, sig), "bad sig");
    // No check: require(timestamp <= block.timestamp + ALLOWED_FUTURE_DELTA, "future-dated");
    _price = price;
}

// RED FLAG — keeper performUpkeep accepts arbitrary performData with no payload bounding:
function performUpkeep(bytes calldata performData) external {
    (uint256 price, uint256 timestamp) = abi.decode(performData, (uint256, uint256));
    // No max-price-deviation check vs current Chainlink feed
    // No timestamp upper-bound check
    _executeSettlement(price);
}
```

**Specific checks:**
1. Does the oracle updater function enforce `timestamp <= block.timestamp + MAX_FUTURE_SECONDS`?
2. Does `performUpkeep` validate that the price embedded in `performData` is within N% of a secondary trusted oracle (e.g., Chainlink)?
3. Can a single `executeBatch` or multicall both update the oracle price AND open/close a trade in the same transaction? If yes, there is zero time for a keeper circuit-breaker to react.
4. Is the oracle signer key in a hot wallet? Check deployer address history — if the signer key ever appeared on-chain in a `transfer` or `approve`, it may have been exposed.

**Bug class:** `oracle-manipulation` + `key-management` | **Ceiling:** Critical

### Hook 2: Keeper Forwarder Trust Scope — `performData` Payload Validation
*Tied to §2.2 registered automation addresses as exploit amplifiers*

Any Chainlink Automation (keeper) integration that passes oracle data or economic parameters in the `performData` payload is a candidate. The key question: does the contract validate the **economic content** of `performData`, or does it trust the registered keeper address to pass safe data?

```bash
# Search scope targets: perp DEXes, lending protocols with liquidation keepers, 
# yield vault rebalancers on Arbitrum/Base/Optimism

# 1. Find all performUpkeep implementations in scope
grep -r "performUpkeep" . --include="*.sol" -l

# 2. For each file, check whether performData is decoded and bounded:
grep -A 20 "function performUpkeep" YourTarget.sol | grep -E "require|revert|MAX|MIN|deviation|staleness"
# No matches = no payload bounding = potential compound-drain vector

# 3. Check keeper registry — is the forwarder address set in an immutable or can it be changed?
grep -r "forwarder\|upkeepId\|KeeperRegistrar" . --include="*.sol"
```

**Report condition:** `performUpkeep` decodes price/amount data without deviation or timestamp bounds AND can atomically trigger a settlement paying from a shared liquidity vault → Critical.

### Hook 3: Oracle Signer Key Operational Security Audit
*Tied to §1.1 Ostium root cause + §3 CVE-2026-48558 Djinn Stealer*

The Ostium exploit likely began with key theft, not a contract flaw. For protocols in your bug bounty scope that run their own oracle signing infrastructure:

**Fast OSINT recon:**
```bash
# 1. Find the oracle signer address (usually in docs, deployment scripts, or on-chain config)
cast call <PROTOCOL_CONTRACT> "oracleSigner()(address)" --rpc-url $ARB_RPC

# 2. Check if the signer address has ever interacted with a DEX, sent to an exchange, or 
#    appears in any public GitHub deployment script (suggests it's a hot key)
cast tx --rpc-url $ARB_RPC $(cast call <PROTOCOL_CONTRACT> "oracleSigner()(address)" --rpc-url $ARB_RPC) 

# 3. Check funding source of the signer address — Tornado Cash or CEX withdrawal = 
#    anonymous origin, consistent with attacker-pre-positioned key
cast logs --address <SIGNER_ADDR> --from-block 0 --rpc-url $ARB_RPC | head -20
```

**Report condition:** Oracle signer key is a hot EOA (not multisig, not MPC), funded from unknown source, and the contract does not have a secondary oracle circuit-breaker that bounds accepted prices against a Chainlink/Pyth reference feed → Tier: High (key compromise scenario). If the signer key is rotatable without timelock, also flag the rotation mechanism as a single point of failure → potential additional High.
