## 1. 🔴 Active Exploits (last 24h)

### 1.1 Taiko Bridge (Ethereum L2) — $1.7M | SGX Signing Key Leaked to GitHub | Ethereum

- **Date of exploit:** June 21–22, 2026 (post-mortems published June 24–25)
- **Chain:** Ethereum (Taiko L2 ↔ L1 bridge)
- **Amount lost:** ~$1.7M in USDC, ETH, and TAIKO tokens; ~2M TAIKO tokens (~$170K) already moved to MEXC before the team froze withdrawals
- **Root cause:** Taiko runs a multi-prover stack called **Raiko**, which uses Intel SGX (Software Guard Extensions) enclaves to sign state proofs that authorize L1 ↔ L2 asset transfers. A Raiko SGX signing key was inadvertently committed to a **public GitHub repository**. The attacker discovered the exposed private key, used it to forge valid Raiko proofs for fabricated withdrawal transactions, and presented those proofs to the Taiko ERC20 Vault on Ethereum mainnet.

  The critical flaw: the L1 bridge contract validated that withdrawal messages carried a valid Raiko SGX signature but did not cross-check that a corresponding `MessageSent` event existed on the Taiko L2 source chain. The attacker could thus withdraw assets that were **never deposited**.

- **Timeline:**
  - Key committed to GitHub: unknown (days/weeks before exploit)
  - Exploit transactions: ~June 21, 2026
  - Bridge withdrawals frozen: ~June 22, 2026, 2:08 AM ET
  - Taiko team urged all users to exit remaining bridge positions via emergency UI
- **Bug class:** `key-management` (SGX key exposed in public repo), `bridge-dvn` (forged proof accepted without on-chain event cross-check)
- **Fund status:** ~$170K TAIKO moved to MEXC; remainder frozen in bridge contracts; recovery unclear

### 1.2 OLPC/LABUBU PancakeSwap V2 Pool — $1.1M | Token Parameter Desync → Reserve Manipulation | BNB Chain

- **Date of exploit:** June 20, 2026
- **Chain:** BNB Chain (PancakeSwap V2)
- **Amount lost:** ~$1.1M USDT; attacker bridged proceeds to Ethereum and routed through Tornado Cash
- **Root cause:** The OLPC token's `decimalsValue` parameter had been **modified by an admin 46 days before the exploit** — a change that altered how the token calculated its effective balance on transfer. This created a **desync between the cached reserve balances** stored in PancakeSwap's V2 pool state and the real-time token balances (since PancakeSwap's `getReserves()` returns cached values that are only synced on `mint/burn/swap`).

  The attacker exploited the gap:
  1. Triggered a transfer that invoked the modified decimal logic, burning a large quantity of tokens to the dead address (`0x000...dEaD`)
  2. This burn deflated the real OLPC balance without updating pool reserves — breaking the constant-product invariant
  3. The attacker then called `sync()` to force reserves to match reality (at the now-deflated ratio), effectively purchasing OLPC tokens at a far below-market price
  4. Swapped back to USDT and drained the pool

- **Bug class:** `price-manipulation` (reserve desync exploited via token parameter change), `oracle-manipulation` (spot price used by any downstream protocol reading this pool's reserves was also corrupted during the attack window)
- **Root cause (systemic):** AMM pools that allow tokens with mutable transfer behavior (changeable decimals, fee-on-transfer toggle, burn-on-transfer switches) create latent reserve-desync risks that can sit dormant for weeks before an attacker times the strike
- **Fund status:** Bridged to Ethereum, routed through Tornado Cash; unrecovered

### 1.3 mySwap Concentrated Liquidity — ~$305K | CL Price Manipulation | Starknet

- **Date of exploit:** June 19, 2026, ~7:15 AM UTC (post-mortems circulating June 23–25)
- **Chain:** Starknet
- **Amount lost:** ~$300–305K drained from concentrated liquidity (CL) pools
- **Root cause:** mySwap CL's price oracle relied on **instantaneous pool tick state** without a time-weighted average (TWAP) buffer. An attacker manipulated the active tick via a large swap, then used the resulting price dislocation to drain liquidity from adjacent tick ranges at off-market rates. Standard CL price manipulation: move the tick into a range with thin liquidity, extract at manipulated price, reset.
- **Bug class:** `oracle-manipulation` (spot tick used without TWAP), `price-manipulation`
- **Fund status:** Unknown; Starknet forensics lagged given tooling immaturity

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 TEE / SGX Key Material in L2 Prover Infrastructure — A Systematic Exposure Pattern

The Taiko exploit formalizes a **new attack class for TEE-backed L2 networks**: signing keys for SGX enclaves that produce validity or optimistic fraud proofs are treated as software secrets rather than HSM-protected credentials — and they leak through the same vectors as any other developer credential.

**Why this pattern is broadening:**
- The post-Dencun, post-Pectra L2 ecosystem is rapidly adopting TEE co-provers (Intel SGX, AMD SEV, ARM TrustZone) as a cheaper alternative to full ZK proofs during early mainnet phases
- Protocols using Raiko-style multi-prover stacks (Taiko, Brevis, RiscZero integrations, Succinct's SP1 deployments) often have an SGX signer as a fallback or primary verifier
- SGX enclave signing keys must exist somewhere in plaintext during the attestation setup phase — developers frequently commit them to CI/CD configs, `.env` files, or key-generation scripts that end up in Git history

**Relevant L2 prover designs to check today:**
- Any bridge relying on a single SGX enclave signature without a threshold (n-of-m) DVN requirement
- Any bridge that validates proofs by verifier signature alone, without requiring a corresponding source-chain event to exist on-chain
- Prover repos with `.env.example` or `config/` directories that have ever held key material

**Source:** [thirdweb Blog — Taiko Bridge Exploit Explained](https://blog.thirdweb.com/taiko-bridge-exploit-explained-how-a-leaked-key-led-to-1-7m-in-forged-withdrawals/)

### 2.2 Mutable Token Parameters as a Time-Delayed Pool Exploit Vector

The OLPC/LABUBU attack adds a concrete case study to the emerging pattern of **admin-changed token parameters creating dormant AMM vulnerabilities** — similar to the earlier TesseraDAO and Aztec pool-desyncs.

The pattern:
```
Admin changes token parameter (decimals, transfer tax, burn rate) at time T
  → No exploit possible at T because pool reserves still reflect old state
  → Pool accumulates liquidity organically over T+1…T+N days
  → At T+N: attacker triggers a transfer under the new parameter regime
      → Real token balance diverges from cached reserves
      → Attacker forces reserve sync at deflated ratio → drains pool
```

**The audit implication:** When reviewing AMM integrations, the risk window isn't just "does this token have fee-on-transfer NOW" but "does this token have any admin function that can retroactively change transfer behavior, and if so, what is the reserve-sync exposure?" The 46-day lag from param change to exploit makes this invisible to automated scanners that only check current state.

## 4. 🎯 Hunt Hooks

**Hook A — Audit TEE/SGX Signing Key Exposure in L2 Prover Repos**

Taiko's exploit is a direct template for any L2 network using a TEE-backed prover. For any such project in scope:

1. **Git history scan:** `git log --all --full-history -- "*.pem" "*.key" "*sgx*" "*.env"` — look for key material ever committed, even if later deleted. GitHub's secret-scanning does not catch all formats.
2. **Single-DVN bridge configs:** Check the bridge's proof-verification contract — does it require only one SGX signer? If `threshold == 1` and that signer's key can be extracted from a public repo, you have a critical `bridge-dvn` + `key-management` finding.
3. **Source-chain event cross-check:** Read the bridge's `proveMessageReceived()` or equivalent function. Does it verify that `MessageSent` (or equivalent) exists on the source chain, or only that the proof signature is valid? Absence of source-chain event verification = the Taiko pattern.
4. **Raiko-adjacent projects:** Brevis Network, Succinct SP1 integrations, any project with "multi-prover" or "TEE co-prover" in its docs — all share this attack surface.

---

**Hook B — Token Parameter Mutability → AMM Reserve Desync Check**

For any AMM integration in scope:

1. Pull the ERC-20 token contracts for all listed pairs. Search for `setDecimals`, `setTransferFee`, `setBurnRate`, `setTaxRate`, or any function that changes transfer behavior with `onlyOwner` / `onlyAdmin` access.
2. For any token with such a function: check whether it has ever been called (`eth_getLogs` for `Transfer` events before and after suspected parameter changes — look for sudden shifts in transfer amounts at the same value).
3. If the token's parameter was ever changed: simulate a large transfer under the current parameter set versus the cached pool reserves — does the pool's `getReserves()` result diverge from the real `balanceOf(pool)`? A divergence > 1% on a pool with > $100K TVL = critical price-manipulation finding.
4. Any AMM that calls `getReserves()` without calling `sync()` first in critical paths (like liquidation triggers, flash loan fee calculations) is vulnerable to reserve desync exploitation.

---

**Hook C — Bridge Withdrawal Verification: Source-Chain Event Cross-Reference**

The core Taiko design flaw — accepting a valid proof signature without requiring the corresponding source-chain event to exist — is likely present in other bridges built on similar proof-of-authority or TEE-proof models:

1. Locate the bridge's message-receiving function (e.g., `processMessage()`, `claimMessage()`, `finalizeWithdrawal()`).
2. Check: does it verify a proof AND check that the corresponding `MessageSent` / `Deposited` / `TokensBurned` event exists on the source chain (via state root inclusion proof or oracle)?
3. If the contract only validates the proof signature (from a DVN, TEE, or oracle) but does NOT verify source-chain event existence, an attacker with the signing key can forge arbitrary withdrawals — the exact Taiko pattern. Severity: Critical.
4. Cross-check against `bridge-dvn` findings from June 2026 briefings (June 12, 14, 17, 19, 22, 25) — this pattern has appeared in 6 of the last 14 days.
