## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Kinto Proxy Backdoor — Emerging Post-Mortem (ERC-1967, Upgradeability)

**Source:** [The Defiant — Kinto Token Tanks 90%](https://thedefiant.io/news/blockchains/kinto-token-tanks-90-as-backdoor-disclosure-lets-attacker-mint-110-000-tokens-and-drain) | [Cryptonews — Kinto Shuts Down](https://cryptonews.com.au/news/kinto-shuts-down-after-1-55m-exploit-derails-recovery-efforts-130715/) | [The Block — Kinto Shutdown](https://www.theblock.co/post/369731/ethereum-layer-2-kinto-shuts-down-in-wake-of-1-6-million-july-exploit)
**Bug class:** `upgradeability`
**Chain:** Ethereum (L2)

The Kinto shutdown makes this the highest-consequence post-mortem of the week. Key technical points from what's now public:

- The backdoor was embedded in an **ERC-1967 proxy contract** and designed to **survive reinitialization and multiple upgrade cycles** — surviving through at least two audit windows
- It allowed an attacker to call a hidden mint function, creating 110,000 $K tokens and draining Uniswap + Morpho lending pools ($1.55M / 577 ETH)
- A coordinated private disclosure program (Venn Network, Dedaub, SEAL 911) had identified a broader class of this vulnerability and patched multiple protocols — but Kinto was not reached in time
- Post-exploit: token down 90% ($7.69 → $0.50), market cap wiped ~$13M, fundraising collapsed → terminal shutdown

**Pattern for hunters:** Proxy implementations where `_ADMIN_SLOT` (ERC-1967: `0xb53127...`) and `_IMPLEMENTATION_SLOT` (ERC-1967: `0x360894...`) are written independently of the canonical upgrade path — especially through `initialize()` functions that are callable after upgrade — create a persistent backdoor surface. Standard audit tools that only diff bytecode between upgrades can miss planted backdoors that exist in the original implementation and survive every upgrade.

- Risk for web3 teams: any Windows-based CI runner, build server, or validator node using AD FS for SSO is a lateral-movement target. Private key material, signing credentials, and `.env` files become accessible post-EoP.

**Action:** Patch all Windows infrastructure immediately. Rotate any secrets stored in or accessible from Windows-based CI or internal SharePoint instances.

## 4. 🎯 Hunt Hooks

### Hook 1: ERC-1967 Proxy Backdoor — Check Upgrade History and Implementation Slot Integrity
*Tied to §1 Kinto shutdown / proxy backdoor (upgradeability)*

The Kinto exploit shows that audit review of current bytecode is insufficient when a backdoor is baked into the original implementation and preserved through upgrades. The hunt: any protocol on your scope using ERC-1967 transparent or UUPS proxies.

```bash
# 1. Read ERC-1967 implementation slot directly (bypasses proxy ABI)
cast storage <PROXY_ADDRESS> 0x360894a13ba1a3210667c828492db98dca3e2076 --rpc-url $RPC
# Expected: address of current implementation

# 2. Read ERC-1967 admin slot
cast storage <PROXY_ADDRESS> 0xb53127684a568b3173ae13b9f8a6016e243e63b6 --rpc-url $RPC
# Compare against what the project claims is the admin

# 3. Fetch implementation bytecode and decompile
cast code <IMPLEMENTATION_ADDRESS> --rpc-url $RPC > impl.bin
# Decompile with heimdall-rs or panoramix; look for selectors not in the proxy ABI

# 4. Check initializer call history: was initialize() called multiple times?
cast logs --address <PROXY_ADDRESS> --from-block 0 --rpc-url $RPC \
  | grep "Initialized\|Upgraded"
# Any post-deployment re-initialization is a red flag

# 5. Diff bytecode across upgrade history
# Fetch bytecode at each block where Upgraded event was emitted; diff selectors
```

**Report condition:** Hidden selector callable on implementation but not exposed through proxy ABI + can mint/burn tokens or drain vault → Critical (likely `upgradeability` + `access-control`).
**Bounty ceiling:** Critical

### Hook 2: GMX-Style Cross-Contract Reentrancy in Position-Close Paths
*Tied to §1 GMX V1 $42M reentrancy (July 9, 2026)*

The GMX V1 exploit used a classic but subtle pattern: the `PositionManager.executeDecreaseOrder()` assumed `_account` was an EOA, sent a small ETH refund via `_transferOutETH()`, and did not guard against reentry. The attacker's `receive()` hook re-entered `Vault.increasePosition()`, causing short-size accounting to diverge from average-price accounting — inflating AUM and enabling unlimited GLP redemptions.

**Any perpetual/margin protocol with this shape is vulnerable:**

```solidity
// Vulnerable pattern — ETH transfer before state finalization:
function decreasePosition(..., address _account, ...) external {
    // ... accounting updates partial ...
    _transferOutETH(refundAmount, _account);  // <-- callback surface
    // ... more accounting that assumes no reentry ...
}

// Flag: check whether the protocol has CEI (checks-effects-interactions) enforced
// OR a reentrancy guard on every function that modifies global accounting state
```

**Targets:** Any Arbitrum/Avalanche/Base perpetual DEX forked from or inspired by GMX V1. Look specifically for `executeDecreaseOrder`, `executeOrder`, or settlement callbacks that (a) send ETH/tokens to an arbitrary address and (b) occur before global position state is fully flushed.

**Bug class:** `reentrancy` | **Ceiling:** Critical
