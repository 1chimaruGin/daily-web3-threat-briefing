| **Bug class** | key-management, supply-chain |
| **Status** | EMURGO denies knowing white hat identity; recovery tool ETA early next week; users warned against impersonation scams |

**New developments (June 28–29 UTC):**

The SecondFi story took a governance turn. On June 28, Charles Hoskinson cited a meeting between Cardano governance body Intersect and SecondFi/EMURGO principals, in which an EMURGO team member reportedly stated that *the identity of the white hat hacker is not known to EMURGO*. The white hat entity secured 129M ADA (≈$18.5M) from further drainage by moving it to a third-party custodian before the main attack wave completed — but EMURGO now publicly denies knowing who that entity is.

This creates a cascading custody risk:
- $18.5M in ADA sits with a **self-selected, unidentified custodian**
- No legal agreement between EMURGO and the custodian has been disclosed
- Community members are calling for police involvement and a forensic audit of the chain of custody

**Tibane Labs SDK root cause (published ~June 28):** A rival wallet builder's post-mortem identifies the proximate cause as an **unaudited third-party SDK** that replaced SecondFi's audited signing code on June 8 — two weeks before the exploit. The SDK exposed a `signTransaction()` override that leaked private key material at the address-derivation layer on first use. This matches the pattern of a supply-chain injection into mobile wallet signing infrastructure.

> ⚠️ **Still-open risk:** Users with wallets generated or first-used between June 8 and June 23 may have unexposed key material in the tainted SDK path. SecondFi has not yet confirmed whether the patch removes the backdoor versus only rotating affected addresses.

### 2.1 Sapphire Sleet (DPRK) Mastra AI npm Supply Chain — 144 Packages Backdoored in 88 Minutes

**Published:** June 17, 2026 — Microsoft Security Blog  
**Technique class:** supply-chain (npm postinstall payload)  
**Relevance to web3:** HIGH — Mastra is used by a growing number of web3 AI agent projects; the same `postinstall` delivery vector was used in the Axios poisoning (UNC1069, early June) and the `tw-style-utils` SStar Agent RAT campaign targeting web3 engineering candidates.

North Korean threat actor **Sapphire Sleet** (also tracked as UNC1069 / BlueNoroff sub-cluster) executed a fully automated 88-minute campaign on June 17 that compromised **144 packages across the Mastra AI framework's npm scope** (`@mastra/*`). The payload used the standard `postinstall` hook to install a RAT that:

1. Exfiltrates `~/.ssh`, `~/.aws`, `~/.config`, `.env` files, and browser credential stores on first run
2. Establishes a persistent keyboard/clipboard hook targeting cryptocurrency clipboard patterns (EVM addresses, seed phrase words, API keys)
3. Phone-homes to a C2 infrastructure overlapping with the April 2026 Axios poisoning campaign

**Affected versions:** Mastra `@mastra/core` `>=0.9.0` and `<1.2.4` (all packages in the scope published between June 15–17 were poisoned; clean versions restored June 18 after Microsoft disclosure).

**Why it matters for web3 hunters:**
- Any web3 protocol backend that uses AI agent frameworks (Mastra, LangChain, CrewAI) should be treated as a potential key-exfil target
- The SStar Agent campaign (same actor, same week) specifically targets web3 engineering job candidates via a fake GitHub repo (`star45674/smart-contract-engineer-role`) — if a target team ran any coding challenges from unknown GitHub repos in late May/early June, their dev machines may be compromised

## 3. 🟢 Advisories / CVEs

### 3.1 Node.js June 2026 — CVE-2026-48618 (HIGH) TLS Wildcard Bypass

| Field | Detail |
|---|---|
| **CVE** | CVE-2026-48618 |
| **Severity** | HIGH |
| **Subsystem** | TLS (hostname normalization) |
| **Affected** | All Node.js active lines before: v22.23.0, v24.17.0, v26.3.1 |
| **Patch published** | June 18, 2026 (Day 12 today) |

A hostname normalization flaw allows the TLS handshake to accept a wildcard certificate (`*.example.com`) for a subdomain that does not match the expected pattern — e.g., `evil.internal.example.com` verified as valid against `*.example.com`. 

**Web3 impact:** Multi-tenant RPC node providers (Alchemy, Infura clones, self-hosted Ethereum nodes behind HAProxy) universally use wildcard certificates. An attacker in a MITM position (post-Mastra-RAT device compromise, rogue coffee-shop Wi-Fi) can intercept JSON-RPC calls to a patched endpoint and redirect them to an attacker-controlled node — poisoning `eth_call` responses, silently swapping `to` addresses in `eth_sendRawTransaction` staging, or returning fake `eth_getBalance` values.

**Action:** Upgrade all Node.js runtime versions immediately. This is **Day 12** of a HIGH-severity unpatched window.

**Source:** [Node.js June 2026 Security Releases](https://nodejs.org/en/blog/vulnerability/june-2026-security-releases) · [Patch Guide](https://www.digitalapplied.com/blog/nodejs-june-2026-security-releases-cve-patch-guide)

### 3.2 Node.js June 2026 — CVE-2026-48933 (HIGH) WebCrypto DoS

| Field | Detail |
|---|---|
| **CVE** | CVE-2026-48933 |
| **Severity** | HIGH |
| **Subsystem** | WebCrypto (`subtle.encrypt()`) |
| **Affected** | All Node.js active lines before: v22.23.0, v24.17.0, v26.3.1 |
| **Patch published** | June 18, 2026 (Day 12 today) |

`subtle.encrypt()` aborts the Node.js process when input length is a multiple of 2 GiB. This is a **crash-as-a-service** primitive: any web3 backend exposing an encryption endpoint (e.g., ECIES-based message encryption, encrypted IPFS content, NFT metadata encryption) can be remotely crashed by a single crafted request.

**Web3 impact:** Bridge relayer nodes, keeper bots, and oracle aggregators that use Node.js WebCrypto for payload signing/encryption are vulnerable to DoS. An attacker who can craft a 2GiB-aligned payload to the target's encryption path can force process restart — creating a window for front-running or missed oracle updates.

**Source:** [Node.js June 2026 Security Releases](https://nodejs.org/en/blog/vulnerability/june-2026-security-releases) · [Patches 12 CVEs — ByteIota](https://byteiota.com/nodejs-june-2026-security-releases-patch-now/)

## 4. 🎯 Hunt Hooks

**Three concrete things to check in your targets today:**

### Hook 1: Third-Party SDK Supply-Chain in Wallet Signing Paths
*Tied to SecondFi / Tibane Labs root cause disclosure*

For any wallet or signing SDK in your bug bounty scope:
1. Trace the `signTransaction()` / `signMessage()` call chain to its leaf — does it hit an audited internal implementation or route through a third-party SDK?
2. Check `package.json` / `gradle.build` / `Podfile` for signing-related packages added or updated in the **last 60 days** that are not pinned to an audited version
3. For mobile wallets: check if the SDK uses a `postinstall` hook — that's the standard NK delivery vector

**Impact class:** key-management · **Bounty sweet spot:** any protocol with a self-custody wallet feature and >$10M TVL

---

### Hook 2: npm Dependency Audit for AI-Adjacent Web3 Tooling
*Tied to Sapphire Sleet Mastra campaign*

For protocols with AI agent features or AI-powered backends:
1. `npm audit` + manual review of all `@mastra/*` packages — confirm you're on `>=1.2.4`
2. Search for `postinstall` scripts in `node_modules` that make outbound network calls (`exec`, `fetch`, `axios`, `got`, `curl`)
3. Check `.env` and CI secrets: if a dev machine was compromised between June 15–18, treat all secrets rotated after that date as potentially still-leaked if the RAT exfiltrated before rotation

**Impact class:** supply-chain · **Bounty context:** Infrastructure compromise scope varies widely — confirm in-scope before testing

---

### Hook 3: Node.js TLS Version Check on JSON-RPC Endpoints
*Tied to CVE-2026-48618 (Day 12, still unpatched)*

For any protocol that runs Node.js-based RPC relay, keeper bot, or oracle aggregator:
1. `node --version` on the deployment — confirm `>=22.23.0 || >=24.17.0 || >=26.3.1`
2. Check if the service uses wildcard TLS certificates (`*.protocol.xyz`) for its RPC or API endpoint
3. Test: submit a request with a subdomain that should fail cert verification — confirm it's rejected

An unpatched instance + wildcard cert + MITM position = the ability to intercept and forge JSON-RPC responses. If found in a bug bounty scope with evidence the team uses Node.js backends, this is a HIGH severity finding (MiTM required reduces it from CRITICAL).

**Impact class:** dos, key-management · **CVSS base (without MiTM requirement):** HIGH
