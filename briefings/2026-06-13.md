## 1. 🔴 Active Exploits (last 24h)

### 1.1 Humanity Protocol — $36M | DPRK Attribution Confirmed | Ethereum + BNB Chain *(Forensic Update, June 13)*

- **Date of exploit:** June 8–9, 2026 *(new post-mortem published June 13)*
- **Chains:** Ethereum, BNB Smart Chain
- **Root cause — full kill chain now confirmed:**
  1. **Phishing email** targeted a Humanity Foundation member, delivering credential-stealing malware that exfiltrated a private key from a hot wallet.
  2. Stolen key gave the attacker direct access to ~6M $H tokens drained immediately.
  3. The attacker then compromised **3-of-6 owners** of an Ethereum Gnosis Safe controlling the bridge's `ProxyAdmin`. With a quorum of signers colluding or their keys stolen, the attacker pushed a **malicious proxy upgrade** — replacing the bridge implementation with attacker-controlled logic.
  4. Via the upgraded contract, **141.18M $H** were drained in a single transaction.
  5. On BNB Smart Chain, a separate `ProxyAdmin` takeover allowed the attacker to **mint 100M additional $H** (~$12.9M), sold across Uniswap and PancakeSwap over ~8 hours.
- **Attribution:** Independent investigation by **Quantstamp** (commissioned by Humanity Protocol) identified tooling and operational patterns *"characteristic of DPRK-linked intrusions"* — consistent with the Lazarus Group's established play of targeting developer endpoints via spearphishing to reach hot wallets and multisig keys.
- **Bug class:** `key-management` (hot-key on developer endpoint), `upgradeability` (malicious ProxyAdmin upgrade), `access-control` (insufficient key threshold / separation)
- **Amount:** ~$36M total (≈$32M CoinDesk estimate; Quantstamp report notes 141.18M $H drained + 100M minted)
- **Status:** Ethereum deployment secured; BNB Smart Chain contract to be **permanently abandoned**; team confirmed no path to protocol resumption on BNB.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 DPRK "Phishing → ProxyAdmin Upgrade" Kill Chain — Pattern Formalized
- **Pattern:** The Humanity Protocol post-mortem formalizes a Lazarus Group playbook being replicated across 2026 DeFi incidents (Drift Protocol $285M, Wasabi Protocol $4.5M, Humanity $36M): **spearphish a developer/key-holder → steal hot-wallet key → use stolen key to reach multisig threshold → malicious proxy upgrade → drain or mint at will**. The ProxyAdmin is the critical choke point: whoever controls it controls the contract's logic.
- **Why it matters:** 76% of 2026 crypto hack losses are attributed to state-backed actors (Chainalysis). The entry point is almost never a smart contract bug — it's a human endpoint.
- **Where it applies:**
  - **Any protocol with a `ProxyAdmin` controlled by a Gnosis Safe:** Check whether the Safe threshold is achievable by a single stolen device (e.g., a dev who holds multiple keys, or keys co-located on one machine). The Humanity Protocol post-mortem specifically notes that a multisig "lived on one laptop."
  - **Hot-wallet key holders:** Developers who also hold signing authority for bridge or proxy admin contracts are the highest-value phishing targets. Hardware keys and air-gapped signing are no longer optional for admin roles.
- **Sources:** [CryptoTimes June 13](https://www.cryptotimes.io/2026/06/13/humanity-protocol-36m-hack-phishing-email-dprk-links-revealed/) · [TRM Labs — DPRK attacks Drift Protocol](https://www.trmlabs.com/resources/blog/north-korean-hackers-attack-drift-protocol-in-285-million-heist)

### 2.2 Code4rena Shutdown Creates Vulnerability Disclosure Coverage Gap
- **Pattern:** Code4rena has announced it will cease operations; Immunefi is absorbing its bug-bounty clients and researcher base. In the short term (May–June 2026), this creates a **disclosure gap** — protocols that were mid-audit-contest with Code4rena may not have active competitive scrutiny for weeks while migrations complete.
- **Where it applies:** Any protocol that recently launched or extended a Code4rena audit contest. These protocols may have open bugs that would have been found by competing wardens but now sit in a coverage vacuum. Immunefi's platform takeover is the right long-term move, but transitions are inherently risky windows.
- **Hunter opportunity:** Check Immunefi's newly migrated bounty programs for protocols whose Code4rena contests were interrupted — these are high-value targets with potentially unreviewed code.
- **Sources:** [AMBCrypto — Immunefi takes over Code4rena's wardens](https://ambcrypto.com/immunefi-takes-over-code4renas-wardens-can-defi-survive-2026-exploit-surge/) · [CoinAlertNews — Code4rena shuts down, Immunefi steps in](https://coinalertnews.com/news/2026/05/13/code4rena-shutdown-immunefi-takeover)

### 2.3 Phala Cloud TEE Infrastructure Breach — Supply Chain Risk for AI/Crypto Infra
- **Pattern:** On June 1, 2026, Phala identified an API endpoint vulnerability in Phala Cloud that allowed unauthorized modification of Offchain KMS CVMs (Confidential VMs). An attacker deployed a malicious pre-launch script that ran inside the trusted execution environment and accessed **decrypted environment variables after boot** — including AWS credentials, RPC keys, and potentially signing keys. TEE (trusted execution environment) isolation does not protect secrets that are decrypted and available in-process.
- **Where it applies:** Any DeFi infrastructure that uses Phala Cloud's Offchain KMS, or any TEE-based key management system. Secrets that are *injected as environment variables at runtime* are accessible to malicious pre-launch scripts if the orchestration API is compromised — the TEE boundary does not help once the script runs inside it.
- **Mitigation:** Rotate all secrets stored in encrypted env vars on affected CVMs; replace affected CVMs; audit pre-launch hook permissions on any TEE-based deployment.
- **Source:** [Phala — Security Incident Notice: Phala Cloud API Vulnerability](https://phala.com/posts/security-incident-notice-phala-cloud-api-vulnerability)

## 3. 🟢 Advisories / CVEs

### 3.1 CVE-2026-48054 — @openzeppelin/wizard — Code Injection via Unsanitized Token Name/URI
- **Affected package:** `@openzeppelin/contracts-wizard` (OpenZeppelin Contracts Wizard)
- **Affected versions:** Versions prior to the patch addressing CVE-2026-48054 (exact patched version — check [OZ Wizard releases](https://github.com/OpenZeppelin/contracts-wizard/releases))
- **Severity:** High — code injection in generated Hardhat and Foundry test scaffolds
- **What it does:** The Contracts Wizard generates test boilerplate using values passed in `opts.name` and `opts.uri`. These fields are not sanitized before interpolation into generated JavaScript/TypeScript/Solidity test files. A malicious `opts.name` value (e.g., one containing shell metacharacters or template injection payloads) can inject arbitrary code into the generated test file, which runs with the developer's local privileges when the test suite executes.
- **Exploitation path:** An attacker who controls a project's Wizard configuration inputs — e.g., via a compromised CI pipeline, a malicious package that wraps the Wizard, or a social-engineering scenario where a dev pastes a crafted config — can achieve RCE on the developer's machine or CI runner at test-run time.
- **Practical risk:** Moderate in isolation; elevated in supply-chain scenarios (CI/CD with auto-generated scaffolds). Also a hygiene signal: any generated boilerplate from older Wizard versions should be re-generated with the patched release.
- **Fix:** Upgrade `@openzeppelin/contracts-wizard` to the latest patched version. Re-generate all Wizard-produced test files if using a vulnerable version.
- **Sources:** [GitHub Advisory Database](https://github.com/advisories) · [CVEDetails — OpenZeppelin](https://www.cvedetails.com/product/99853/Openzeppelin-Contracts.html?vendor_id=25308) · [Snyk — @openzeppelin/contracts](https://security.snyk.io/package/npm/@openzeppelin/contracts)

## 4. 🎯 Hunt Hooks

**Hook 1 — Map ProxyAdmin ownership on every upgradeable contract in your targets.**
For any proxy-based protocol (TransparentUpgradeableProxy, UUPS, Beacon), identify who controls the `ProxyAdmin` or `owner()` capable of upgrades. Run: check if that controller is a Gnosis Safe, and if so, query `getOwners()` and `getThreshold()`. Red flags: (a) threshold of 2-of-3 where two keys are held by the same person, (b) Safe owners that are EOAs rather than hardware-wallet addresses, (c) any mention in the docs that signing keys are stored on developer laptops. If you can demonstrate that a realistic phishing/social-engineering scenario would achieve the required quorum, that's a high-impact finding. Humanity Protocol's post-mortem is your template.

**Hook 2 — Check your targets' CI pipelines for OpenZeppelin Wizard-generated test files.**
Search target repos for `contracts-wizard` in `package.json` or `package-lock.json`. If an older version is present, check whether generated test files are committed to the repo and whether they interpolate token names or URIs. If a CI pipeline auto-regenerates tests from Wizard config on every run, a poisoned config value is a path to RCE in CI. For bug bounty: if the target has a CI/CD bug bounty scope, this is a valid finding vector worth reporting.

**Hook 3 — Audit all TEE-based key management integrations for pre-launch hook injection.**
For any protocol using Phala Cloud, AWS Nitro Enclaves, or similar TEE-based KMS: check the pre-launch hook architecture. The Phala breach shows that TEE isolation fails if the orchestration layer (API, config service, CI) is compromised and can inject a malicious startup script. Look for: (a) pre-launch scripts that run before the enclave seals, (b) secrets injected as environment variables at runtime vs. sealed inside the enclave at build time, (c) API endpoints that accept per-deployment configuration without strong authentication. This class of infrastructure bug is systematically underexplored in smart contract audits because it sits outside the on-chain scope.
