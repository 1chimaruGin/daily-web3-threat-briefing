## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 EIP-7702 Post-Pectra Attack Surface Map

Since Ethereum's Pectra hard fork shipped with EIP-7702, three independent security teams have now published comprehensive attack-surface maps that consolidate the real-world exploit chain clearly. Key findings across [Nethermind's post](https://www.nethermind.io/blog/eip-7702-attack-surfaces-what-developers-should-know), [Halborn's advisory](https://www.halborn.com/blog/post/eip-7702-security-considerations), and [Zealynx's wallet audit guide](https://www.zealynx.io/research/smart-contracts/eip-7702-wallet-security):

**Attack surface 1 — Missing access control in delegated BatchCall code:**
When an EOA delegates to a `BatchExecutor` or `BatchCall` contract that has no caller allow-list on its `batch()` function, any external address can call `batch()` on behalf of the EOA and transfer out all assets. This was the exact root cause in the QNT reserve pool drain (~$55K, April 2026): `BatchCall.batch()` accepted calls from any `msg.caller`, not just the owning EOA or an authorized list.

**Attack surface 2 — Permanent code injection via single signed authorization tuple:**
Once a victim EOA signs a single EIP-7702 authorization tuple (e.g., via a phishing link), every future call to that address executes attacker-supplied bytecode — regardless of whether the victim is watching. One signed tuple = persistent backdoor. A $1.54M phishing event in August 2025 used exactly this primitive; the 2026 write-ups confirm attackers are now packaging it as a repeatable kit.

**Attack surface 3 — Legacy contract confusion:**
Contracts written before EIP-7702 assume EOAs can't have code. Any `extcodesize(addr) == 0` guard used as an "is this a safe EOA?" check is bypassable by an EIP-7702-delegated EOA. AMMs, flash-loan wrappers, and liquidation checks that rely on this assumption should be audited.

**2026 real-world exploit chain (total ~$8M+):**
- IPOR Fusion PlasmaVault: $336K (Jan 2026) — legacy vault didn't check EIP-7702 delegation on admin EOA
- QNT reserve pool: ~$55K (Apr 2026) — missing caller auth in `BatchCall.batch()`
- DxSale: ~$7.3M (Jun 2, 2026) — compromised owner key + EIP-7702 batch delegation drained 1,400+ LP pools on BNB Chain in one tx

The DxSale incident is the clearest case study: the attacker combined an admin key compromise with EIP-7702 batch delegation to atomically execute 1,400+ LP unlock + drain calls in a single transaction — something impossible before Pectra. This pattern will recur.

- **Sources:** [Nethermind](https://www.nethermind.io/blog/eip-7702-attack-surfaces-what-developers-should-know) · [Halborn](https://www.halborn.com/blog/post/eip-7702-security-considerations) · [Zealynx](https://www.zealynx.io/research/smart-contracts/eip-7702-wallet-security) · [Cryptopolitan $1.54M phishing](https://www.cryptopolitan.com/eip-7702-user-loses-1-54m-phishing-attack/) · [CryptoTimes DxSale](https://www.cryptotimes.io/2026/06/02/tesseradao-tsr-plunges-99-after-attacker-mints-99m-tokens/)

### 2.2 Automated Scanning Campaign: Uninitialized ERC1967 Proxies

OWASP's February 2026 Smart Contract Top 10 added **Proxy & Upgradeability Vulnerabilities (SC10)** as a new category — the first new entry since 2023. The stated reason: uninitialized ERC1967 proxies became a *systematized automated scanning attack* in 2025-2026, not a one-off.

The pattern: bots continuously scan for any proxy contract where the `_initialized` storage slot (slot 0 in OpenZeppelin Upgradeable contracts) reads `0`. A zero value means no one has called `initialize()` yet, meaning any address can call it and take ownership of the proxy. Newer deployments that haven't run their post-deployment initialization script within minutes of deployment are the primary targets.

The shift from "manual hunt" to "automated bot campaign" is the signal. Protocols that deployed proxy-based contracts recently and hadn't secured initialization within the first few minutes of deployment are already at risk. Check [OWASP SC10:2026](https://owasp.org/www-project-smart-contract-top-10/2026/en/src/SC10-proxy-and-upgradeability-vulnerabilities.html).

## 3. 🟢 Advisories / CVEs

### 3.1 🚨 Lazarus Group "Mach-O Man" macOS Malware — Active Campaign

- **Disclosed:** April 21, 2026 by CertIK's Skynet team and Bitso's Quetzal Team via ANY.RUN sandbox
- **Target:** Crypto/fintech executives and protocol team members using macOS
- **Attack chain:**
  1. Attacker sends urgent Telegram DM from a compromised or spoofed account with a Zoom/Teams/Google Meet invite
  2. Victim's browser loads a fake "fixing your connection" page (ClickFix social engineering)
  3. Victim pastes a terminal command to "fix audio/video" — installs a modular Mach-O binary kit
  4. 4-stage payload: fingerprinting → persistence → credential harvest (macOS Keychain, browser cookies, 1Password vaults, crypto wallet files) → exfiltration with self-deletion
- **Attribution:** Lazarus Group / DPRK's Chollima division. Chainalysis attributes 76% of all 2026 crypto-hack value to DPRK actors. The Mach-O Man kit is linked to the pre-hack foothold phase in the Drift ($285M, Jan 2026) and KelpDAO ($292M, Apr 2026) campaigns.
- **Relevance:** If your target protocol's team members are on macOS, the deployer/admin keys on those machines are considered at-risk. Key compromise is the #1 root cause vector in large 2026 incidents — the Mach-O Man kit is a primary delivery mechanism.
- **Action for hunters:** When reviewing protocols, check whether multisig or admin key ceremonies were conducted remotely via video call. Flag any protocol where a single EOA admin key lives on a developer's MacBook without hardware wallet backup as critical key-management risk.
- **Sources:** [CoinDesk](https://www.coindesk.com/tech/2026/04/22/lazarus-group-has-become-especially-dangerous-with-new-mach-o-man-attack-certik) · [CybersecurityNews](https://cybersecuritynews.com/mach-o-man-macos-malware-lazarus/) · [Cryptopolitan](https://www.cryptopolitan.com/lazarus-group-malware-kit-macos-crypto-execs/) · [Bitcoin News](https://news.bitcoin.com/mach-o-man-malware-steals-macos-keychain-data-in-lazarus-group-crypto-campaign/)

### 3.2 Supply Chain: Malicious npm Packages Targeting Ethereum Dev Tools

- **Campaign status:** Active and ongoing as of June 2026
- **Packages confirmed malicious:** `@nomisfoundation/hardhat-configure`, `@monicfoundation/hardhat-config`, and similar typosquat variants mimicking legitimate Hardhat/Nomic Foundation packages
- **Payload:** On install, these packages exfiltrate private keys, mnemonics, Hardhat `.env` files, and any configuration data containing API keys or RPC endpoints to attacker-controlled servers
- **Vector:** Developers searching for Hardhat plugins on npm install lookalike packages, especially common in automated CI/CD `npm install` steps that pull latest or unversioned plugins
- **Affected surface:** Any CI pipeline, dev machine, or deployment script that runs `npm install` on Hardhat plugin names without exact version pinning
- **Action:** Audit `package.json` and CI scripts for unverified plugin names. Prefer exact semantic versions and verify the publisher is `@nomicfoundation` (not `@nomisfoundation` or similar). Any machine that ran `npm install` with an unvetted Hardhat plugin variant should be treated as compromised — rotate all keys that touched that environment.
- **Sources:** [Socket.dev investigation](https://socket.dev/blog/malicious-npm-campaign-targets-ethereum-developers) · [Security Affairs](https://securityaffairs.com/172671/malware/malicious-npm-packages-target-ethereum-developers.html)

## 4. 🎯 Hunt Hooks

**Hook A — EIP-7702 BatchCall/Executor caller whitelist (from EIP-7702 attack map)**
In any contract exposing a `batch()`, `execute()`, or `multicall()` function that is meant to be called only by the owning EOA: verify that `msg.sender` is explicitly validated before execution. The QNT root cause was simply that `BatchCall.batch()` had no `require(msg.sender == owner || authorized[msg.sender])` guard. Any delegated executor contract that lacks this check is drainable by any external caller who knows the interface. Priority targets: any protocol deployed or updated after the April 2026 Pectra fork.

**Hook B — ERC1967 proxy `_initialized` slot scan (from uninitialized proxy campaign)**
Query the `_initialized` storage slot (slot 0x0) on every proxy contract your target protocol deploys. In Foundry: `cast storage <proxy_addr> 0`. A return value of `0` means `initialize()` is callable by anyone and you can take ownership of the entire contract. Scan especially any proxies deployed in the last 30 days. Automated bots are already doing this at scale — but you can move faster on specific protocols you're hunting.

**Hook C — Admin key + EIP-7702 delegation combo risk (from DxSale pattern)**
Look for protocols where a single EOA acts as both admin/owner AND has made any EIP-7702 delegations (visible in Etherscan under the account's delegation tab post-Pectra). If that EOA's authorization tuple has ever been signed for a delegate contract that allows batch-execution with no caller guard, an attacker who compromises the key can atomically trigger operations across hundreds of subsidiary contracts in one transaction. Check protocols with many sub-pools or LP positions (like DEX aggregators, yield vaults, launchpads) — the blast radius when this pattern is present is the entire pool ecosystem.
