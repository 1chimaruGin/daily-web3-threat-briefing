## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Alephium TokenBridge — Forged VAA via Fallback-Mode Guardian Injection *(On-Chain Report Published June 3)*

- **Exploit date:** May 30, 2026 | **Forensic report:** June 3, 2026
- **Chain:** Ethereum ↔ Alephium (Wormhole-derived bridge)
- **Amount:** ~$815K drained; 13.76M unbacked wALPH minted (>100% of circulating supply at time of attack); 500K wALPH sold on Uniswap before team burn
- **Root cause — full kill chain:**
  1. Attacker deployed a fake contract on Ethereum emitting crafted `LOG7` events that mimicked the syntactic structure of legitimate Wormhole bridge message events.
  2. The fake contract's payloads referenced attacker-controlled destination addresses while appearing structurally valid to guardian node log watchers.
  3. Attacker **temporarily disrupted guardian node connectivity** (exact method not disclosed — likely targeted network interference during a maintenance window), forcing the bridge into **reduced-quorum fallback mode**, which lowers the signing threshold for VAA approval.
  4. Under fallback mode, guardian nodes signed the forged messages. The resulting **Verified Action Approvals (VAAs)** passed the on-chain threshold check.
  5. Forged VAAs submitted to the bridge contract → bridge released $815K in real USDC/USDT/WETH/WBTC from Ethereum escrow + minted 13.76M wALPH on Alephium-side.
  6. Team burned the remaining unbacked wALPH in the attacker's wallet on June 2; 500K wALPH sold prior to burn remain backed by protocol-supplied native ALPH.
- **Critical insight — this is NOT a key theft:** Guardian private keys were never compromised. The attacker exploited the **gap between how guardians authenticate Wormhole events (by log structure) and how the fallback mode relaxes quorum requirements under connectivity stress.** This is a new class of bridge-DVN attack: exploit the verifier network's *failure modes* rather than steal its keys.
- **Bug class:** `bridge-dvn` (fallback quorum bypass), `signature-replay` (forged VAA accepted as authentic)

### 2.2 Gravity Bridge — Revised Root Cause: IBC Denom Mapping Poisoning

- **Background:** The June 12 briefing covered Gravity Bridge's $5.4M drain primarily as a "validator key compromise." Updated analysis from rekt.news and subsequent reporting frames the attack differently.
- **Revised root cause:** The attacker minted **worthless tokens on Osmosis** and **poisoned the IBC denom registry** with fabricated denom strings designed to resolve as high-value assets in Gravity Bridge's relayer validation. The bridge's token acceptance logic trusted the relayer-supplied denom field without cryptographically verifying it against the origin chain's canonical `x/bank` denom registry. By spoofing the denom mapping, the attacker withdrew real USDC, WETH, and USDT from Gravity's Ethereum-side escrow against synthetic denom claims.
- **Bug class (revised):** `bridge-dvn` (unverified denom source), `unverified-contract` (denom registry not bound to canonical ICS-20 asset records) — **not** primarily `key-management`
- **Where it applies:** Any Cosmos-ecosystem bridge (Gravity, Axelar, IBC-enabled DEXes) that accepts IBC denom strings from relayers without reverse-verification against the canonical asset registry on the originating chain. If a bridge's asset registry can be updated by a relayer or governance action without time-locking, it is exploitable via this pattern.

## 4. 🎯 Hunt Hooks

**Hook 1 — Probe bridge fallback modes for guardian quorum drift.**
For any Wormhole-derived, IBC, or LayerZero bridge in your scope: determine what happens when guardian / DVN / oracle connectivity is degraded. Specifically: (a) Does the bridge define a reduced-quorum "fallback mode" that lowers the signature threshold? (b) Is the connectivity failure detectable from outside (e.g., by timing attacks during scheduled upgrades or guardian rotation events)? (c) Does the bridge validate that VAA source addresses match the canonical Wormhole Core Contract, or does it accept any conformant log structure from any address? The Alephium kill chain is the reference PoC: `LOG7` events from a fake contract passed guardian validation under fallback conditions. Any bridge with reduced-quorum fallback is a live target for this pattern — report to the protocol's bug bounty before demonstrating on-chain.

**Hook 2 — Audit IBC denom registry trust in Cosmos bridges.**
For any protocol bridging assets between Cosmos chains (Osmosis, Injective, dYdX) and EVM chains via Gravity / Axelar / IBC wrappers: check how the bridge's relayer logic validates the `denom` field in ICS-20 packets. Red flags to grep for: (a) denom string accepted from IBC packet without lookup against origin chain's `x/bank` module canonical records; (b) asset registry mapping controlled by a mutable contract that can be updated by a relayer without governance delay; (c) new denom registrations accepted with no time-lock or whitelist. The Gravity Bridge incident demonstrates that poisoning the denom mapping lets an attacker withdraw real assets against synthetic claims — a critical finding on any Cosmos bridge with this architecture.

**Hook 3 — Prioritize Aave V4 on Sherlock before competition intensifies.**
Aave V4 at $2.5M critical bounty via Sherlock is the best risk-adjusted opportunity in the market this week. V4's novel components vs. V3: the **Liquidity Layer** (unified pool routing), **position hook architecture** (external contracts called during borrow/repay), and **cross-chain yield routing**. Starting points: (a) Can a malicious hook contract reenter the Aave pool during a borrow or repay callback before internal accounting updates? (b) Does the hook registration whitelist enforce that hook contracts are verified and non-upgradeable? (c) Does the cross-chain yield router validate slippage bounds per-chain before initiating a transfer, or does it assume the downstream chain's price is in sync? Access control in the hook registry is the highest-probability critical-class finding.
