## 1. 🔴 Active Exploits (last 24h)

### 1.1 Polymarket — $3M Frontend Supply-Chain Injection (June 25–26, 2026)

| Field | Detail |
|---|---|
| **Protocol** | Polymarket (prediction market) |
| **Chain** | Polygon → Ethereum (bridge for fund movement) |
| **Amount** | ~$2.94M (pUSD, USDC-backed stablecoin) |
| **Wallets drained** | < 15 user wallets |
| **Bug class** | supply-chain |
| **Status** | Polymarket contained the issue; full user reimbursement confirmed |

**Attack anatomy:**

1. An attacker compromised a **third-party CDN/vendor** supplying JavaScript to the Polymarket frontend.
2. The compromised vendor delivered a **malicious script** to users visiting the site during the attack window (June 25 UTC morning).
3. The injected JS intercepted wallet interactions and **silently drained pUSD** (Polymarket's USDC-backed stablecoin) as users connected their wallets or interacted with the UI.
4. The attacker bridged the stolen pUSD from Polygon to Ethereum and swapped everything into approximately **1,893 ETH**, consolidating funds into a single EOA.
5. No smart contract vulnerability was involved — the on-chain protocol functioned as designed. The entire attack lived in the supply chain.

**Key context:** This is Polymarket's **second breach in under two months**. In May 2026 the platform lost ~$700K from an operations wallet due to a six-year-old private key compromise. The repeat nature suggests systemic infrastructure security gaps beyond the typical smart contract audit scope.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 CVE-2026-12866 — expr-eval `toJSFunction()` RCE, CVSS 9.8 (All Versions Affected)

- **Published:** Late June 2026
- **Source:** [threat-modeling.com CVE-2026-12866 advisory](https://threat-modeling.com/cve-2026-12866-expr-eval-javascript-code-execution/) · [BleepingComputer — expr-eval RCE](https://www.bleepingcomputer.com/news/security/popular-javascript-library-expr-eval-vulnerable-to-rce-flaw/)

**What it is:**

`expr-eval` is a JavaScript expression parser/evaluator (~800K npm weekly downloads) heavily used in **DAO governance portals** to calculate voting power from user-supplied formulas. CVE-2026-12866 (CVSS 9.8) allows an attacker to craft a malicious expression passed to `toJSFunction()` that escapes the evaluator sandbox and executes **arbitrary OS-level code** on the server.

**Web3-specific threat model:**

When `expr-eval` runs on a governance frontend server, exploitation enables:
- RCE on the server process, which typically holds **session tokens, private keys for relayers, or admin API credentials**
- Exfiltration of cached wallet session keys (localStorage-derived) through server-side log/state access
- Serving a modified governance proposal ABI to users, enabling a **malicious-proposal approval** at the wallet-signing step

**Affected surface:** Any DAO governance UI, voting dashboard, or quorum-calculation service using `expr-eval` (any version) to evaluate user-supplied or parameter-driven expressions.

**Mitigation:** No upstream patch yet for the `expr-eval` npm package itself. Switch immediately to `expr-eval-fork ≥ 3.0.0`, which enforces a function allowlist and blocks the `toJSFunction()` escape path. Alternatively, remove user-supplied expression inputs entirely from backend evaluation paths.

**Related prior CVE:** CVE-2025-12735 (CVSS 9.8, the `evaluate()` sandbox bypass) was patched in 2025; CVE-2026-12866 targets a **separate** entrypoint (`toJSFunction`) and affects projects that believed they had already patched by avoiding `evaluate()` with raw user input.

### 2.2 Supply-Chain Attack Pattern: CDN Vendor Compromise → Wallet Drainer

The Polymarket incident is the clearest 2026 case of what has become a repeating pattern across Web3 frontends:

1. **Attacker targets a shared CDN or analytics vendor** (not the protocol itself)
2. **Malicious JS delivered via compromised CDN tag** — no CORS, CSP, or SRI check on the consuming site catches it in time if the CDN domain itself is trusted
3. **Script waits for wallet-connect events**, then clones or intercepts the signing flow
4. **Funds drained silently** — users see normal UI; no on-chain anomaly until post-facto

**Structural gap:** Polymarket (and many DeFi frontends) do not use **Subresource Integrity (SRI) hashes** for third-party scripts. SRI allows browsers to verify that the script delivered by a CDN matches a hash baked into the `<script>` tag at deploy time — any tampering causes the browser to refuse execution.

**Sources:** [BleepingComputer](https://www.bleepingcomputer.com/news/security/polymarket-customers-lose-3-million-in-supply-chain-attack/) · [Cryip analysis](https://cryip.co/polymarket-frontend-hack-third-party-vendor-3-million-june-2026/)

## 4. 🎯 Hunt Hooks

**Hook A — Frontend Dependency Audit: SRI Hashes on Third-Party Scripts**

The Polymarket supply-chain attack (and the broader CDN-vendor-compromise pattern) exploits a near-universal missing control in Web3 frontends. For any target in scope:

1. Open the production `index.html` (or equivalent). For every `<script src="...">` or `<link rel="stylesheet">` loading from an external domain, check whether a `integrity="sha384-..."` SRI attribute is present.
2. Flag **any external CDN tag without SRI** — common culprits: analytics snippets, customer support widgets, A/B test scripts, social login buttons.
3. Check whether the site enforces a **Content-Security-Policy** header with `require-sri-for script`. If not, SRI is advisory-only and a compromised CDN domain bypasses it entirely.
4. Verify if the site's **CSP `script-src`** includes wildcards (`*.amazonaws.com`, `*.cloudflare.com`) — these are functionally equivalent to no CSP for supply-chain purposes.
5. Report class: **Insufficient Subresource Integrity / supply-chain injection vector** — typically Medium to Critical depending on what the frontend can access (wallets, session keys, governance proposals).

---

**Hook B — expr-eval Usage in DAO Governance Portals**

CVE-2026-12866's CVSS-9.8 rating means any DAO governance portal using expr-eval for voting-power formulas is a near-critical bounty candidate:

1. Check `package.json` / `package-lock.json` / `yarn.lock` in the governance portal repo for `expr-eval` as a direct or transitive dependency.
2. Grep the source for `toJSFunction(`, `.evaluate(`, `Parser.parse(` — any of these called with user-supplied or parameter-derived input are in-scope.
3. If the governance portal runs server-side (SSR/Next.js/Node.js), the RCE is against the server process — trivially Critical (full server takeover, key exfiltration).
4. If purely client-side, scope the impact to cross-site scripting, governance-proposal tampering, and wallet-drain UX.
5. Confirm remediation: `expr-eval-fork ≥ 3.0.0` must be the resolved dependency (not just a dev override).

---

**Hook C — Repeat-Breach OPSEC: Private Key Lifecycle on Multi-Breach Protocols**

Polymarket's two breaches in two months (May: old private key; June: CDN vendor) share a root pattern: **no systematic vendor/key lifecycle management**. This is increasingly exploitable across other protocols:

1. For any target that has disclosed a prior breach: pull the post-mortem. Did they address **only** the specific compromised component, or did they do a full key rotation + vendor audit?
2. Check the protocol's current infrastructure for **residual exposure**: old deployer wallets with remaining balances, admin keys shared across team members, CDN vendors without contract-level integrity controls.
3. A repeat-breach protocol is 3–4× more likely to have a third unclosed vector — the security team is reactive, not proactive. Treat any prior breach disclosure as a signal to dig deeper, not as evidence the problem is fixed.
4. On-chain: look for **admin/timelock key reuse** between the old compromised wallet and the new setup. Attackers frequently find that "key rotation" only rotated one of several privileged roles.
