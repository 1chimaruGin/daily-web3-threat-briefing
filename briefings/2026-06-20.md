## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 The Abandoned-Contract Drain Pattern — Formalized

Four of June's incidents share a structural DNA that now has enough instances to call a pattern: **deprecated/abandoned smart contracts with residual funds and no active pause authority**.

| Incident | Abandoned Since | Attacker Action | Loss |
|----------|----------------|-----------------|------|
| DxSale (Jun 2) | 2021 (admin key unrotated) | Exploited hidden backdoor in 5-year-old locker | $7.3M |
| TesseraDAO (Jun 2) | Weeks post-launch | Ownership taken via admin key theft | $2.5M |
| Aztec Connect (Jun 14) | Mar 2023 (official shutdown) | Exploited immutable ZK proof-boundary gap | $2.1M |
| Aztec Connect (Jun 17) | Mar 2023 | Second deprecated sub-contract drained | $2.1M |

**The kill chain is the same in every case:**

```
Step 1: Protocol shuts down (or is abandoned), admin keys rotated away / renounced
Step 2: Smart contracts remain deployed — immutable, unpaused, holding residual liquidity
Step 3: No active team monitoring deprecated address space
Step 4: Attacker reads storage, identifies residual balance > gas cost to exploit
Step 5: Exploit the original vulnerability — no patch is possible, no pause authority exists
Step 6: Drain, bridge out (often via Tornado Cash or cross-chain hop)
```

**Why this matters for hunters:** A deprecated contract that was too small to attack at peak TVL (when whitehats and blackhats were both watching) becomes a profitable target years later once monitoring stops. If a protocol has $5M+ residual in an immutable V1 contract, it is a live target — and the bug bounty program probably still covers it.

**Sources:** [DxSale backdoor analysis](https://crypto.news/dxsale-exploit-drains-7-3m-in-bnb-through-hidden-contract-backdoor/) · [Aztec Connect drain](https://coinjournal.net/news/aztec-network-loses-over-4-million-in-three-days-to-two-subsequent-hacks/) · [Aztec Labs statement](https://www.cryptopolitan.com/aztec-labs-product-2-1m-exploit/)

### 2.2 OWASP SC Top 10:2026 — Chained Attack Grammar

OWASP published **Smart Contract Top 10:2026** (Feb 18, 2026, led by CredShields), built on 122 deduplicated incidents from 2025 totalling **$905.4M in smart-contract-only losses**. Two structural observations for hunters:

**New entry: SC10 — Proxy & Upgradeability Vulnerabilities**

The 2026 list formally adds a standalone proxy category. The recurring kill chain OWASP documents:

```
Uninitialized proxy  →  attacker calls initialize(attacker)  →  full ownership
Single-sig proxy admin  →  1 key compromise = full upgrade authority
Missing storage gap  →  upgrade collides with inherited storage layout
```

OWASP cites automated bot campaigns that continuously scan for `_initialized == 0` in storage slot 0 of newly deployed ERC1967 proxies. Deployment windows measured in blocks, not minutes.

**SC03 — Oracle & Price Manipulation** remains a top-3 category. The notable 2026 evolution: attackers now chain flash loans *with* oracle manipulation *with* governance bypass in a single tx, rather than using each technique independently. CertiK and Hacken both document this in recent writeups: a single flashloan callback can (a) inflate spot price, (b) trigger an oracle-dependent liquidation, and (c) redeem a stale governance snapshot to approve the exfiltration — all atomically.

**Sources:** [OWASP SC Top 10:2026](https://scs.owasp.org/sctop10/) · [CredShields announcement](https://chainwire.org/2026/02/18/credshields-leads-owasp-smart-contract-top-10-2026-to-raise-smart-contract-security-standards/) · [CyberSecurityNews coverage](https://cybersecuritynews.com/owasp-smart-contract-top-10-2026/)

### 2.3 Compromised Accounts Overtake Code Bugs as Primary Loss Vector in 2026

Broader 2026 context now confirmed by Chainalysis and multiple research outlets: **>50% of DeFi incident losses in H1 2026 stem from compromised keys/accounts**, not flawed smart contract logic. The Humanity Protocol ($36M, Jun 9), June 14 bridge ($127M), and Gravity Bridge ($5.4M, Jun 4) all share the same root cause vector.

For hunters: if you're targeting a protocol with a large bug bounty, the largest payout opportunities may now be in **key ceremony auditing, admin multisig configuration, and validator set hygiene** — not just Solidity logic bugs. Protocols running 5-of-9 multisigs with no HSM enforcement on validator nodes are critical-risk regardless of contract quality.

## 4. 🎯 Hunt Hooks

**Hook A — Scan for deprecated/residual contracts in your target's address space**
For any protocol you're hunting: pull their full deployment history from Etherscan/BscScan/Solscan. Look for V1, V2, or "legacy" contracts that are still deployed but marked inactive in docs. Check their ETH/token balances: `eth_getBalance` + ERC-20 `balanceOf`. If a "deprecated" contract holds >$50K and there is no `pause()` function, or the owner is `address(0)` (renounced), write up a proof-of-concept drain — this is a live bug bounty finding. Document whether the original audit covered the deprecated version.

**Hook B — Proxy admin single-sig check (SC10 hunt)**
For every upgradeable contract in your target: call `StorageSlot.getAddressSlot(EIP1967_ADMIN_SLOT).value` (slot `0xb53127684a568b3173ae13b9f8a6016e243e63b6eb8ee8e4f3f1b4e6a3d8e44`). If the result is an EOA rather than a multisig, flag it. If it's a multisig, check the threshold — anything less than 3-of-n on a protocol with $1M+ TVL is a plausible `upgradeability` / `access-control` critical. Cross-reference against the protocol's published security model.

**Hook C — Oracle source and liquidity depth for new collateral assets**
When a lending protocol adds a new collateral type, check immediately: (1) what oracle feeds the price — spot TWAP, Chainlink, or a custom adapter? (2) what is the on-chain liquidity depth for that asset at the main DEX pair? If TVL exceeds 10% of the DEX liquidity depth, a flash-loan oracle attack is economically viable. This was the structural flaw in the KelpDAO ($292M, Apr 19) and Moonwell (earlier 2026) incidents. Use Dune Analytics or on-chain queries to validate liquidity before the oracle can be manipulated.
