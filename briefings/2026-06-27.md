## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 DARKNAVY: Aztec escapeHatch() — Definitive ZK Witness-Binding Gap Analysis

- **Published:** ~June 25–27, 2026
- **Source:** [DARKNAVY Blog — Aztec Private Rollup Bridge Escape-Hatch Claim-Proof Drain](https://www.darknavy.org/web3/exploits/aztec-private-rollup-bridge-escape-hatch-claim-proof-drain/)
- **Incident being analyzed:** June 17, 2026 — Aztec RollupProcessor / Private Rollup Bridge, 1,158 ETH + 150,000 DAI + 0.47 renBTC (~$2.21M)

**Root cause (circuit-level):**

The `escapeHatch()` function in Aztec's TurboPlonk circuit contains a **missing equality constraint** between two witnesses for `old_data_root`:

```
Witness A (private): used for membership proof inside the join-split component
Witness B (public input): the root checked by the L1 Solidity verifier contract
```

Because no constraint binds A = B, the prover is free to set them independently. The attacker:

1. Set **Witness B** (public) to a valid on-chain Merkle root (so the Solidity verifier's `require(root == expectedRoot)` passes)
2. Set **Witness A** (private) to a *fake* Merkle root they constructed themselves
3. Proved ownership of fabricated notes against the fake private root
4. The L1 contract saw a valid on-chain root, approved the proof, and released 1,158 ETH to the attacker's EOA

**Why this generalizes beyond Aztec:**

The pattern — *distinct private vs. public witnesses for the same value, with no binding constraint enforced in the circuit* — is a structural risk in **any ZK system** where:
- An emergency / escape-hatch path takes user-supplied proof inputs (bypassing the normal sequencer)
- The verifier contract accepts a public root as a parameter without independently querying the canonical state
- The circuit was designed to trust the prover to supply consistent inputs (a known anti-pattern in ZKP design)

**Affected designs to audit:**
- Any bridge or rollup with `escapeHatch()`, `forceWithdraw()`, `emergencyExit()`, or `claimByProof()` functions
- ZK protocols where the verifier's public inputs are partly caller-supplied rather than derived on-chain
- Deprecated / immutable ZK contracts that were never patched after initial deployment

## 4. 🎯 Hunt Hooks

**Hook A — ZK Escape Hatch: Audit Witness Binding in Emergency Withdrawal Circuits**

The Aztec witness-binding gap (Section 2.1) is the single most actionable ZK bug class this month. For any ZK rollup or bridge in scope:

1. **Locate the emergency path:** Find `escapeHatch()`, `forceWithdraw()`, `emergencyExit()`, `claimByProof()`, or equivalent. Any function that lets a user submit a ZK proof directly (bypassing the sequencer) is in scope.
2. **Check public input sourcing:** Does the Solidity verifier derive the public root from canonical on-chain state (e.g., `currentRoot = latestRootFromContract()`), or does it accept `publicRoot` as a **caller-supplied parameter**? Caller-supplied = red flag.
3. **Check circuit constraint:** Does the circuit contain an explicit `public_input_root === private_merkle_root` constraint? If not (i.e., the private and public roots are independent witnesses), an attacker can split them — proving against a fake tree while the L1 sees a valid root.
4. **Deprecated/immutable targets:** Check if the protocol has any deprecated contracts still holding funds. If they're immutable and were audited years ago, ZK circuit-level bugs were likely not in scope for those audits.

**Critical severity** if exploitable: entire bridge balance drained in one transaction.

---

**Hook B — Abandoned-Contract Inventory Check (June 2026 Pattern)**

Four of June's 16 incidents involved contracts that were deprecated 1–4 years before the exploit but still held funds (Aztec Connect, Aztec Private Rollup Bridge, DxSale legacy lockers, JaredFromSubway MEV v1 wallets). For any protocol in scope:

1. Pull the deployer address and list all contracts it deployed. Flag any contract that has not received a `call` or `delegatecall` in > 6 months but has a non-zero ETH/token balance.
2. For each such contract: read its source code (if verified) and check for `escapeHatch()`, `adminWithdraw()`, `setFee()`, or any privileged function that hasn't been removed and whose access control is now stale.
3. Cross-check with the protocol's documentation — does it publicly acknowledge the deprecated contract? If not, it's almost certainly not in scope for bug bounty… or is it? Many programs have expanded scope retroactively when researchers found abandoned-contract drains.
4. Check if any deprecated contract is still referenced by the active protocol's UI, routing, or governance contracts — that's an active-scope argument regardless of official deprecation status.

---

**Hook C — Key-Management Attack Surface: Operator vs. Deployer Separation**

With 50%+ of Q2 2026 incidents tracing to key compromise, the next audit cycle should include:

1. **Multisig structure review:** How many signers on the protocol's timelock/admin multisig? Are the signers identified on-chain and are their hardware wallet procedures documented? A 2-of-3 Gnosis Safe with one signer on a shared laptop is not materially safer than a 1-of-1.
2. **Bridge DVN key inventory:** For any LayerZero/Axelar/Wormhole-based bridge: how many DVN signers are there, and what is the threshold? A single-DVN bridge with a software key is the Taiko pattern.
3. **GitHub Actions / CI secret exposure:** Run `git log --all --full-history -- "*.env" "*.pem" "*.key" "*secret*" "*private*"` against any in-scope repo — look for key material ever committed, even if deleted. GitHub's native secret scanning misses custom key formats. The Taiko SGX key was discoverable this way.
