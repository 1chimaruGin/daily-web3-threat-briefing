## 1. 🔴 Active Exploits (last 24–48h)

### 1.1 SecondFi (Cardano) — $2.4M Confirmed, Up to $20M at Risk — Wallet Key Generation Vulnerability (June 24, 2026)

| Field | Detail |
|---|---|
| **Protocol** | SecondFi (Cardano self-custody wallet) |
| **Chain** | Cardano |
| **Confirmed loss** | ~$2.4M (16M ADA across 374 wallets) |
| **Worst-case** | Up to $20M+ (SlowMist estimate; independent audit pending) |
| **Bug class** | key-management |
| **Status** | Patch deployed for unaffected wallets; no reimbursement plan announced |

**What happened:**

SecondFi's proprietary wallet-address generation software contained a critical flaw that exposed cryptographic material at the **address derivation layer** — not the seed phrase or mnemonic layer. Three separate attackers drained 16M ADA ($2.4M) from 374 user wallets before the team detected the breach.

The SecondFi team intercepted and custodied an additional **129M ADA** (~$20M) before it was drained. That capital is now in third-party custody and constitutes the "up to $20M at risk" figure cited in SlowMist's initial warning.

**Critical technical distinction — why seed phrases don't save you:**

> *"The vulnerability activates at the address level when a transaction is signed. Moving a seed phrase to a different wallet does not change the compromised address behaviour."*
> — SecondFi incident notice

The bug lives in the address-derivation step, downstream of the mnemonic. Users who simply import their recovery phrase into a different wallet app still have compromised addresses; they must migrate funds to a **freshly generated address** using unaffected software. This is a rare but extremely dangerous class of bug — auditors who test seed phrase → key derivation often miss flaws at the subsequent address-encoding step.

**Response:** Patch rolled out for unaffected users. 374 affected wallets confirmed. Independent audit by a blockchain security firm is ongoing. No reimbursement path announced as of June 29 UTC.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Q2 2026 Closes: 83 Incidents, $775M Stolen — Record Quarter Analysis

Q2 2026 (April–June) is now confirmed as the **most-hacked quarter ever by incident count**: 83 separate attacks, $775M stolen across DeFi, bridges, and infrastructure.

**Key breakdown:**
- **Bridge/cross-chain attacks** accounted for ~$351M (46% of Q2 losses) — highest single attack-vector concentration on record
- **Lazarus Group** (DPRK-linked) attributed to KelpDAO ($293M) and Drift ($280M) via social engineering + infrastructure compromise — two incidents alone account for 73% of Q2 dollar losses
- **Incident frequency:** ~1 hack per day on average across Q2
- **TVL impact:** DeFi TVL fell from $164B to ~$71.7B across the quarter (partly driven by capital flight post-exploits, not solely losses)
- **June alone:** $74.9M across 29 incidents — the "long tail" of frequent smaller exploits

**Pattern shift:** For the first time, compromised accounts (social engineering, key theft, insider threat) account for **>50% of attacks by incident count**, overtaking traditional smart-contract bugs. This matches the SecondFi and Humanity Protocol cases this month.

### 2.2 Wallet Address-Derivation Bugs: An Under-Audited Attack Surface

The SecondFi case highlights a class of vulnerability that receives far less attention than smart-contract logic bugs: **flaws in wallet address generation software**.

Standard audit scope covers:
- Seed phrase generation entropy
- BIP-32/BIP-39/BIP-44 key derivation correctness
- Private key storage

Under-audited areas that SecondFi exposes:
- **Address encoding step** (e.g. Bech32 for Cardano, checksum derivation for EVM) — a bug here can produce addresses that appear valid but leak entropy to an attacker
- **Custom address generation wrappers** that deviate from the reference implementation
- **Hardware acceleration paths** that may produce subtly different outputs than the pure-software reference

**Why this matters for hunters:** Any wallet SDK, browser extension, or mobile app that implements custom address generation (rather than using battle-tested libraries verbatim) is a candidate target. The vulnerability class is near-impossible for users to detect before being drained.

## 4. 🎯 Hunt Hooks

**Hook A — Custom Wallet Address-Generation Code (high-signal for non-EVM chains)**

The SecondFi exploit targeted the address-derivation layer — a blind spot in most audits. For any in-scope wallet app, SDK, or browser extension:

1. Identify whether the project uses a **reference implementation** (e.g. `@emurgo/cardano-serialization-lib` for Cardano, `bitcoinjs-lib` for Bitcoin, `ethers.js` for EVM) or a **custom wrapper/fork**.
2. If custom: diff the address-encoding step against the reference. Look for: non-standard entropy sources, custom Bech32/checksum encoding, or any step between raw private key and final address string.
3. Test with known-good vectors from BIP-32/BIP-39 test vectors — a mismatch is an instant finding.
4. Check whether the software produces **deterministic addresses** for the same seed (non-determinism = entropy leak).
5. Bug class: **key-management / cryptographic implementation error** — typically Critical or High.

---

**Hook B — MEV Bot Approval Accumulation: Approval-Farming Attack Vector**

The JaredFromSubway exploit worked by gradually accumulating token approvals through fake profitable trades. This pattern is reproducible against any on-chain bot that:

1. **Automatically grants approvals** to contracts it interacts with (rather than one-per-swap approvals)
2. Has no **approval expiry or revocation logic**
3. Does not **validate counterparty contracts** against an allowlist before granting max-approval

To hunt: find MEV bots, keepers, or aggregator routers with large outstanding approvals. Check `Approval` event logs for the contract — if a single contract has max-allowance approvals from many addresses, it is either legitimate (DEX router) or a pending drain. Any bot contract granting unlimited approvals to external addresses is reportable.

---

**Hook C — Bridge DVN Single-Point-of-Failure (Q2 2026 Meta-Trend)**

46% of Q2 losses came from bridge/cross-chain attacks. The recurring pattern across Kelp, Secret Network/Axelar, and Taiko is a **single verifier or signer** controlling bridge message acceptance. For any bridge in scope:

1. Find the message verification logic — typically a `verify()` or `processMessage()` function.
2. Count how many independent signers/DVNs must agree for a message to be accepted. If the answer is 1, or if there is a fast-path bypass that requires only 1, that's a Critical finding.
3. Check whether the set of approved DVNs is upgradeable — if an admin can swap in a malicious DVN without a timelock, the protocol is one admin key away from total drain.
4. LayerZero-based bridges: verify the `requiredDVNs` array in `SetConfigParam` events — protocols frequently under-configure this after deployment.
