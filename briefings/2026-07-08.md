## 1. 🔴 Active Exploits (last 24h)

### 1.1 BonkDAO — $20M Governance Takeover (Solana)

**Date:** July 6–7, 2026 (confirmed/reported July 7)
**Chain:** Solana
**Protocol:** BonkDAO community treasury (BONK meme coin governance)
**Amount drained:** ~4.426 trillion BONK ≈ **$19.3–20M**
**Attack cost:** ~$4.4M (tokens acquired to reach quorum, partially recoverable via open market sale)
**Bug class:** `access-control`, `logic-error`

**Attack sequence:**

1. June 30: Anonymous wallet submits BIP #76 — "Sowellian BonkDAO" — with boilerplate social text ("rebuild from the ashes, monetize holdings") concealing a single operative line: *transfer 4.43 trillion BONK to wallet `...JHvQ`*.
2. July 4–5: Separate attacker wallet buys 88.23 billion BONK (~$4.4M) across Bybit and Binance, acquiring just over 1% of circulating supply — the exact quorum threshold required for proposal passage.
3. July 6 (vote executes): Only 7 wallets vote out of 18,000+ DAO members (2.9% participation). The attacker's wallet supplies 882.38B YES votes against an 879.95B quorum threshold — passes by the narrowest possible margin. No other faction has coordinated NO votes.
4. Smart contract automatically executes the embedded transfer. By 3:30 PM ET, funds move a second time from `...JHvQ` to `...eh42`, making tracing harder.

**Governance weaknesses exploited:**

| Weakness | Detail |
|---|---|
| Low quorum threshold | Only 1% of total supply required — economically reachable for ~$4.4M against a $20M treasury |
| No timelock | Proposal executed atomically with vote conclusion; no delay window for defenders to react |
| Liquid token eligibility | Unstaked, freely purchased tokens counted toward quorum immediately |
| Low participation culture | 7 of 18,000+ members voted; no alerting system for large YES swings |

**Key distinction:** No smart-contract exploit occurred. The attacker followed every governance rule perfectly. This is an economic attack on governance design, not a code bug.

### 1.2 Ethereum MEV Same-Block Backrun — $2M Trader Loss

**Date:** July 7, 2026 (~2:00 AM UTC)
**Chain:** Ethereum
**Protocol/Venue:** Uniswap v3 (AVAIL/WETH low-liquidity pool)
**Amount lost (victim):** ~$2.01M (1,126.44 ETH in → 5,776 LIT tokens out, worth ~$14,500 — 99.3% loss)
**Amount extracted (block builder):** $1.8M captured by **Titan block builder** as builder reward
**Bug class:** `front-running` (backrun variant), `logic-error` (routing failure)

**What happened:**

A trader submitted a large swap (1,126.44 ETH ≈ $2.01M) whose DEX aggregator route passed through an extremely thin-liquidity AVAIL/WETH pool on Uniswap v3. The swap executed at approximately **120× the sustainable AVAIL price**, destroying almost the entire value.

In the same block, Titan block builder extracted $1.8M via **same-block backrun arbitrage**: after the victim's imbalanced swap artificially spiked the AVAIL price, Titan's bot sold AVAIL back into equilibrium, capturing the spread. Security firm GoPlus Security classified this as a "real, highly imbalanced backrunner arbitrage" — distinguishing it from a classic sandwich attack because **no front-running trade preceded the victim's swap**. Titan has earned $112.6M in block-building revenue in 2026 per DefiLlama.

**Root cause:** The routing algorithm selected a pool with insufficient liquidity depth for a trade of this size, and no effective slippage protection blocked execution. The MEV extraction is a consequence, not a cause — but the $1.8M profit to Titan was entirely a function of the routing failure.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 DAO Governance Economic Attack Pattern — Generalizes from BonkDAO

**Classification:** Technique disclosure (no new CVE — governance design flaw)
**Chains at risk:** Any EVM or Solana DAO using token-weighted governance
**Bug class:** `access-control`, `logic-error`

The BonkDAO incident formalizes an attack class that has been theorized but rarely executed at scale: **governance economic attack via temporary vote acquisition**. The key economic invariant that determines exploitability:

```
Attack cost  <  Treasury value × quorum threshold
```

For BonkDAO: $4.4M cost vs. ($20M treasury × 1% quorum) = attack cost ≈ 22% of treasury. This remains economically rational because the attacker recovers 80%+ of token purchase cost by selling acquired tokens after the attack — net cost may be as low as $500K–$1M in slippage and fees.

**DAO parameters that create this attack surface:**

1. **Quorum < 5% with no minimum absolute vote count** — easy to accumulate in liquid markets
2. **No timelock between vote passage and execution** — prevents coordinated defensive responses
3. **Liquid/unstaked tokens eligible immediately** — bought on exchange → vote in next block
4. **No on-chain alerting for large YES-vote swings** — human response time exceeds attack window
5. **Low voter participation culture** — legitimate token holders don't actively monitor governance

**Where to hunt this in your targets:** Check on-chain governance parameters via their Governor contract's `quorumNumerator()`, `votingDelay()`, `votingPeriod()`, and `TimelockController` config (or Solana program equivalent). A DAO where `costToReachQuorum < 5% of treasury` with `executionDelay == 0` is an economic exploit waiting to happen.

### 2.2 Same-Block Backrun: Routing Vulnerabilities Enable Block-Builder MEV Extraction at Scale

**Classification:** Technique / MEV attack surface
**Chain:** Ethereum (all EVM chains with MEV-boost or PBS)
**Bug class:** `front-running` (backrun class)

The $2M Ethereum backrun reinforces an underappreciated attack surface in DEX aggregator routing: **routes through illiquid pools expose traders to deterministic same-block extraction**. Unlike sandwich attacks, backrun extraction is harder to prevent because:

1. No front-running trade occurs — only **post-trade arbitrage** by sophisticated block builders
2. Block builders with PBS (Proposer-Builder Separation) have **atomic same-block access** — they can guarantee their backrun transaction executes in the same block as the victim swap
3. Protection depends entirely on **pre-submission slippage guards** — if the aggregator route doesn't check pool depth before submission, no recovery is possible

**For auditors:** Any DEX aggregator, yield optimizer, or lending protocol that performs on-chain swaps as part of its logic is potentially subject to this if it routes through low-TVL pools. Look for `swapExactTokensForTokens` or `exactInput` calls routed through paths where the intermediate pool's TVL is orders of magnitude below the swap size.

**Source:** [Sandwiched and Silent: Behavioral Adaptation and Private Channel Exploitation in Ethereum MEV — arXiv](https://arxiv.org/pdf/2512.17602) *(background research on MEV extraction patterns)*

## 4. 🎯 Hunt Hooks

### Hook 1: Audit Your Target DAO's Governance Economics
*Tied to BonkDAO $20M drain (§1.1 and §2.1)*

For any protocol with a DAO governance component in your bug bounty scope, compute **governance attack cost** and compare to **treasury value**:

**Step 1 — Pull on-chain governor parameters:**
```bash
# For OZ Governor-style contracts
cast call <GOVERNOR_ADDR> "quorumNumerator()(uint256)" --rpc-url $RPC
cast call <GOVERNOR_ADDR> "votingDelay()(uint256)" --rpc-url $RPC
cast call <GOVERNOR_ADDR> "votingPeriod()(uint256)" --rpc-url $RPC
cast call <TIMELOCK_ADDR> "getMinDelay()(uint256)" --rpc-url $RPC
```

**Step 2 — Compute attack cost:**
```
quorum_tokens = total_supply × quorum_numerator / quorum_denominator
attack_cost_USD = quorum_tokens × current_token_price
treasury_USD = <fetch from DefiLlama or protocol docs>

attack_feasible = (attack_cost_USD < 0.15 × treasury_USD) AND (timelock_delay < 48h)
```

**Step 3 — Classify severity:**
- `timelock_delay == 0` + `attack_cost < 10% of treasury` → **Critical** (direct economic exploit)
- `timelock_delay < 24h` + `attack_cost < 25% of treasury` → **High** (insufficient response window)
- Any DAO where liquid token purchases can reach quorum within 24h is reportable

**Checklist extras:** Does any proposal template allow arbitrary `call()` to treasury? Can governance tokens be flash-loaned to reach quorum within one block? (Snapshot-style off-chain governance is immune to flash loans; on-chain Governor contracts with no `block.number` snapshot requirement are not.)

**Bug class:** `access-control` | **Ceiling:** Critical

### Hook 2: DEX Aggregator Pool-Depth Slippage Checks
*Tied to $2M ETH MEV backrun (§1.2 and §2.2)*

Any DeFi protocol that programmatically executes swaps — vault rebalancers, liquidation bots, yield harvesters, lending protocol collateral swaps — may be vulnerable to same-block backrun extraction if swap routing lacks pool-depth validation.

**What to look for in Solidity:**

```solidity
// RED FLAG: Swap with no pool-depth pre-check and loose slippage param
function _rebalance(address tokenIn, address tokenOut, uint256 amountIn) internal {
    ISwapRouter.ExactInputSingleParams memory params = ISwapRouter.ExactInputSingleParams({
        tokenIn: tokenIn,
        tokenOut: tokenOut,
        fee: poolFee,
        recipient: address(this),
        amountIn: amountIn,
        amountOutMinimum: 0,     // ← CRITICAL: zero slippage protection
        sqrtPriceLimitX96: 0
    });
    swapRouter.exactInputSingle(params);
}

// RED FLAG: Multi-hop route through a pool with unknown liquidity
// especially if the intermediate pool was added to the route list
// without TVL minimums validated on-chain
```

**Checklist:**
1. Find all `exactInputSingle` / `exactInput` / `swapExactTokensForTokens` calls in your target
2. Check `amountOutMinimum` — if zero or set only by a caller-controlled parameter with no floor, it's exploitable
3. Check if the routing path can include pools the attacker could have previously seeded with minimal liquidity
4. In multi-hop paths: any intermediate pool with low TVL becomes the weakest link

**Bug class:** `front-running`, `logic-error` | **Ceiling:** High to Critical depending on swap volume
