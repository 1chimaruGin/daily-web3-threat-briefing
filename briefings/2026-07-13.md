### 2.2 BonkDAO Attacker Creates On-Chain "BONK 2.0" Governance Laundering Structure

**Source:** [BonkDAO Attacker Moves $19M Loot Into New 'BONK 2.0' DAO — The Defiant](https://thedefiant.io/news/hacks/bonkdao-attacker-moves-19m-loot-into-new-bonk-2-0-dao) | [Crypto News](https://cryptonews.net/news/security/33114375/)
**Bug class:** `access-control`, `logic-error`

This is a novel post-exploit behavior pattern: after draining $20M from BonkDAO via Proposal #76, the attacker moved $19M into a **new 3-of-3 multisig DAO** — governed by the voter wallet, the exploiter wallet, and a financially-linked third wallet. They branded this structure "BONK 2.0."

**Why this matters for hunters:**

1. **Governance laundering is real** — the attacker is attempting to legitimize stolen funds through a new DAO, not just send them to exchanges. This pattern makes cross-chain tracing harder and may exploit jurisdictional gray areas.

2. **The original attack pattern** (BIP #76 attack): $4.4M to buy >1% of BONK supply → submit a "boring" 10-clause proposal with a malicious treasury-drain clause buried inside → passed with **2.9% voter turnout** and **99.9% yes votes** (only the attacker voted). No code was exploited — pure governance parameter abuse.

3. **Hunt target:** Any DAO where (a) quorum threshold is ≤ 5%, (b) a single entity can reach quorum by buying tokens on open market, (c) proposals allow arbitrary `call()` or `delegatecall()` to external contracts, and (d) timelock is ≤ 24h. Proposal text alone prevents nothing.

**Check these DAO governance parameters for any target:**
- `quorumNumerator()` / `quorumDenominator()` — if ratio < 5%, one buyer can control votes
- `votingDelay` / `votingPeriod` — low delay means proposals pass before community notices
- `proposalThreshold` — how much tokens needed to submit a proposal (not the same as quorum)
- Whether `execute()` accepts arbitrary calldata without an allowlist of target contracts

### 2.3 Supra Oracle BLS Verifier — Identity-Point Bypass (Technical Deep-Dive)

**Tied to:** Bonzo Lend $9M (first covered 2026-07-12)
**Source:** [Bonzo Lend loses $9M after oracle flaw inflates SAUCE price — Crypto.news](https://crypto.news/bonzo-lend-loses-9m-after-oracle-flaw-inflates-sauce-price/) | [Cross-chain oracle risk analysis — CryptoRank](https://cryptorank.io/news/feed/d9859-bonzo-exploit-drains-9m-from-hederas-largest-lending-protocol-underscoring-cross-chain-oracle-risk)
**Bug class:** `oracle-manipulation`

Now that post-exploit analysis is circulating, the **root technical cause** is worth formalizing:

**Flaw:** Supra's oracle verifier contract passed zero-value BLS public-key inputs to Hedera's pairing system precompile. The pairing precompile returned `true` because both inputs represented the **mathematical identity point** (i.e., the neutral element in the elliptic curve group) — a well-known cryptographic edge case where any signature "verifies."

The verifier failed to reject zero-value inputs *before* calling the pairing check:
```
if (verifier(pubkey, message, sig)) {
    // ← pubkey = 0x00 (identity point)
    // pairing(0, G2) == pairing(G1, 0) is true by definition
    acceptPriceUpdate(price);  // malicious price accepted
}
```

**Generalization:** This class of bug appears anywhere BLS or pairing-based signature aggregation is used without explicit infinity/zero-point rejection. Affected contexts include:
- Oracle verifiers using BLS committee signatures (Supra, Chainlink Threshold, others)
- BLS wallet implementations (EIP-2537 ecosystem)
- ZK proof verifiers that use BN254 or BLS12-381 pairings with untrusted public inputs
- Bridge DVN verifiers using aggregated committee signing

Supra deployed a fix; other oracle and bridge operators using BLS-adjacent schemes should audit their verifier for this input pre-check.

## 4. 🎯 Hunt Hooks

### Hook 1: BLS/Pairing Verifier — Reject the Identity Point
*Tied to §2.3 Supra oracle BLS bypass → Bonzo Lend $9M*

Any protocol that uses BLS aggregated signatures or pairing-based ZK proof verification should be checked for identity-point input rejection. This is a **one-line check** that was missing from Supra's verifier:

```solidity
// In any BLS verifier or pairing-based ZK verifier:
function verifySignature(
    BLS.G1Point memory pubkey,
    bytes memory message,
    BLS.G2Point memory sig
) internal view returns (bool) {
    // ← MISSING CHECK: is pubkey the identity point?
    require(!pubkey.isZero(), "Identity point rejected"); // this line must exist
    require(!sig.isZero(), "Zero signature rejected");
    
    // Only THEN call the pairing precompile
    return BLS.pairing(pubkey, BLS.negG2(hashToG2(message)), BLS.G1(), sig);
}
```

**Where to look in any target:**
1. Find the oracle price submission or signature verification contract
2. Search for `pairing(`, `ecPairing(`, `BLS12381.pairing(`, or Hedera system contract calls `0x169` (BN256 pairing)
3. Check if there's a pre-flight `require(!pubkey.isZero())` or equivalent before the pairing call
4. If the check is absent — this is likely a Critical finding on any lending or oracle protocol

**Bug class:** `oracle-manipulation` | **Ceiling:** Critical | **Bounty potential:** $500K–$2M+

### Hook 2: DAO Governance — One-Buy-Quorum Attack Surface
*Tied to §2.2 BonkDAO $20M pattern*

The BonkDAO attack required no code exploit — only token acquisition and a creative proposal. For any DAO target on your list, run this governance audit checklist:

```bash
# OpenZeppelin Governor-compatible DAOs (most EVM DAOs):

# 1. What's the quorum?
cast call <GOVERNOR> "quorumNumerator()(uint256)" --rpc-url $RPC
# If < 500 basis points (5%), one buyer may reach quorum

# 2. What's the voting delay / period?
cast call <GOVERNOR> "votingDelay()(uint256)" --rpc-url $RPC    # in blocks
cast call <GOVERNOR> "votingPeriod()(uint256)" --rpc-url $RPC
# Low delay + short period = proposals pass before community notices

# 3. Can execute() call arbitrary contracts?
# Read the proposal execution code — does it restrict target addresses?
# If execute() accepts any (target, calldata) pair, arbitrary treasury drains are possible

# 4. Is there a timelocked guardian/canceller role?
cast call <GOVERNOR> "timelock()(address)" --rpc-url $RPC
# No timelock or short timelock (< 48h) = no recovery window
```

**Report condition:** Quorum reachable with market-purchasable tokens + no proposal allowlist + no guardian canceller = governance takeover = Critical finding.

**Bug class:** `access-control`, `logic-error` | **Ceiling:** Critical

### Hook 3: Supply Chain — Check Your Target's npm Org and CI Pipelines
*Tied to §2.1 SlowMist H1 2026 report (supply chain = $298M loss leader)*

The SlowMist report confirms supply-chain attacks as the largest single loss category in H1 2026. Before hunting smart contracts, quickly audit your target's off-chain attack surface — it often has lower competition and equal bounty potential:

```bash
# 1. Find the project's npm org packages
npm search --json @<project-org> | jq '.[].name'

# 2. Check publish date vs. expected — a recently published patch to an old version is suspicious
npm info @<project>/<package> time --json | jq 'to_entries | sort_by(.value) | last'

# 3. Check the CI/CD config for PRIVATE_KEY / MNEMONIC env vars used in deploy scripts
grep -r "PRIVATE_KEY\|MNEMONIC\|SECRET" .github/workflows/
# Any workflow that handles private keys and accepts external pull requests is vulnerable

# 4. Check if the project pins dependency versions or uses floating semver
grep -r '"@openzeppelin\|"ethers\|"web3' package.json
# Floating versions (^x.y.z) mean a compromised upstream silently propagates
```

**Bug class:** `supply-chain`, `key-management` | **Ceiling:** Critical | **Scope:** Many Immunefi programs now include infrastructure scope — always read the fine print.
