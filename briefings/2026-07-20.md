### CoinDCX $44M — Post-Mortem Details (Updated July 20)

**Bug class:** `key-management`, `supply-chain`
**Chain:** Solana → Ethereum (cross-chain laundering)

The fuller post-mortem confirms the attack had a **6-day reconnaissance window** before the main drain:

| Timestamp | Event |
|---|---|
| ~July 13 | Attacker performs a 1 USDT "dry run" test transaction to validate wallet access |
| July 19, 02:37 UTC | Full drain begins: $44.2M USDC/USDT siphoned from operational liquidity wallet in a single sweep |
| July 19, post-drain | Funds split across 6 wallets, bridged from Solana to Ethereum, then routed via Tornado Cash |

**Root cause:** A CoinDCX software engineer was socially engineered via a fake recruiter offering freelance "paid task" work. Over months, the engineer unknowingly completed tasks that installed malware on their company laptop, giving attackers persistent access. The compromised device had access to the operational wallet used for liquidity provisioning on partner exchanges. **Customer funds were not touched.**

**Key operational failure:** A single employee's workstation held access to a $44M operational wallet with no hardware-signing requirement and no anomalous-transfer circuit-breaker.

**Chains:** All EVM chains — any browser-extension wallet
**Conference:** 26th Privacy Enhancing Technologies Symposium, July 20–25, Calgary, Canada
**Paper:** "The Masks We (Think We) Wear: Privacy Threats of Browser-Extension Wallets in the Web3 Ecosystem" — [arXiv 2607.06141](https://arxiv.org/html/2607.06141v1)

The KU Leuven DistriNet paper (covered in the July 19 briefing) is presented **today at PETS 2026**, meaning:

1. **PoC code is now being shared among conference attendees** — expect public release in days to weeks
2. **Vendor disclosure windows are expiring** — MetaMask, Rabby, OKX, and Bybit declined to fix; Coinbase and Coin98 patched in February

The five-category attack surface studied across 85 extensions (35M Chrome installs):

| Category | Finding | Exploitable outcome |
|---|---|---|
| Address-linking via shared RPC | 17 wallets send ≥2 user addresses in one request to third-party RPC | RPC provider de-anonymizes wallet graph |
| Cross-site address leakage | Wallet pings external server with address on every DApp visit | Full browsing-correlation profile |
| **Permission persistence** | 22 of 36 EVM wallets: site permission survives cookie clear + browser restart | Silently retained address-read access; re-exploitable after "logout" |
| Real-name linkage | KYC DApp address leaked ties to real identity | On-chain de-anonymization |
| No revoke-all equivalent | No wallet-side "clear all site data" | Accumulated grant explosion |

**Why it matters right now for hunters:** Wallet extension bug-bounty scopes with active programs (MetaMask, Trust Wallet, Rabby) are valid targets for the permission-persistence and cross-site tracking findings. MetaMask's response ("known issue, no immediate plans to fix") means the finding is documented but unfixed — a fresh reproduction with a new PoC is a valid new submission.

### 2.2 Across Protocol — Solana Event-Spoofing Root Cause (Asymmetric Research, April/July 2026)

**Bug class:** `bridge-dvn`, `logic-error`
**Chain:** Solana
**Technical reference:** [Asymmetric Research — Across Solana Event Spoofing](https://blog.asymmetric.re/across-solana-event-spoofing/)

Asymmetric Research disclosed in April 2026 the vulnerability class that underlies the July 17 Across Protocol Solana attack. The post-mortem blog now serves as the canonical technical reference; a full Risk Labs post-mortem is expected this week. Key insight:

**EVM vs. Solana event semantics differ dangerously for bridge relayers:**

| Property | EVM | Solana |
|---|---|---|
| Events are first-class | Yes — `emit` opcode, stored in receipts | No — reconstructed from instruction data or program logs |
| Failed txns emit events | No — revert rolls back event | **Yes — program logs are written even for failed transactions** |
| Canonical indexing | `eth_getLogs` with bloom filter | `getTransaction` / log parsing (non-standard) |

The Across Solana spoke pool (built on Anchor framework) reconstructed deposit events from transaction logs. An attacker could **craft a transaction that fails on-chain but emits a deposit-event-shaped log**, causing naive relayers to see a "deposit" with no real funds behind it and fill the order from their own capital.

**Mitigations per Asymmetric Research:**
- Validate that the transaction actually *succeeded* before reading events from it
- Verify emitting program ID matches expected spoke pool address
- Cross-check event data against actual account state changes

**Pattern generalizes to all Solana bridges** — any bridge that sources events from failed/reverted transactions is vulnerable to this class. Across deployed a fix in April; the July 17 incident may represent a related-but-distinct variant.

**Sources:** [Asymmetric Research blog](https://blog.asymmetric.re/across-solana-event-spoofing/) | [CryptoTimes — Across July 17](https://www.cryptotimes.io/2026/07/17/across-protocol-reports-first-attack-on-solana-after-34b-in-bridge-volume/) | [BloomingBit](https://en.bloomingbit.io/feed/news/116465)

### 2.3 Ostium Oracle Key Compromise — Pattern Update (July 15)

**Bug class:** `key-management`, `oracle-manipulation`
**Chain:** Arbitrum (real-world asset perp DEX)

The Ostium incident (July 15, $23.75M USDC drained) represents the clearest executed example of the oracle-signer-key compromise attack class in 2026, now that SlowMist, PeckShield, and Blockaid have all published their analyses. Core technique:

1. Attacker compromises the private key of an **off-chain oracle signer EOA** — not a smart contract vulnerability
2. Uses the stolen key to sign a fabricated BTC price of **~$5,000** (real market: ~$115,000–$120,000)
3. Uses a legitimate, already-registered `PriceUpKeep Forwarder` to push the fake price on-chain
4. Executes ~20 looped long positions at the stale price, extracting $23.75M from the OLP vault
5. Routes ~10,540 ETH through Tornado Cash

**The vulnerability is not in the smart contract — it is in who holds oracle signing authority.** A single EOA with no multisig requirement controlling price-feed submissions is a critical single point of failure. Ostium's OLP vault had $63M in liquidity; the attacker drained 38% before detection.

**Sources:** [Coingabbar](https://www.coingabbar.com/en/crypto-currency-news/ostium-hack-2026-oracle-exploit-defi-24m-usdc-arbitrum) | [crypto.news — Blockaid](https://crypto.news/blockaid-uncovers-18m-exploit-that-forces-ostium-halt/) | [CryptoBriefing](https://cryptobriefing.com/ostium-suspends-trading-olp-vault-exploit/)

## 4. 🎯 Hunt Hooks

### Hook 1: Oracle Signer Authority — Is It an EOA or a Multisig?
*Tied to §2.3 Ostium oracle key compromise — $23.75M drained*

The Ostium attack succeeded because a single EOA held oracle signing authority with no additional controls. For any perpetual DEX, lending protocol, or options vault target with a bug bounty:

```
// On-chain check: who is the oracle signer / price updater?
address public priceKeeper;        // check if this is an EOA (no code)
address public oracleUpdater;
address public priceSetter;

// Red flags in access-control checks:
// 1. signer is an EOA  → assembly { size := extcodesize(signer) } returns 0
// 2. No multisig requirement for price submissions
// 3. No circuit breaker: |new_price - last_price| / last_price > threshold → revert

// What to check in the oracle contract:
function updatePrice(uint256 price) external onlyKeeper {
    // GOOD: price deviation guard
    require(
        price > lastPrice * 95 / 100 && price < lastPrice * 105 / 100,
        "price deviation too large"
    );
    // BAD: no deviation check → any signed price accepted
}
```

**Report condition:** Oracle signing key held by EOA with no multisig, no deviation circuit-breaker → Critical.

### Hook 2: Solana Bridge Relayer — Does It Accept Events from Failed Transactions?
*Tied to §2.2 Across Protocol Solana event spoofing*

The generalizable pattern: Solana relayers that reconstruct deposit events from transaction logs without verifying the transaction succeeded are vulnerable to event spoofing.

```typescript
// VULNERABLE pattern in Solana bridge relayer:
const tx = await connection.getTransaction(txSig, { commitment: "confirmed" });
const depositLog = tx.meta.logMessages.find(l => l.includes("DepositEvent"));
// ⚠️ Does NOT check: tx.meta.err === null
// A failed transaction can still emit logs → fake deposit event

// SAFE pattern:
const tx = await connection.getTransaction(txSig, { commitment: "confirmed" });
if (tx.meta.err !== null) {
  throw new Error("Transaction failed — discarding any emitted events");
}
const depositLog = tx.meta.logMessages.find(l => l.includes("DepositEvent"));

// Additional check: program ID matches expected spoke pool
const spokeProgramId = new PublicKey("expectedSpokePoolProgramId");
const isFromSpoke = tx.transaction.message.accountKeys
  .some(key => key.equals(spokeProgramId));
if (!isFromSpoke) throw new Error("Event not from expected spoke pool");
```

**Report condition:** Solana cross-chain bridge relayer processes events from failed transactions or without program-ID verification → Critical.

### Hook 3: Operational Wallet Access Controls — Device Isolation and Transfer Limits
*Tied to §1 CoinDCX $44M — 6-day dwell, single device holding $44M wallet access*

The CoinDCX attack pattern (malware → long dwell → drain large operational wallet) applies to any CEX, protocol treasury, or bridge operator where:
- Operational/liquidity wallets are accessible from internet-connected workstations
- No hardware signing requirement for large transfers
- No anomaly detection on transfer velocity or destination novelty

```
// What to look for in CEX / bridge operator bug bounty targets:
// 1. Operational wallet signing flow:
//    - Is private key on HSM/hardware wallet, or software keystore on a server?
//    - Is there a time-lock or multisig for transfers > $X threshold?

// 2. Transfer velocity circuit-breaker:
//    - Does the system reject transfers that exceed N% of wallet balance in 1 hour?
//    - Is there a "first-time destination" hold period?

// 3. Anomaly detection:
//    - Is there out-of-band alerting for new destination addresses?
//    - CoinDCX's dry run (1 USDT, 6 days prior) went undetected

// Bounty framing: "Operational wallet key management" or
// "Missing transfer rate-limiting on hot wallet" are valid scope items
// for CEX programs with infrastructure/key-management scope
```

**Report condition:** Operational wallet accessible from internet-connected workstations with no hardware signing requirement and no velocity/anomaly controls → High/Critical depending on TVL at risk.
