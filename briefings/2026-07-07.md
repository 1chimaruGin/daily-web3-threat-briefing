## 1. 🔴 Active Exploits (last 24h)

### 1.1 Summer.fi (Lazy Summer Protocol) — $6M Vault Accounting Drain

**Date:** July 6, 2026
**Chain:** Ethereum
**Protocol:** Summer.fi Lazy Summer Protocol — `LazyVault_LowerRisk_USDC` / `FleetCommander` contract
**Amount lost:** ~$6.017M DAI
**Bug class:** `flash-loan`, `logic-error`
**Detected by:** Blockaid, PeckShield, CertiK

**Attack flow (reconstructed from multiple sources):**

1. Attacker pre-accumulated shares in the Silo: Varlamore USDC Growth vault over multiple blocks ahead of the attack.
2. In a single transaction: borrowed **~$65.4M** via flash loan.
3. Donated the pre-accumulated Silo vault shares **directly to the Ark contract** (the connector between FleetCommander and underlying lending protocols), inflating `totalAssets()` without going through the normal deposit flow.
4. Deposited **~$64.8M** through the normal path — `FleetCommander` calculated share price against the now-inflated `totalAssets()`, issuing under-valued shares.
5. Immediately redeemed those shares for **~$70.9M**, pocketing ~$6M net after repaying the flash loan.
6. Converted all stolen DAI through Uniswap V3 pools. Funds moved quickly.

The affected vault's APY display briefly spiked to **~2,080,000%** — a detectable on-chain anomaly that Blockaid's monitoring caught in real time.

**Protocol response:** Summer.fi paused all Lazy Summer Protocol vaults via the guardian mechanism. As of reporting, Summer.fi attributed the root cause to accounting within the Silo integration ("outside the control of the Lazy Summer Protocol"), but has not published a formal post-mortem. Investigation ongoing.

**Root cause assessment:** This is a variant of the vault donation / `totalAssets()` manipulation class — generically identical to the ResupplyFi ERC-4626 pattern covered in the July 3 and July 6 briefings, but applied to a multi-vault `FleetCommander` architecture rather than a single ERC-4626 vault. The key difference: the attack surface is the **Ark connector**, not the vault itself — any vault that allows external contracts to update the asset count visible to the share-price calculation is vulnerable to this family of attacks.

**Notable detail:** This is the second DeFi exploit recorded by DefiLlama for July 2026, and the second consecutive incident in this summer's "ERC-4626 accounting manipulation" cluster (ResupplyFi Jul 3 → Summer.fi Jul 6).

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Ill Bloom — PRNG Weakness in Software Wallets (Coinspect Disclosure)

**Published:** July 6, 2026 — Coinspect Security Research
**Chains affected:** Bitcoin, Ethereum, Polygon, Rootstock, Tron, Solana
**Total stolen (cumulative):** ~$5M (coordinated drains since May 2026)
**Bug class:** `key-management`
**CVE:** Not yet assigned

**What it is:** Security firm Coinspect disclosed a vulnerability class they call "Ill Bloom" — insecure pseudorandom number generation (PRNG) used during seed phrase generation in certain software wallets. The flaw reduces the effective entropy of BIP-39 mnemonic generation far below the expected 128–256 bits, making recovery phrases **computationally predictable** to an attacker who can identify a vulnerable wallet's generation parameters.

**Technical root cause:** Affected wallets used platform-specific or non-cryptographically-secure PRNG sources at the time of wallet creation rather than the OS-level CSPRNG (`/dev/urandom`, `SecureRandom`, `crypto.getRandomValues`). The resulting seed space is small enough that a targeted scan can enumerate and drain vulnerable addresses.

**Scope and impact:**
- Wallets created from approximately **2018 to early 2022** are most at risk (the vulnerable libraries were widely included in mobile wallet SDKs of that era)
- Hardware wallets are **not affected** — the flaw is in software wallet key generation only
- Most current major software wallets (MetaMask, Trust Wallet current versions) are not vulnerable
- **$3.14M was drained in a single coordinated sweep of 431 wallets on May 27, 2026** — primarily BTC and ETH
- Additional drains in early July 2026 bring the cumulative total to ~$5M
- Coinspect warns the vulnerable wallet population may be significantly larger than confirmed drains suggest

**What to do:**
- Run your wallet addresses through the public checker at [illbloom.org](https://illbloom.org/) (read-only, takes a public address only)
- If flagged: sweep funds to a freshly generated wallet on a current, patched wallet implementation **immediately** — do not simply generate a new seed on the same old app
- Bug bounty hunters: if any of your target protocols hold treasury in software wallets rather than multisigs, these may now be at elevated risk

### 2.2 Summer.fi Attack Class — "Ark Connector Donation" Generalizes ERC-4626 Inflation Beyond Single-Vault Scope

*See §1.1 for full incident details*

The technical significance here is architectural: the ResupplyFi post-mortem (covered Jul 6) described donation attacks against a **single** ERC-4626 vault. Summer.fi demonstrates the same economic distortion is possible against **multi-vault aggregator architectures** where:

1. An underlying lending protocol (the "Ark") can receive assets from an external path not controlled by the aggregator's accounting
2. The aggregator's `totalAssets()` reads the Ark's balance, which can be inflated
3. A flash loan gives the attacker enough capital to make the distortion profitable

**Concrete hunt hook:** Any yield aggregator, vault router, or "fleet commander" contract that fetches `totalAssets()` from underlying protocol connectors rather than tracking deposits internally. The attack surface is the **connector layer**, not just the vault layer.

## 3. 🟢 Advisories / CVEs

### 3.1 NEW — CVE-2026-46242 "Bad Epoll" — Linux Kernel Local Privilege Escalation to Root

**Disclosed:** July 3–6, 2026
**Researcher:** Jaeyoung Chung
**Affected:** Linux kernel **5.10 through 6.11** (patch: upstream commit `a6dc643c6931`, merged April 24, 2026)
**Severity:** HIGH (local privilege escalation → root)
**PoC:** Public (99% reliable exploit in the wild)
**Patch status:** Upstream since April 24, but most distributions have **not yet shipped backports**

**What it is:** A use-after-free race condition in the Linux kernel's `epoll` subsystem. Two kernel paths can attempt to free the same internal event-poll object simultaneously, allowing a controlled heap overwrite. An ordinary unprivileged process can exploit this to gain root access to the machine — on Linux desktops, servers, and Android devices.

**Why it matters for web3:**
- Ethereum execution clients (Geth, Nethermind, Besu) and consensus clients (Lighthouse, Prysm, Teku) run on Linux servers — an attacker with any foothold can escalate to root and steal validator keystores or signing keys
- CI/CD runners for smart contract testing (GitHub Actions self-hosted, GitLab Runner) are prime targets — combine with IronWorm's CI foothold (Jul 6 briefing) for a full kill chain
- Hardware crypto signing devices that run embedded Linux (certain HSMs, some signing server appliances) are in scope
- The fix was silently merged 70 days before disclosure — check your kernel version against `uname -r` now

**Check and patch:**
```bash
uname -r  # Check your kernel version
# Patch is in: 5.10.x, 6.1.x, 6.6.x LTS trees from April 24 onward
# bpftool prog list  # Also look for unexpected eBPF programs (IronWorm IoC)
```

Debian, Ubuntu, RHEL, and SUSE are pulling the backport; reboot after upgrade (or use live-patching).

## 4. 🎯 Hunt Hooks

### Hook 1: Find Vault Aggregators With External `totalAssets()` Feeds (Summer.fi Pattern)
*Tied to Summer.fi FleetCommander exploit (§1.1)*

The July 2026 "vault accounting manipulation" cluster now has two public exploits in four days. More protocols with similar architecture exist. Hunt them **today** before copycats do.

**What to look for:**

```solidity
// RED FLAG: aggregator pulls totalAssets from external connector/Ark
function totalAssets() public view override returns (uint256) {
    uint256 total = 0;
    for (uint i = 0; i < arks.length; i++) {
        total += IArk(arks[i]).totalAssets(); // ← can this be inflated externally?
    }
    return total;
}

// RED FLAG: Ark/connector reads raw balanceOf rather than tracking deposits
function totalAssets() external view returns (uint256) {
    return IERC20(asset).balanceOf(address(this)); // ← donation inflates this
}
```

**Checklist:**
1. Find all vault aggregators/fleet commanders in your targets — search for `FleetCommander`, `Allocator`, `Strategy`, `Vault` patterns that iterate over sub-vaults
2. For each: does `totalAssets()` read `balanceOf()` from underlying connectors? If yes — **donation inflatable**
3. Can a flash loan make the inflation profitable? (Does TVL × share-price delta > flash loan fee?)
4. Is there a guardian/pause mechanism? (Matters for severity / exploitability window)
5. Focus on **multi-protocol yield aggregators** launched in the last 6 months — they inherit this architecture and may not have received a dedicated security review

**Bug class:** `flash-loan`, `logic-error` | **Ceiling:** Critical

### Hook 2: Check Your Protocol's Treasury / Multisig for Ill Bloom Exposure
*Tied to Ill Bloom PRNG disclosure (§2.1)*

Bug bounty hunters often have visibility into protocol treasury addresses. Any treasury address generated by a software wallet between 2018 and 2022 is potentially in the Ill Bloom vulnerable set.

**Quick due-diligence:**
1. Identify treasury EOA addresses for your bug bounty targets
2. Run them through [illbloom.org](https://illbloom.org/) (read-only public address check)
3. If flagged: this is a **critical severity** finding for the bug bounty program — the treasury could be drained by an attacker who has run the same scan
4. Even if the address isn't flagged, document the wallet software used — Coinspect notes the vulnerable wallet population is likely larger than current detections

**Note:** Some programs may not have a defined scope for "key management advisory" — treat it as a critical external asset finding and file it, noting that the vulnerability is in the key generation, not the smart contract.

**Bug class:** `key-management` | **Ceiling:** Critical (direct treasury drain risk)

### Hook 3: Linux Kernel LPE as Pivoting Vector on Self-Hosted CI (Bad Epoll)
*Tied to CVE-2026-46242 (§3.1)*

Many smart contract projects run self-hosted GitHub Actions or GitLab runners on Linux VMs with unpatched kernels. A publicly available 99%-reliable local root exploit now exists for those runners.

**Threat scenario for a bug hunter:**
- A self-hosted runner processes a PR that installs an IronWorm-contaminated npm package → IronWorm gains user-level foothold on runner → Bad Epoll escalates to root → full access to signing keys, `.env` files, AWS credentials present on the runner

**What to check if you run your own hunting infrastructure:**
```bash
uname -r     # Is your kernel in 5.10–6.11 range?
apt list --installed | grep linux-image    # Ubuntu/Debian
rpm -q kernel    # RHEL/CentOS
# Apply kernel update, then reboot (or use kpatch/ksplice for live patch)
```

Report to target programs if their self-hosted runner infrastructure is publicly auditable (some are via GitHub's API — `repos/{owner}/{repo}/actions/runners`).

**Bug class:** `key-management` (keys accessible after privesc) | **Infra context:** HIGH
