## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 ResupplyFi Full Post-Mortem — ERC-4626 "First-Depositor" Donation Attack (Now Public)

**Published:** Multiple sources this week — Ackee Blockchain, Guardrail AI, QuantumLoom, ainvest
**Chain:** Ethereum
**Protocol:** ResupplyFi cvcrvUSD vault (launched on Convex/Curve ecosystem, live ~2 hours before exploit)
**Amount lost:** $9.56M
**Bug class:** `flash-loan`, `rounding`

**Root cause (technical):** The ERC-4626 vault's share-price oracle relied on a `totalAssets() / totalSupply()` integer division. With zero or near-zero supply, integer division rounds down to **zero** — which then allows a "vault inflation" manipulation:

1. Attacker borrows $4,000 USDC via Morpho flash loan → converts to 4,000 crvUSD
2. Donates 2,000 crvUSD **directly to the vault contract** (not through `deposit()`), inflating `totalAssets` without minting shares
3. Deposits 2 crvUSD → mints **1 wei** of cvcrvUSD shares. That 1 wei now represents the entire inflated vault balance
4. Uses the 1 wei position to borrow 10,000,000 reUSD (the protocol's full borrow limit) — the inflated AUM makes the collateral appear sufficient
5. Repays flash loan, keeps ~$9.56M profit

**Why no minimum initial liquidity:** The vault had no `initialDeposit` guard and no minimum share supply. The very first depositor sets the exchange rate. The attack was profitable 2 hours after deployment.

**What makes this generalizable:**
- ERC-4626 donation attacks are not new (first documented ~2022) but continue to appear in newly deployed vaults
- Any vault that: (a) allows direct token transfers to increase `totalAssets`, (b) uses spot division rather than virtual shares or minimum supply, and (c) launched with low initial liquidity is vulnerable
- The mitigation is well-established: use **virtual shares** (OpenZeppelin v5.x ERC4626 adds `_decimalsOffset()` to inflate precision) or enforce a dead-share minimum deposit at initialization

### 2.2 IronWorm Comprehensive Autopsy — Rust npm Worm With eBPF Rootkit, Tor C2, Self-Propagation

**Published:** JFrog Security Research, Dark Reading, SC Media, OX Security — June–July 2026
**Malware family:** IronWorm (Shai-Hulud variant / "rustier cousin")
**Scale:** 37 malicious npm packages; 57 backdated commits across 9 organizations; 32,000+ monthly downloads on affected packages

> **Note on prior coverage:** The Miasma/Shai-Hulud campaign was covered in the 07-05 briefing. IronWorm is a **distinct but related** family — it's Rust-built (vs. Shai-Hulud's JS/Python origins), ships a kernel-level eBPF rootkit, and has a self-propagating CI/CD mechanism not present in Miasma. Worth a dedicated entry.

**Delivery:** Published through the compromised `asteroiddao` npm account. Packages install a 976 KB Rust ELF binary via `preinstall` hook. The binary is packed with a custom-modified UPX stub (defeats signature-based unpackers) and encrypts every internal string with a unique per-call-site key.

**Credential scope (86 env vars, 20+ file paths swept):**
- AWS, GCP, Azure, Vault, Kubernetes, Docker, GitHub tokens
- npm publishing tokens
- All major AI provider API keys (Anthropic, OpenAI, Gemini, Cohere, Mistral, Groq, Perplexity, xAI)
- **Crypto-specific:** Hardhat/Foundry `.env` files, SSH keys, browser-stored web3 session credentials
- **Exodus wallet hook:** Weakens Electron sandbox to capture BIP-39 seed mnemonic at wallet unlock

**Stealth:** eBPF kernel rootkit hides IronWorm processes, files, and network connections **from standard `ps`, `lsof`, and `netstat`**. Standard container security scans will not surface it if the rootkit loads first.

**C2:** Runs over Tor — downloads the Tor expert bundle, writes its own `torrc`, beacons to `/api/agent` over a hidden service. Exfiltrated data never traverses clearnet.

**Self-propagation via CI:** When executed in a GitHub Actions context, IronWorm requests an OIDC token and exchanges it for a short-lived npm publishing token → pushes trojanized versions of legitimate packages the repo depends on. No credential storage required. This is how 57 commits across 9 organizations were made.

**Anomalous operator behavior:** The attacker hardcoded their own BIP-39 mnemonic into the wallet-stealer's **skip list** (so their own wallet isn't stolen). Researchers assess this as a rehearsal campaign — the full deployment tooling may be in preparation.

**IoCs and detection:**
```
npm account:  asteroiddao (compromised; all packages should be removed/replaced)
Binary size:  ~976 KB Rust ELF (preinstall hook)
Strings:      "shai-hulud", "arrakis", "spice", "iron-worm" in module metadata
CI:           Unexpected OIDC → npm token exchange in Actions logs
Filesystem:   Unexpected .tor/ or torrc files in project/module directories
eBPF:         Unexpected BPF programs loaded — check: `bpftool prog list`
```

## 4. 🎯 Hunt Hooks

### Hook 1: Find ERC-4626 Vaults With No Virtual-Share Protection
*Tied to ResupplyFi ERC-4626 post-mortem (§2.1)*

The ResupplyFi post-mortem is now widely available and readable. Expect copycats to start targeting similar vaults. Go check your bug bounty targets **today** for this pattern.

**What to look for:**

```solidity
// VULNERABLE pattern — spot totalAssets/totalSupply with no virtual shares
function convertToShares(uint256 assets) public view returns (uint256) {
    uint256 supply = totalSupply();
    return supply == 0 ? assets : assets * supply / totalAssets();
}
```

**Checklist:**
1. Find all ERC-4626 vaults in your targets. Search for `convertToShares`, `previewDeposit`, `_convertToShares`.
2. Check: does `_decimalsOffset()` return 0? (OZ v4.x default). If yes and no other virtual-share mechanism — **potentially vulnerable**.
3. Check: is there a minimum initial deposit requirement or dead-share initialization at deployment? If not — **potentially vulnerable** to first-depositor inflation.
4. Check: can tokens be transferred directly to the vault address (bypassing `deposit()`)? `token.balanceOf(address(this))` used in `totalAssets()` is the tell.
5. Pay special attention to **vaults launched in the last 72 hours with TVL < $100K** — these are in the highest-risk window (attack usually happens within hours of deployment).

**Bug class:** `flash-loan`, `rounding` | **Ceiling:** Critical (protocol-level drain)

### Hook 2: Audit Your npm CI Pipeline for IronWorm Exposure
*Tied to IronWorm supply-chain campaign (§2.2)*

Before starting any hunting session today — especially if you run CI/CD pipelines for smart contract testing (Hardhat, Foundry via npm wrappers, Slither, etc.) — do a 3-minute exposure check:

```bash
# 1. Check npm audit for known-bad packages (won't catch all IronWorm variants)
npm audit --audit-level=moderate

# 2. Look for IronWorm IoCs in preinstall hooks of installed packages
cat node_modules/.package-lock.json | grep -i "preinstall\|postinstall" | head -30

# 3. Check for unexpected eBPF programs (Linux only)
bpftool prog list 2>/dev/null | grep -v "^$"

# 4. Look for IronWorm metadata strings in node_modules
grep -r "shai.hulud\|arrakis\|iron.worm\|miasma" node_modules/ --include="*.json" 2>/dev/null | head -20

# 5. Check GitHub Actions OIDC logs for unexpected npm publishes
# In Actions: look for "npm publish" calls not in your workflow definition
```

If your CI pipeline resolved any `asteroiddao`-namespaced package, or any package whose `preinstall` executes a binary >500KB, treat the CI runner environment as compromised and rotate all secrets that were available in that environment.

**Bug class:** `supply-chain`, `key-management`

### Hook 3: Verify Move-Chain Bridge Struct-Type Validation (Carry-Forward from 07-05)
*Tied to Hexens/Aptos MoveVM type-confusion (§2.1 of 2026-07-05 briefing)*

The Hexens/Aptos MoveVM type-confusion disclosure from July 4 showed that Move's resource-type guarantees can be broken at the VM level. The Aptos patch is live, but **bridges that accept Move-chain state proofs** as a source of truth are still worth checking.

**What to find:**
- Bridges where Aptos or Sui is a source chain → find the proof-verification logic
- Check if the bridge validates Move struct type tags independently, or delegates type safety to the Move VM
- Highest-risk: bridge accepts a generic storage proof without checking that the resource at a given address is the expected struct type (e.g., a `Coin<USDC>` proof where the bridge doesn't verify the Coin inner type matches)

The Aptos patch PR has been public since February 27 — read it to understand the exact fix and search for its inverse in less-audited Move VM forks (Sui's VM is independent, but Movement Labs and Linera carry shared architecture).

**Bug class:** `bridge-dvn`, `logic-error` | **Ceiling:** Critical
