### 1.1 Allbridge Core ‚Äî $1.65M | Flash Loan Pool-Ratio Manipulation | Solana

- **Date:** July 20, 2026 (UTC)
- **Chain:** Solana ‚Üí Ethereum (post-drain bridge)
- **Bug class:** `flash-loan`, `price-manipulation`
- **Amount lost:** ~$1.65M

Allbridge Core's stablecoin bridge on Solana was drained via a flash-loan pool-ratio skew. The attacker flash-borrowed **$1.12M USDC from Kamino** (Solana lending), dumped it into Allbridge's USDC/USDT pool, skewing the pool ratio, then immediately withdrew at the manipulated rate ‚Äî extracting the spread. Flash loan repaid; net profit ~$1.65M.

This is the **exact attack that Allbridge suffered on BNB Chain in 2023** (~$650K). After that incident, Allbridge patched the BNB pool with pool-ratio invariant checks. The Solana deployment **was never updated with the same fix.**

**Attack flow:**
1. Flash borrow $1.12M USDC from Kamino
2. Deposit into Allbridge USDC/USDT pool ‚Üí pool ratio heavily skewed toward USDT
3. Withdraw at the stale-rate formula (pre-skew exchange rate still cached) ‚Üí extract ~$1.65M
4. Repay flash loan
5. Bridge stolen funds Solana ‚Üí Ethereum via deBridge; attacker swapped into ETH as of July 21 UTC

**Response:** Allbridge paused all pools and protocol operations; advised all LPs to remove liquidity; contacted Kamino; requested opportunistic arbitrage profits returned. PeckShield and CertiK both confirmed funds had bridged to Ethereum by the time of their reports.

### 2.1 Summer.fi Announces Full Shutdown ‚Äî Vault Offboarding NAV Bug Officially Documented

**Date of announcement:** ~July 17‚Äì20, 2026 (The Defiant July 20); underlying exploit July 6, 2026
**Bug class:** `logic-error` (deactivated strategy still counted in vault NAV)
**Chain:** Ethereum (Lazy Summer vaults on mainnet, Base, Arbitrum)
**Amount lost in original exploit:** ~$6.04M
**Reference:** [blog.summer.fi ‚Äî Sunsetting Summer.fi and the Labs Company](https://blog.summer.fi/sunsetting-summer-fi-and-the-labs-company/)

Summer.fi announced it will fully shut down by **August 31, 2026**, ending seven years of operation. The root cause is now fully documented in their post-mortem and is instructive for all yield-aggregator hunters.

**Root cause:** Summer.fi's Lazy Summer Protocol routes vault deposits across yield strategies called "Arks." During offboarding, an Ark is "capped" (removed from new allocations), but the offboarding process **did not simultaneously remove the Ark from the NAV computation.** The attacker exploited the gap:

1. Identified a capped (offboarding) Ark backed by Silo vault tokens that had declined in real value
2. Donated the Silo tokens into the capped Ark at their (stale) book price, inflating the vault's apparent NAV
3. Minted vault shares at the inflated NAV, then immediately redeemed them for the inflated equivalent ‚Äî extracting $6.04M in a single atomic transaction
4. A $65.4M Morpho flash loan funded the donation; APY briefly displayed as **2.08 million %**

This attack required ~3 months of patient position accumulation before the one-block execution.

**State of user funds:** ~$4M in depositor assets remain temporarily illiquid. The Lazy Summer DAO is processing withdrawals through August 31.

### 2.2 Incomplete Security Fix Pattern ‚Äî Allbridge as Case Study

**Bug class:** `flash-loan`, `price-manipulation`

The Allbridge Core July 20 incident is the clearest 2026 example of a **partial patch leaving a full attack surface on another chain deployment.** The 2023 BNB fix applied pool-ratio invariant checks on that deployment only; the Solana program was never updated. The attack worked identically.

This pattern recurs across multi-chain DeFi: fixes are often applied to the chain where an attack was first detected, without systematic propagation to all equivalent deployments.

**Generalized risk:** Any multi-chain protocol that deploys equivalent smart contracts (or programs) to multiple chains where a security fix was applied to ‚â§ N-1 chains is vulnerable on the unfixed chains.

## 4. üéØ Hunt Hooks

### Hook 1: Flash Loan Pool-Ratio Invariants in Bridge Stablecoin Pools
*Tied to ¬ß1.1 Allbridge Core $1.65M ‚Äî flash-loan skew on Solana USDC/USDT pool*

Single-sided stablecoin bridge pools that do not enforce a post-swap pool-ratio deviation limit are susceptible to the Allbridge attack. Check target protocols for:

```solidity
// VULNERABLE: no pool-ratio guard ‚Äî flash loan can skew ratio arbitrarily
function swap(uint256 amountIn, address tokenIn) external {
    pool.deposit(tokenIn, amountIn);
    uint256 amountOut = calcOutput(amountIn);  // uses current (skewed) ratio
    pool.withdraw(tokenOut, amountOut);
    // ‚öÝÔ∏è No check: if ratio was pre-skewed by flash loan, amountOut is inflated
}

// SAFE: pool-ratio deviation guard
function swap(uint256 amountIn, address tokenIn) external {
    uint256 ratioBefore = pool.getPoolRatio();
    pool.deposit(tokenIn, amountIn);
    uint256 ratioAfter = pool.getPoolRatio();
    require(
        ratioAfter >= ratioBefore * MIN_RATIO_BPS / 10000 &&
        ratioAfter <= ratioBefore * MAX_RATIO_BPS / 10000,
        "pool ratio deviation exceeds limit"
    );
    ...
}
```

**Solana equivalent (Rust):** Look for pool programs that compute output amounts from a spot ratio without checking the ratio has not moved significantly in the same transaction.

**Report condition:** Bridge stablecoin pool with no atomic pool-ratio invariant check ‚Üí Critical (one-block flash-loan exploitable, no residual risk required).

### Hook 2: Multi-Chain Patch Coverage Audit
*Tied to ¬ß2.2 Allbridge incomplete-fix pattern*

When auditing any multi-chain protocol, verify that security patches applied after a prior exploit are present in **all** chain deployments:

```bash
# 1. Find the fix commit in the primary chain (e.g., EVM) repo
git log --oneline --grep="fix" -- contracts/StablePool.sol

# 2. Check if the equivalent Solana/Move/Rust program was updated in the same PR
git log --oneline -- programs/stable-pool/src/lib.rs

# 3. Compare on-chain deployment timestamps
#    If EVM pool contract was redeployed 3 months before Solana program ‚Üí gap exists

# 4. Search for chain-specific config guards that might exclude a chain's tests
# Rust: grep -r "cfg(feature" programs/ | grep -v "test-bpf"
# Foundry: check if invariant tests run against a specific forked network only
```

**Report condition:** Security invariant (e.g., pool-ratio check, reentrancy guard, flash-loan callback validation) present in one chain's deployment but absent in another ‚Üí High/Critical (same known attack vector, unpatched surface).

### Hook 3: Yield Vault Strategy Offboarding ‚Äî Deactivated Strategy in NAV Calculation
*Tied to ¬ß2.1 Summer.fi $6.04M vault NAV bug*

Any ERC-4626 meta-vault or Yearn-style yield aggregator that manages multiple strategies through an offboarding/deprecation flow should be checked for the Summer.fi pattern ‚Äî deactivated strategy still included in `totalAssets()`:

```solidity
// VULNERABLE: capped/deprecated strategy still in NAV loop
function totalAssets() public view override returns (uint256 total) {
    for (uint256 i = 0; i < strategies.length; i++) {
        // ‚öÝÔ∏è strategies[] array still contains capped strategies;
        //    estimatedTotalAssets() may use stale oracle or book price
        total += IStrategy(strategies[i]).estimatedTotalAssets();
    }
}

// SAFE: offboarding explicitly removes strategy from NAV
function totalAssets() public view override returns (uint256 total) {
    for (uint256 i = 0; i < strategies.length; i++) {
        if (!isOffboarding[strategies[i]]) {   // or: allocationBPS[s] > 0
            total += IStrategy(strategies[i]).estimatedTotalAssets();
        }
    }
}

// Also check: is offboarding a two-step process (cap, then wait, then remove)?
// If yes: is NAV excluded from step 1, or only step 2?
// The Summer.fi bug: exclusion only happened at step 2, window exploitable at step 1
```

**Report condition:** Yield vault's NAV calculation includes deprecated/capped strategies with stale or manipulable valuations during an offboarding window ‚Üí High/Critical depending on vault TVL and ease of donation attack.
