## 1. 🔴 Active Exploits (last 24–48h)

### 1.1 Edel Finance — $403K | Flash-Loan Wrapped-Token Exchange-Rate Manipulation

| Field | Detail |
|---|---|
| **Protocol** | Edel Finance (V1 lending market, tokenized xStocks collateral) |
| **Chain** | Ethereum / Canton Network |
| **Date** | July 1, 2026 |
| **$ Lost** | ~$403K bad debt (CertiK initial estimate: $204K; team-confirmed: $403K) |
| **Bug class** | flash-loan, price-manipulation |
| **Status** | V1 paused; team absorbing losses 1:1; attacker bridged funds to Tornado Cash |

**Root cause:** Edel V1 accepted `wGOOGLx` (a wrapped form of the xStocks tokenized Alphabet/Google share) as collateral. The contract derived `wGOOGLx` collateral value by multiplying a Chainlink spot price for `GOOGLx` by an internal exchange rate governing the `GOOGLx ↔ wGOOGLx` conversion.

The Chainlink feed was correct — it accurately reported Alphabet's share price in USD. The vulnerability was entirely in the **internal wrapper exchange rate**, which the attacker inflated ~78× (7,700%) using a flash loan to manipulate the GOOGLx/wGOOGLx pool. The protocol then treated `wGOOGLx` collateral as worth 78× its real value and allowed the attacker to borrow real assets against that phantom collateral.

**Attack flow:**
1. Take flash loan to acquire large `GOOGLx` position
2. Dump `GOOGLx` into the wGOOGLx wrapper pool — skews the `GOOGLx → wGOOGLx` exchange rate dramatically
3. Deposit `wGOOGLx` as collateral → protocol prices it at ~78× face value
4. Borrow real assets (USDC / ETH) up to the inflated collateral limit
5. Repay flash loan; exit; bridge proceeds to Tornado Cash

**Key lesson:** External oracle integrity (Chainlink) ≠ internal rate integrity. Any lending protocol that converts collateral through a second step (wrapper, vault share, LP token) before applying an external price is exposed to this class if that internal rate isn't TWAP-protected or is derived from a pool that can be spot-manipulated.

### 2.1 Chainalysis Research: AI-Accelerated Exploitation of Unverified Contracts — $36.7M in H1 2026

**Published:** ~June 10, 2026 (full post-mortem analysis week of July 1)
**Technique class:** unverified-contract (supply-chain + logic-error via AI-assisted decompilation)
**Bug class tag:** unverified-contract

Chainalysis identified **five protocols in H1 2026** where the exploited contract was **unverified on the relevant block explorer** at time of exploit, with combined losses of **$36.7M**.

**The AI uplift:** Modern decompilers (Dedaub, Heimdall, Panoramix) produce readable Solidity from compiled bytecode. Attackers feed decompiled output into LLMs for vulnerability analysis at scale — systematically sweeping all unverified contracts with TVL above a threshold. The largest single case was **Truebit on Ethereum** (January 8, 2026, $26.2M) — the targeted contract had been unverified since 2021 but still held user funds. The same attacker address had drained a smaller protocol 12 days earlier, demonstrating systematic escalation from small test targets to large paydays.

**Why this matters for hunters:**
- Unverified contracts are excluded from most bug bounty programs — attackers exploit them freely
- If your target has unverified satellite contracts (price calculators, fee collectors, migration helpers) holding TVL, those are likely *under-reviewed* and high-value
- Decompile-and-audit of unverified contracts is a legitimate hunt strategy

### 2.3 Wrapped Tokenized-Asset Collateral: New Attack Surface Formalized

The Edel Finance exploit (§1.1) formalizes a class that will recur as tokenized RWA (real-world assets) proliferates in DeFi lending:

**Vulnerable pattern:** `CollateralValue = ExternalPrice(underlying) × InternalRate(wrapper ↔ underlying)`

When `InternalRate` is derived from a spot pool rather than a time-weighted average, a flash loan can shift it dramatically within a single transaction while `ExternalPrice` remains correct. The Chainlink feed is not at fault; the collateral accounting is.

**Exposure mapping:**
- Any lending protocol accepting: vault shares (ERC-4626), LP tokens, wrapped rebasing tokens, or RWA wrappers as collateral
- Specifically: xStocks-based tokens, Ondo Finance's ouTokens, tokenized fund shares, and wrapped LSTs where the exchange rate is determined by a pool rather than an on-chain accumulator

This is a variant of the classic "LP token collateral" oracle manipulation — applied to a new asset class (tokenized equities) that protocols are still learning to price safely.

## 4. 🎯 Hunt Hooks

### Hook 1: Audit Lending Collateral for Wrapper-Token Exchange-Rate Vulnerability
*Tied to Edel Finance $403K (§1.1)*

Any lending protocol accepting non-atomic collateral (vault shares, wrapped RWA tokens, LP tokens) is a candidate for this attack class. Concrete check:

1. **Identify the collateral pricing path:** does the protocol call `exchangeRate()`, `convertToAssets()`, or equivalent on the collateral before applying an external price feed? If yes, that intermediate rate is the attack surface.
2. **Check if the rate is spot or time-weighted:** find the pool or function that determines the wrapper↔underlying exchange rate. If it reads a spot reserve ratio (not a TWAP), it can be flash-loan-shifted in one tx.
3. **Simulate:** fork the target chain, take a large flash loan of the underlying, dump into the wrapper pool, then call `borrow()` against `wToken` collateral. If the health factor improves, you have the bug.

**Bug class:** price-manipulation, flash-loan | **Bounty floor:** High–Critical for most lending protocols

---

### Hook 2: Query Block Explorers for Unverified Contracts with Active TVL
*Tied to Chainalysis AI-assisted unverified-contract attack report (§2.1)*

Attackers are already doing this systematically with AI decompilers. Beat them there:

1. **Etherscan API:** `?module=contract&action=getsourcecode&address=<addr>` — `SourceCode: ""` means unverified. Combine with DeFiLlama TVL data to find unverified contracts with real money.
2. **Decompile the bytecode:** run through [Dedaub](https://library.dedaub.com/) or Heimdall. Feed the output into an LLM prompt: "Find access-control gaps, missing auth checks, and delegatecall misuse in this decompiled Solidity."
3. **Scope check:** most bug bounty programs only cover verified contracts. But some programs list "any contract in the ecosystem controlling user funds." If unverified contracts are in-scope, this is a greenfield opportunity; if not, disclose anyway for reputational credit.

**Bug class:** unverified-contract, access-control | **Effort:** Medium (tooling-heavy) | **Payoff:** potentially critical

### Hook 3: Verify SimpleHelp Patch Status Across Your Target's Infra
*Tied to CVE-2026-48558 CISA KEV deadline (§3.1)*

Today's CISA deadline means any team still running SimpleHelp < 5.5.8 is in violation of federal patch mandates — and is running known-compromised RMM software.

1. **Shodan/Censys:** search `product:"SimpleHelp"` — the RMM exposes a distinct HTTP fingerprint on port 5950. Pre-5.5.8 instances exposed to the internet are confirmed critical.
2. **Bug bounty framing:** demonstrating that an unpatched SimpleHelp instance exposes `PRIVATE_KEY`/foundry keystores to the Djinn Stealer payload is a valid critical infrastructure finding under most programs' scope (key management / infrastructure compromise).
3. **Post-deadline urgency:** with the CISA deadline passed, incident responders are under pressure. If you find an exposed RMM instance and report responsibly, expect rapid triage.

**Bug class:** key-management, access-control | **CVSS base:** 10.0 | **Bounty floor:** Critical
