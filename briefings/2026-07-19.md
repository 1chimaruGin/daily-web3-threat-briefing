## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 macOS Infostealer — Telegram Session Hijacking + Wallet App Replacement (SlowMist, July 2026)

**Bug class:** `key-management` / `supply-chain`
**Chains:** Multi-chain (any software wallet on macOS)
**Status:** Active, no CVE assigned as of July 19

SlowMist has published a forensic breakdown of a macOS information-stealing malware that combines two uniquely dangerous capabilities:

**Capability 1 — Wallet app trojanization.** The malware replaces Ledger Live and Trezor Suite binaries with attacker-controlled "web loader" fakes. When the user opens what they believe is their hardware wallet manager, the fake app prompts them to enter their recovery phrase — which is then exfiltrated immediately. Affected targets include Exodus, Atomic, Electrum, Wasabi, and Monero for software wallets, plus Bitcoin Core, Litecoin Core, Dash Core, and Dogecoin Core full-node clients.

**Capability 2 — Telegram session theft (credential-free).** Instead of stealing login passwords or bypassing 2FA, the malware copies authenticated Telegram Desktop session files (stored as SQLite databases under `~/Library/Group Containers/`). With these files, an attacker can restore full Telegram session access on a different device — **without needing the phone number, verification code, or two-step password**. This bypasses all standard Telegram authentication controls.

**Distribution vector:** Social engineering / trojanized downloads (exact distribution mechanism not yet public).

**Immediate actions per SlowMist:**
1. On macOS: `Settings → Devices → Terminate All Other Sessions` in Telegram
2. Change Telegram two-step verification password and Desktop passcode
3. Verify Ledger Live / Trezor Suite binary integrity (compare hash against official release)
4. Move funds from any potentially-touched software wallet to a new clean wallet

**Sources:** [SlowMist Medium — Telegram Account Compromised, Wallet Swapped](https://slowmist.medium.com/telegram-account-compromised-wallet-swapped-how-does-macos-malware-break-through-your-defenses-dad9bfed9c02) | [SC Media](https://www.scworld.com/brief/macos-malware-hijacks-telegram-sessions-targets-crypto-wallets) | [gncrypto.news](https://www.gncrypto.news/news/macos-malware-steals-telegram-sessions-targets-crypto-wallets/) | [cyberpress.org](https://cyberpress.org/infostealer-hijacks-telegram-wallets/)

### 2.2 PETS 2026 — KU Leuven Wallet Extension Paper Drops Today (Conference Starts July 20)

**Bug class:** `logic-error` (systemic privacy design flaws)
**Chains:** Ethereum (and any EVM chain) — all browser extension wallets
**Paper:** "The Masks We (Think We) Wear: Privacy Threats of Browser-Extension Wallets in the Web3 Ecosystem" — [arXiv 2607.06141](https://arxiv.org/html/2607.06141v1)

Researchers at KU Leuven's DistriNet group tested 85 of the most popular crypto browser extension wallets (combined ~35 million Chrome Web Store installs) and identified five systemic privacy weakness categories:

| Weakness | Scope | Finding |
|---|---|---|
| **Address-linking via RPC requests** | 17 of 85 wallets | Wallet sends two or more user addresses in a single request to a third-party RPC server — that server learns the addresses are linked to the same person |
| **Cross-site tracking** | Multiple wallets | Wallet pings outside servers carrying the user's address in the clear on every DApp visit, enabling site-to-site address correlation |
| **Permission persistence** | 22 of 36 EVM wallets | Granted site permissions survive cookie clears and browser restarts — a site retains address-read access indefinitely once granted |
| **Real-name linkage** | Scenario-demonstrated | On a site with KYC/email, the same address leaks tie a "pseudonymous" crypto identity to a real person |
| **Legacy permission grants** | 36 EVM wallets | No "revoke all permissions" equivalent to browser cookie clearing in most wallets |

**Vendor response at disclosure:**
- MetaMask: "Known issue, no immediate plans to fix"
- Rabby: "Attack is virtually impossible; vulnerability does not exist"
- OKX: Closed as "informational"
- Bybit, Backpack, Core: Marked "low-risk" or "out of scope"
- Coinbase Wallet, Coin98: Fixed by Feb 2026 retest ✓

**Why it matters today:** PoC techniques from the PETS 2026 presentation (July 20–25, Calgary) will circulate starting tomorrow. Expect wallet-vendor disclosure windows and de-anonymization PoC code to become public during the conference week.

**Sources:** [The Hacker News](https://thehackernews.com/2026/07/study-of-85-crypto-wallet-extensions.html) | [KU Leuven press release](https://nieuws.kuleuven.be/en/content/2026/ku-leuven-researchers-expose-hidden-privacy-risks-in-crypto-wallets) | [cybersecurity-research.be](https://cybersecurity-research.be/millions-of-cryptocurrency-users-face-hidden-privacy-risks-in-browser-wallets/) | [Techzine](https://www.techzine.eu/news/security/142820/ku-leuven-discovers-privacy-leaks-in-popular-crypto-wallets/)

### 2.3 Ill Bloom — Ongoing PRNG-Weak Wallet Sweeps ($5M+ Confirmed)

**Bug class:** `key-management` (weak PRNG → low-entropy BIP-39 mnemonic)
**Chains:** Bitcoin, Ethereum, Polygon, Rootstock, Tron
**Checking tool:** [illbloom.org](https://illbloom.org/)

Coinspect disclosed this in early July but sweeps are ongoing. The flaw: some older and lesser-known mobile apps and browser extensions generated BIP-39 recovery phrases using a cryptographically weak PRNG. The resulting phrases have far fewer bits of effective entropy than the standard 128/256, making them brute-forceable at scale.

**Scale confirmed:**
- 2,114 wallets identified as at-risk
- May 27 coordinated sweep: 431 wallets drained → ~$3.1M
- Follow-up: $2.1M USDT stolen from a single exposed wallet
- **Total confirmed: >$5M** across BTC, ETH, Polygon, Rootstock, Tron

Hardware wallets and mainstream software wallets (MetaMask, Trust Wallet) are **not** affected — the flaw is in older/niche wallets. If users created wallets with lesser-known mobile apps or browser extensions circa 2018–2022, checking is advisable.

**Sources:** [techtimes.com](https://www.techtimes.com/articles/319796/20260706/crypto-wallets-lose-5m-broken-random-number-generator-ill-bloom-disclosure.htm) | [cryptobriefing.com](https://cryptobriefing.com/crypto-wallets-at-risk-from-ill-bloom-vulnerability-5m-stolen/) | [beincrypto.com](https://beincrypto.com/ill-bloom-vulnerability-crypto-wallets/) | [coingeek.com](https://coingeek.com/ill-bloom-flaw-puts-2114-digital-wallets-at-risk-report/) | [phemex.com](https://phemex.com/news/article/ill-bloom-wallet-vulnerability-threatens-thousands-of-crypto-accounts-91917)

## 4. 🎯 Hunt Hooks

### Hook 1: Hardware Wallet App Integrity — Can the Binary Be Swapped?
*Tied to §2.1 macOS infostealer — Ledger/Trezor replacement*

The macOS malware targets the **install path of hardware wallet management apps**, not the hardware device itself. The key failure: macOS does not notify users if a binary they trust changes between launches, and Ledger Live / Trezor Suite do not perform in-app self-integrity checks before displaying the "enter recovery phrase" prompt.

**What to look for in bug bounty targets that ship macOS wallet apps:**
```
1. Does the app verify its own binary signature on launch?
   - Expected: codesign --verify --strict --deep <app_bundle>
   - If absent: malleable to replacement attack

2. Does the app ever prompt for recovery phrase on desktop
   (hardware wallets NEVER should — phrase entry is on-device only)?
   - A desktop recovery-phrase prompt is ALWAYS a red flag

3. Is the update mechanism code-signed and verified, or does it
   pull from a URL the process has write permission to?
   - Unsigned updater + writable path = dropper primitive
```

**Bug class:** `key-management`, `supply-chain` | **Ceiling:** Critical

### Hook 2: Browser Extension Wallet — Permission Persistence Audit
*Tied to §2.2 PETS 2026 — KU Leuven wallet extension paper (conference opens July 20)*

With PoC code circulating from the conference starting tomorrow, targets holding wallet extension bounties are high-value this week. The specific finding to check:

```javascript
// Test: After clearing ALL browser cookies and restarting:
// 1. Visit a DApp that previously connected your wallet
// 2. Call: window.ethereum.request({ method: 'eth_accounts' })
// If this returns your address WITHOUT a new approval prompt → permission persisted

// The dangerous pattern in extension manifest background scripts:
// Permission store not tied to browser session storage:
chrome.storage.local.set({ approvedOrigins: [...existing, newOrigin] })
// vs. the safe pattern using session-scoped storage:
chrome.storage.session.set({ approvedOrigins: [...existing, newOrigin] })
// local storage persists forever; session storage clears on browser close
```

**Report condition:** Granted permissions survive full browser restart + cookie clear → High/Critical depending on scope.

### Hook 3: PRNG Seeding in Wallet Keygen — Audit for Ill Bloom Pattern
*Tied to §2.3 Ill Bloom — $5M+ active sweeps*

The Ill Bloom class is not limited to the 2,114 identified wallets — any wallet that used a non-CSPRNG (e.g. `Math.random()`, seeded from timestamp, or platform entropy without `crypto.getRandomValues()`) is in scope. This is a pattern to grep for if you have access to wallet source code as a target:

```javascript
// VULNERABLE patterns in wallet keygen code:
Math.random()                           // never use for key material
new Date().getTime()                    // timestamp-seeded = predictable
seedrandom(seed)                        // deterministic PRNG — if seed is guessable, keys are
Buffer.alloc(16).fill(0)               // zero-filled entropy

// SAFE pattern:
const entropy = new Uint8Array(32)
crypto.getRandomValues(entropy)         // browser/Node Web Crypto API
// or in Node.js:
import { randomBytes } from 'crypto'
const entropy = randomBytes(32)         // OS CSPRNG

// Check in Rust wallet code:
rand::thread_rng()                      // WARN if seeded; OK if using OsRng
rand::rngs::OsRng                      // SAFE
```

**Report condition:** Key material derived from non-CSPRNG entropy source → Critical.
