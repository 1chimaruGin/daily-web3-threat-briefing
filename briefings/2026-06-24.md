> **Window:** Last ~24–48h (June 22–24 UTC). One confirmed drain: the JaredFromSubway counter-MEV honeypot ($15M, June 22) — post-mortems circulated June 23–24. Second signal: ENS lookalike display flaw exposed on June 23 in the wake of the drain.

## 1. 🔴 Active Exploits (last 24h)

### 1.1 JaredFromSubway.eth MEV Bot — ~$15M | Counter-MEV Honeypot Trap | Ethereum

- **Date:** June 22, 2026 (post-mortem reports confirmed and circulating June 23–24)
- **Chain:** Ethereum
- **Root cause:** The attacker deployed 66 fake token contracts precisely mimicking the `name()`, `symbol()`, `decimals()`, and full ERC-20 interface of high-value assets (WETH, USDC, USDT). Each fake was paired with a sham liquidity pool seeded with a small amount of real tokens to produce valid on-chain price signals. The routing was structured so jaredfromsubway's automated sandwich-attack logic would classify these pools as live MEV opportunities.

  When the bot engaged — calling `approve()` and `transferFrom()` on the fake contracts — those calls granted ERC-20 allowances to attacker-controlled `spender` addresses. The fake `approve()` implementation silently recorded these allowances against real token contracts (WETH, USDC, USDT) held by the bot's wallets. After allowances accumulated across ~dozens of the bot's execution wallets over several hours, the attacker triggered a single multicall that swept all approved real assets in one transaction.

- **Bug class:** `logic-error` (bot's automated decision engine treated unverified addresses as legitimate token contracts), `unverified-contract` (fake tokens accepted without registry validation), `access-control` (inadvertent unlimited `approve()` grants to malicious spenders)
- **Amount lost:** ~$15M confirmed across all affected wallets (earlier reports cited $7.5M for the first discovered sweep transaction; total climbed as additional wallets were identified)
- **Fund status:** Attacker's EOA holding stolen WETH/USDC/USDT on Ethereum; not yet moved to mixer as of June 24 UTC
- **Context:** jaredfromsubway.eth has been Ethereum's most active sandwich bot since 2023, with estimated lifetime MEV extraction >$300M. An earlier iteration of this same honeypot technique drained $7.5M from the same operation. This recurrence suggests the bot's allowance-management logic was not hardened after the first event.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Counter-MEV Honeypot Design — A Formalized New Attack Class

The JaredFromSubway drain fully documents a repeatable attack pattern that warrants its own bug class in on-chain security tooling: the **counter-MEV honeypot**.

**Full mechanism:**

```
Step 1 — Deploy fake tokens
  → N contracts that return "WETH", "USDC", "USDT" from name()/symbol()
  → Identical decimals(), totalSupply() to real assets
  → Unverified bytecode on Etherscan (no source submission)

Step 2 — Seed fake liquidity pools
  → Pair fake tokens against small real WETH/USDC amounts in Uniswap V3 forks
  → Real token amount creates a valid price signal
  → Pool structure mimics high-frequency sandwich targets

Step 3 — Let MEV bot engage
  → Bot detects pool creation via mempool scanning
  → Bot calls approve(attacker_spender, MAX_UINT256) on fake token
  → Fake token's approve() writes to attacker's allowance mapping against
     the REAL token contract (or records legitimately, spender = attacker)

Step 4 — Sweep
  → After allowances accumulate across enough bot wallets (hours to days)
  → Attacker calls transferFrom(bot_wallet, attacker, MAX) on real WETH/USDC/USDT
  → All bot wallets drained in a single multicall transaction
```

**Where this applies beyond MEV bots:**
- Any permissionless AMM that allows unverified token listings (e.g., Uniswap V3/V4, Curve factory pools) — the same approval accumulation can target user wallets that interact with fake pools
- Lending protocols with open token whitelists — if a user approves a protocol to move an "asset" that turns out to be a fake, and the protocol stores that approval context, the attacker gains a standing allowance
- On-chain liquidation bots, arbitrage keepers, DEX aggregators — any automated agent that calls `approve()` without validating the recipient is a canonical contract address is vulnerable

### 2.2 ENS Lookalike Display Flaw — Identity Spoofing on Major Wallet UIs

**Date disclosed:** June 23, 2026

In the aftermath of the JaredFromSubway drain, an on-chain message appeared to mock the bot operator. The message was widely attributed to MetaMask on social media, but MetaMask formally denied it — the sender was a lookalike ENS name using Unicode near-homoglyphs, not any official MetaMask address.

**The design flaw:**
- ENS names containing Unicode characters that visually resemble ASCII (e.g., Cyrillic `а` vs Latin `a`, Unicode `‌zero-width non-joiner`, homoglyph substitutions) register as distinct ENS names but render identically in most wallet UIs and block explorers
- Etherscan, MetaMask, Rainbow, and most major interfaces do not punycode-encode or flag Unicode ENS names — `metаmask.eth` (Cyrillic 'а') displays identically to `metamask.eth`
- No wallet confirmed as of June 24 has deployed a visual safeguard

**Attack surface for auditors:**
- Protocols that build on-chain allowlists keyed by ENS name (not address) are directly exploitable: `keccak256(abi.encodePacked("metamask.eth"))` vs `keccak256(abi.encodePacked("metаmask.eth"))` produces different hashes, but a human-reviewed governance proposal listing a "name" is ambiguous
- Frontend code resolving `ethers.provider.resolveName(userInputtedENS)` without Unicode normalization will silently resolve lookalike names to attacker addresses
- Any protocol governance flow where human participants approve a listed ENS name (not address) is susceptible to a spoofed proposal

**Web3 kill chain for CVE-2026-48618:** An attacker positioned between a Node.js RPC client (hardhat, ethers.js, web3.js bot) and an HTTPS RPC endpoint (Infura, Alchemy, QuickNode) can present a certificate for `*.infura.io` that bypasses hostname validation via the Unicode dot separator trick. The attacker receives all JSON-RPC calls — including `eth_sign`, `eth_sendRawTransaction`, and any call that transmits private keys or signed payloads in the clear. Exact kill chain: MITM at ISP/VPN/Wi-Fi layer → intercept `eth_accounts` + `eth_sign` responses → steal signing credentials.

**Action:** `node --version` → if < v22.23.0 / v24.17.0 / v26.3.1, patch now. Any blockchain bot, keeper, relayer, oracle updater, or CI/CD pipeline running on unpatched Node.js is a live TLS MITM target.

## 4. 🎯 Hunt Hooks

**Hook A — Enumerate Unlimited ERC-20 Approvals on Protocol/Bot Wallets**

The counter-MEV drain is a proof-of-concept that unconstrained ERC-20 `approve()` calls are a critical, under-audited attack surface on automated on-chain agents. For any keeper, liquidation bot, arbitrage contract, or MEV relay in scope:

1. Pull all historical `Approval` events emitted where the wallet is `owner`:
   ```
   eth_getLogs({
     topics: [keccak256("Approval(address,address,uint256)"), wallet_address_padded]
   })
   ```
2. Flag any `spender` not present in the protocol's official contract registry.
3. Flag any `value == type(uint256).max` (unlimited approval) to an external/unverified address.
4. Cross-reference each `spender` against Etherscan bytecode verification status — unverified spender = immediate critical escalation.

For protocols (not bots), run the same check on any user-facing `approve()` flow: does the protocol ever store an approval mapping to a mutable or upgradeable spender? If the spender contract can be replaced via a proxy, the stored allowance becomes a post-upgrade drainpipe.

**Hook B — Fake Token Detection in Permissionless Liquidity Systems**

For any AMM with permissionless pool creation in scope:

1. Retrieve all pools created in the last 7–14 days.
2. For each new token in those pools: check bytecode verification on Etherscan. Unverified = flag.
3. Compare `name()` / `symbol()` return values against a canonical token registry (CoinGecko API, Uniswap token list). Any unverified contract claiming to be "WETH" or "USDC" is the honeypot setup.
4. Check the pool's liquidity ratio: a pool seeded with $500 real USDC / $50M "fake USDT" is not an arbitrage opportunity — it's a trap. Any pool where one leg's contract is unverified and the price is significantly off-peg is suspect.
5. If the AMM's router blindly calls `transferFrom()` on user-provided token addresses without validating against a whitelist → critical `unverified-contract` finding.

**Hook C — ENS Unicode Normalization in On-Chain Access Control**

For any protocol that parses, stores, or compares ENS names in smart contract logic:

1. Check `keccak256(abi.encodePacked(name))` usage — if the contract compares raw ENS name strings, a Unicode lookalike will produce a distinct hash and pass undetected through allowlist checks, OR bypass access control if the attacker pre-registers the homoglyph.
2. Look for `IENSResolver.resolve(name)` calls in governance or access-control flows — if a human-readable name from a proposal is resolved on-chain without normalization, a spoofed proposal with a lookalike ENS name routes to the attacker's address.
3. In frontend audits: check whether `ethers.provider.resolveName()` input is sanitized with Unicode normalization (NFKC) before display — raw Unicode ENS names should be punycode-encoded before showing to users. Flag any UI that displays the raw ENS label without punycode conversion.
