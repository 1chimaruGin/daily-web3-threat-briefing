## 1. 🔴 Active Exploits (last 24h)

### 1.1 Wanchain Cardano-BNB Bridge — ~$10–13M | Signature Encoding Flaw | Cardano → BNB Chain

- **Date:** July 21, 2026 (UTC)
- **Chain:** Cardano → BNB Chain
- **Bug class:** `signature-replay`, `bridge-dvn`
- **Amount lost:** ~$10–13M (515.2M NIGHT / Midnight tokens)

Wanchain took its Cardano-to-BNB Chain bridge offline on July 21 after attackers drained **515.2 million NIGHT tokens** across four transactions in an eight-minute window. The attacker sold ~300M NIGHT on exchanges immediately, crashing the token by 30%+ to a new all-time low (~$0.016).

**Root cause (BlockSec analysis):** Wanchain's bridge built signed withdrawal messages by **raw-concatenating 14 variable-length redeemer fields without delimiters or length prefixes.** This means different combinations of field values can collapse to the same byte string and produce the same hash. The attacker crafted a different set of field values — one corresponding to a vastly larger withdrawal — that hashed identically to a legitimate, already-signed message. A single valid bridge signature therefore authorized a payout approximately **65,000× larger than Wanchain had approved.**

Crucially, Wanchain's bridge contract already contained Cardano's `SerialiseData` function (which would have length-prefixed fields correctly) but never called it for message construction.

**Attack anatomy:**
1. Identify a prior legitimate cross-chain redemption bearing a valid bridge-guardian signature
2. Construct alternative field combination (larger amount, same output hash) that passes the signature check
3. Submit four transactions draining the treasury in rapid succession
4. Immediately swap ~300M NIGHT to stablecoins/ETH on-chain before bridge can halt

**Response:** Wanchain bridge paused; investigation ongoing. Midnight Foundation confirmed its core protocol, validators, and consensus infrastructure were unaffected. No bounty offer announced at time of writing.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Ill Bloom — Broken RNG in Wallet Seed Generation (Ongoing Exploitation)

**Disclosed:** July 6, 2026 by Coinspect Security Research
**Bug class:** `key-management`
**Chains affected:** Bitcoin, Ethereum, Polygon, Rootstock, Tron
**Total stolen:** ~$5M (2,114+ wallets compromised as of disclosure date)

The "Ill Bloom" disclosure remains the most broadly relevant ongoing advisory for hunters who work on wallet integrations or dApps that include wallet creation flows. Coinspect identified that certain mobile software wallets used defective user-space random number generators (instead of platform-provided cryptographically secure RNG) to generate BIP-39 mnemonic phrases, making seed entropy dramatically smaller than the required 128/256 bits.

Attackers pre-computed the reduced entropy keyspace offline and are actively sweeping matching addresses. The attack is fully automated and targets wallets created with vulnerable apps; hardware wallets and most major software wallets are **not** affected.

**Coinspect published a free wallet-checker tool at [illbloom.org](https://illbloom.org/) — check your own addresses and advise users of any dApp you audit.**

**Why it matters for bug hunters:** Any dApp or SDK that wraps wallet creation (onboarding flows, embedded wallets, social-login wallets) should be scrutinized for RNG source. If a mobile app calls a framework's `Math.random()` equivalent or a date-seeded PRNG instead of `crypto.getRandomValues()` (browser) or `SecureRandom` (Android) for entropy, this is an immediately critical finding.

**Chain:** Aptos
**Potential exposure:** ~$70B (stablecoins + bridges on Aptos at time of discovery)
**Funds lost:** $0 (responsible disclosure + fast patch)

Hexens researchers discovered a flaw in the **Aptos Move virtual machine** — the engine that runs every smart contract on Aptos — that could have allowed an attacker with a $3,000 commodity server to simulate approximately 1/3 of the validator network. Under real network conditions, the attack had a >90% success rate and required no insider access or special permissions. **Why it matters for hunters:** This confirms that Move VM / Aptos-layer bugs exist with systemic, protocol-level impact. Aptos smart contract auditors should review validator interaction surfaces and epoch boundary conditions for related classes. The full technical writeup is at the CoinDesk article linked below.

## 4. 🎯 Hunt Hooks

### Hook 1: Bridge Message Encoding — Variable-Length Field Boundaries
*Tied to §1.1 Wanchain Cardano Bridge $10–13M — signature encoding collision*

The Wanchain exploit is the sharpest 2026 demonstration of **ABI encoding vs. raw concatenation** as a trust boundary. Any bridge that constructs signed cross-chain messages by stringing together variable-length fields without typed delimiters is potentially vulnerable to the same class.

**What to check in bridge contracts/circuits:**

```solidity
// VULNERABLE — raw concatenation: different (amount, recipient) pairs
// can produce the same hash if field lengths are unconstrained
bytes memory msg = abi.encodePacked(amount, recipient, chainId, nonce);
bytes32 hash = keccak256(msg);

// SAFE — abi.encode includes type metadata and length prefixes:
bytes32 hash = keccak256(abi.encode(amount, recipient, chainId, nonce));
// Even safer: EIP-712 typed structured data
bytes32 hash = _hashTypedDataV4(
    keccak256(abi.encode(WITHDRAWAL_TYPEHASH, amount, recipient, chainId, nonce))
);
```

**For non-EVM bridges (Cardano / Move / Rust):** verify that the off-chain guardian/validator serialises message fields using a length-prefixed or schema-typed encoding (e.g., Cardano's `SerialiseData`, BCS for Move, Borsh for Solana), not ad-hoc byte concatenation.

**Report condition:** Bridge signs a message assembled with `abi.encodePacked` (or its non-EVM equivalent) over multiple variable-length parameters where an attacker can control at least two adjacent fields → Critical.

### Hook 2: DVN/Guardian Signature Scope — Chain-ID and Nonce Binding
*Tied to §1.1 Wanchain; also §ongoing Altura `bridge-dvn`*

Two active incidents (Wanchain and Altura) involve bridge guardian / DVN verifier key failures. Beyond encoding attacks, audit whether guardian-signed messages cryptographically bind:

- **Chain ID** of both source and destination chains (prevents cross-chain replay)
- **Contract address** of the bridge on each chain (prevents deployment address replay)
- **Nonce or timestamp** (prevents message replay across calls)
- **Exact amount and recipient** in the signed payload (prevents field substitution)

If any of these are missing from the signed scope, a valid signature from one context can be replayed in another.

```solidity
// Minimal safe signed payload for a bridge withdrawal:
bytes32 msgHash = keccak256(abi.encode(
    DOMAIN_SEPARATOR,   // includes chain-id + contract address
    keccak256(abi.encode(
        WITHDRAWAL_TYPEHASH,
        srcChainId,
        dstChainId,
        nonce,          // per-user monotonic; check for gaps
        amount,
        recipient
    ))
));
require(ECDSA.recover(msgHash, sig) == trustedGuardian, "bad sig");
```

**Report condition:** Guardian signature does not include chain ID, contract address, or nonce → High/Critical (cross-chain or replay attack vector).

### Hook 3: Wallet / SDK Seed Generation — RNG Source Audit
*Tied to §2.1 Ill Bloom — broken RNG, $5M stolen, 2,114+ wallets*

If a target dApp or SDK you're hunting includes a wallet onboarding or key generation flow, check the entropy source:

**Browser/JS — safe vs. vulnerable:**
```javascript
// VULNERABLE: seeded or user-space PRNG
const entropy = Math.random().toString();  // DO NOT USE for key material

// SAFE: platform CSPRNG
const entropy = new Uint8Array(32);
crypto.getRandomValues(entropy);   // Web Crypto API — required for key gen
```

**React Native / mobile:**
```javascript
// VULNERABLE: React Native's Math.random() is not cryptographically secure
// SAFE: use react-native-get-random-values + the polyfill for crypto.getRandomValues()
import 'react-native-get-random-values';
import { generateMnemonic } from 'bip39';
```

**Server-side (Node.js):**
```javascript
const { randomBytes } = require('crypto');   // SAFE — wraps OS CSPRNG
// Never: Math.random(), Date.now() seeds, or any user-supplied entropy for key material
```

**Report condition:** Any key-generation code path reachable in production that sources entropy from a non-CSPRNG → Critical (entire keyspace predictable offline).
