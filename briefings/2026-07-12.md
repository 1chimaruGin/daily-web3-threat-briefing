## 1. 🔴 Active Exploits (last 24h)

### Bonzo Lend — Oracle Price Manipulation, $9M (Hedera Mainnet) 🆕

**Date:** July 11, 2026 (~00:51 UTC)
**Protocol:** Bonzo Finance / Bonzo Lend — the largest lending protocol on the Hedera network
**Chain:** Hedera Mainnet; $5.25M subsequently bridged to Ethereum
**$ Lost:** ~$9.05M total; ~$5.25M confirmed cross-chain to Ethereum (hacker holds 2,360 ETH + 15.58 WBTC as of last report)
**Bug class:** `oracle-manipulation`, `price-manipulation`

**Attack summary:**

The attacker submitted a manipulated SAUCE/HBAR price to Bonzo's on-demand oracle contract. SAUCE was trading at approximately 0.2 HBAR on-chain; the malicious price update inflated the reported value by **roughly 12 orders of magnitude**.

Eight seconds after the false price went live, the attacker deposited just **250 SAUCE** (real value: a few dollars) and used it as collateral to borrow:

- **6.6 million USDC**
- **34.5 million Wrapped HBAR (WHBAR)**

No flash loan was needed — the oracle inflation made pennies of collateral appear to be worth hundreds of millions. Funds were then rapidly bridged out through Hedera's canonical bridge to Ethereum, with the attacker converting to ETH and WBTC.

**Root cause:** Bonzo's on-demand oracle accepted the price update without (a) a TWAP or staleness window, (b) a deviation circuit-breaker comparing against a secondary reference price, or (c) an authorized-sender allowlist for price submissions. The oracle update was permissionlessly writable by any address.

**Current status:** Bonzo Lend deposits and borrowing halted. Team has acknowledged the exploit and is investigating. No recovery attempts confirmed yet.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Ill Bloom — Weak PRNG in Wallet Seed Generation, $5M+ Drained

**Disclosed by:** Coinspect (security firm)
**Bug class:** `key-management`
**Chains affected:** Bitcoin, Ethereum, Tron, Rootstock (RSK), Polygon
**Wallets at risk:** 2,114 confirmed vulnerable addresses (as of June 30 analysis)
**Total stolen:** $5M+ confirmed; $3.1M in a coordinated sweep on May 27 (431 wallets), $2.1M USDT from a single exposed wallet subsequently

**Technical summary:**

The "Ill Bloom" vulnerability is a **weak entropy / insecure PRNG flaw** in the seed-phrase generation logic of a subset of older and lesser-known software wallets (mobile apps and browser extensions, some dating back to 2018). When the wallet initializes a new seed phrase, it draws from an insufficiently seeded random number generator. The resulting 12- or 24-word BIP-39 phrase has far fewer bits of effective entropy than the standard requires, making it feasible for an attacker with the Ill Bloom tooling to brute-force and recover the private key from the observable on-chain address alone.

**Affected wallet category:** Primarily older/obscure wallets — **hardware wallets are unaffected**. Major mainstream software wallets (MetaMask, Trust Wallet, Phantom) have not been identified as vulnerable. The real attack surface is the long tail of lesser-known wallets.

**Attack pattern observed:** The May 27 coordinated sweep hit 431 wallets simultaneously, suggesting the attacker ran automated scanning against a pre-built table of brute-forced keys ("rainbow table" analogy for seed phrases). The wallets were drained in a single coordinated block of transactions.

**Remediation:**
1. Any wallet whose seed phrase was generated in an older or obscure app should be treated as compromised — migrate all funds to a new seed phrase generated in audited software or hardware.
2. Hunters: this is an active drainer; any wallets you control in old apps need immediate migration.

### 2.2 Summer.fi Post-Mortem Published: Stale Ark NAV + Fleet Commander Accounting Gap

**Source:** [Lazy Summer USDC Vault Exploit Post-Mortem — blog.summer.fi](https://blog.summer.fi/lazy-summer-usdc-vault-exploit-post-mortem-what-happened-and-what-comes-next/)
**Bug class:** `flash-loan`, `logic-error`

The Summer.fi team published their full post-mortem. Key clarification from yesterday's analysis: this was **not a pure ERC-4626 donation attack** — it was an **operational failure** combined with stale-oracle NAV.

The precise chain of events:
1. A Silo Finance "Varlamore USDC Growth" vault became stranded after the November 2025 Stream Finance collapse. Its on-chain `totalAssets()` continued accruing interest on locked USDC it could never access — an insolvent but interest-accruing ghost balance.
2. Bonzo's Fleet Commander offboarded this Silo "Ark" (capped deposits to zero) but **failed to exclude it from NAV calculations**. The ghost balance remained visible in `totalAssets()`.
3. The attacker accumulated the Silo vault tokens (worthless) over ~3 months, then donated them to the Ark in a flash-loan-funded atomic transaction.
4. The vault saw the donated tokens as a NAV increase and the share price jumped. The attacker redeemed pre-accumulated shares at the inflated price.

**The lesson vs. yesterday:** The single most important check is not just donation-inflation of `totalAssets()`, but **whether removed/offboarded strategies are fully excluded from accounting**. An Ark with zero deposit cap can still be `totalAssets()`-counted if the exclusion logic is incomplete.

## 4. 🎯 Hunt Hooks

### Hook 1: On-Demand Oracle — Can Any Address Submit a Price Update?
*Tied to §1 Bonzo Lend $9M*

The Bonzo Lend exploit is a textbook permissionless oracle poisoning. Before testing the complex paths, start here — it's the fastest Critical bug class to identify:

```bash
# 1. Find the oracle contract address from the lending protocol's config
cast call <LENDING_POOL> "getPriceOracle()(address)" --rpc-url $RPC

# 2. Check if price submission is access-controlled
cast call <ORACLE> "owner()(address)" --rpc-url $RPC
# Also check for role-based: hasRole(PRICE_UPDATER_ROLE, <your_address>)

# 3. Try calling the update function with a test price (on a fork, NOT mainnet)
# e.g. common signatures:
cast call <ORACLE> "setAssetPrice(address,uint256)()" <ASSET> <INFLATED_PRICE> --rpc-url $FORK_RPC

# 4. If updateable — check for deviation guard:
cast call <ORACLE> "maxPriceDeviation()(uint256)" --rpc-url $RPC
# No function? No deviation guard exists.

# 5. Check for TWAP or staleness window:
cast call <ORACLE> "validityPeriod()(uint256)" --rpc-url $RPC
# Zero or absent = no staleness check.
```

**Report condition:** Price updater is permissionless (any EOA can call) AND no deviation circuit-breaker exists. For any lending protocol, this is direct loss-of-funds = Critical.

**Also check:** Does the protocol use a "fallback" oracle that is less protected than the primary? Attackers often target the fallback.

**Bug class:** `oracle-manipulation`, `price-manipulation` | **Ceiling:** Critical | **Bounty potential:** $500K–$2M+

### Hook 2: Offboarded Strategy Still In NAV? The "Ghost Ark" Pattern
*Tied to §2.2 Summer.fi post-mortem*

Any yield aggregator that allows strategy rotation (Yearn, Beefy, Morpho, custom Fleet Commander-style) is vulnerable if the offboarding logic doesn't atomically remove the strategy from `totalAssets()` at the same time it caps deposits. Check:

```solidity
// Red flag pattern — strategies list not cleaned up on offboarding:
function totalAssets() public view returns (uint256) {
    uint256 total = 0;
    for (uint i = 0; i < strategies.length; i++) {
        // ← strategies.length may still include "removed" strategies
        //   if removal only set a depositCap to zero but didn't pop the array
        total += IStrategy(strategies[i]).totalAssets();
    }
    return total;
}

// Check the offboarding function:
function offboardStrategy(address strategy) external onlyOwner {
    strategyDepositCaps[strategy] = 0; // ← only caps; does NOT remove from accounting!
    // Missing: strategies.remove(strategy) or equivalent
}
```

**Test steps:**
1. Find all strategies in a vault's strategy registry
2. Look for any with `depositCap == 0` or `isActive == false` but still listed
3. Check if their `totalAssets()` return value is still counted in the vault's total
4. Check if those strategies hold any stale/bad-debt tokens that report inflated values

**Bug class:** `logic-error`, `flash-loan` | **Ceiling:** Critical

### Hook 3: Wallet Seed Entropy Audit for Old Integrations
*Tied to §2.1 Ill Bloom*

If your target is a protocol that ships its own wallet, embedded key generation, or uses a custom BIP-39 library (common in older mobile DeFi apps, browser extensions, and embedded wallet SDKs), audit the entropy source:

```javascript
// Red flag: using Math.random() or Date.now() as entropy source
const entropy = Math.random().toString(36); // ← CRITICAL: not cryptographically secure
const mnemonic = bip39.entropyToMnemonic(Buffer.from(entropy));

// Red flag: seeding with predictable values
const seed = crypto.createHash('sha256').update(Date.now().toString()).digest();

// Correct pattern:
const entropy = crypto.randomBytes(32); // Node.js / browser crypto.getRandomValues()
const mnemonic = bip39.entropyToMnemonic(entropy);
```

**Where to look:**
- Custom wallet SDKs shipped by protocols (look for `generateMnemonic`, `createWallet`, `entropyToMnemonic` in mobile or web codebases)
- Browser extension wallets with custom key derivation
- Any wallet codebase older than 2021 that hasn't been audited for PRNG use

**Bug class:** `key-management` | **Ceiling:** Critical | **Bounty scope:** Immunefi programs for wallet providers, mobile wallet apps
