## 1. 🔴 Active Exploits (last 24h)

### 1.1 42DAO — Balance Coin (BLC) Oracle Exploit — ~$915K | Oracle Manipulation | BNB Chain

- **Date:** July 22, 2026 (UTC)
- **Chain:** BNB Chain
- **Bug class:** `oracle-manipulation`, `price-manipulation`
- **Amount lost:** ~$912K–$915K (confirmed by PeckShield and SlowMist)

42DAO's Balance Protocol stablecoin **BLC** was hit by a Median Oracle manipulation on July 22. The attacker fed an artificially low Bitcoin price into the protocol's BTCB/USD price feed, triggering unwarranted liquidations of collateralized vaults. With the liquidation logic compromised, the attacker was then able to mint approximately **4.5 million BLC tokens from the zero address** — tokens backed by nothing — and immediately swap them for real Binance-pegged USDT and BTCB via PancakeSwap V2.

A second wave ~2 hours later used the same technique to generate another 5,900 BLC and drain additional liquidity reserves. Combined, the two transactions extracted the full ~$915K.

**Attack anatomy:**
1. Submit falsified (artificially low) BTCB price to Balance Protocol's Median Oracle
2. Fake price triggers forced liquidations — vault accounting breaks
3. Mint ~4.5M BLC from address(0) (no collateral backing)
4. Swap BLC → USDT + BTCB on PancakeSwap V2 while price still stale
5. Repeat with 5,900 BLC ~2 hours later; drain additional reserves

**Impact:** BLC depegged from $1.00 to $0.001209 — a 99.75% collapse. Affected holders have no insurance fund. As of this briefing, 42DAO has **not issued a post-mortem or recovery statement**.

**Root cause note:** The Median Oracle used by 42DAO is an on-chain price aggregator fed by the protocol's own price reporters (not a decentralised oracle network like Chainlink). A single manipulated submission was sufficient to skew the median under certain quorum/weighting conditions.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 July 2026 Oracle Attack Wave — Three Distinct Kill Chains, One Month

**Pattern disclosed:** July 2026 ongoing (synthesized Jul 23)
**Bug class:** `oracle-manipulation`, `key-management`, `price-manipulation`
**Chains affected:** Arbitrum, BNB Chain, Hedera, Ethereum

July 2026 has produced the densest oracle-attack sequence on record, and the incidents reveal **three structurally distinct oracle kill chains** that hunters should treat as separate vulnerability classes:

**Kill Chain A — Trusted Feed Poisoning (42DAO BLC, Bonzo Lend):**
The attacker is an authorized price reporter, or exploits a weak quorum requirement, to push a falsified price directly into the oracle mechanism. No key theft required — the protocol's own price-update access controls are the weak point.
- 42DAO (Jul 22): Median Oracle quorum bypassed; ~$915K
- Bonzo Lend (Jul 12): 12-orders-of-magnitude price manipulation via on-chain reporter; ~$9M

**Kill Chain B — Oracle Signer Key Compromise (Ostium):**
The attacker exfiltrates or social-engineers the private key of a trusted oracle signer. They then submit fake price reports — often backdated or future-dated — offline, bypassing any on-chain rate limiting. The attack is off-chain until submission.
- Ostium (Jul 15): Fake BTC price of $5,000 (real: ~$60K); $18–24M drained from OLP vault

**Kill Chain C — Stale / Deactivated Oracle in NAV (Summer.fi):**
The protocol relies on book-value pricing from a deprecated or offboarding strategy whose real value has diverged from its on-chain accounting value. The attacker donates assets at book price to inflate NAV, then arbitrages.
- Summer.fi (Jul 6): Silo vault tokens donated into capped Ark; stale NAV → $6.04M

**Significance for hunters:** A target protocol's "oracle audit" must check for all three kill chains separately — C-suite level access controls, signer key operational security, quorum mechanics, AND deprecated-strategy NAV accuracy. Most audit checklists still treat oracle manipulation as a single item.

### 2.2 OpenZeppelin Contracts Wizard CVE-2026-48054 — Developer Toolchain Supply Chain Reminder

**Disclosed:** June 2026. Patch available since OZ Wizard ≥ 0.10.9.
**Bug class:** `supply-chain`
**Severity:** HIGH (CVSS 8.8)
**Still relevant:** Still widely unpatched in dev environments as of July 2026.

A quick reminder since many protocol teams remain on older Wizard versions: CVE-2026-48054 allows code injection into Hardhat and Foundry test files generated by the Wizard when `opts.name` or `opts.uri` parameters contain newline characters or other delimiter-breaking input. The injected code runs **on the developer's machine** during test execution — a supply-chain ACE into the build environment where private keys, `.env` files, and Foundry keystores live.

**If you find any project's CI/CD pipeline generating OZ Wizard scaffolds without pinning `@openzeppelin/wizard>=0.10.9`, that's a confirmed-exploitable supply-chain entry point into their build environment.** This is especially dangerous when CI runners have access to mainnet deployment keys.

## 4. 🎯 Hunt Hooks

### Hook 1: On-Chain Median Oracle Quorum Requirements
*Tied to §1.1 — 42DAO BLC $915K, oracle quorum bypassed on BNB Chain*

Any protocol that maintains its own on-chain price oracle (not using Chainlink, Pyth, or another hardened external network) should be audited for quorum manipulation. The 42DAO attack succeeded because a single falsified submission could skew the Median Oracle under its quorum/weighting rules.

**What to check:**

```solidity
// RED FLAG: oracle median computed from too few reporters
// or reporters are insufficiently permissioned
function getMedianPrice(address asset) external view returns (uint256) {
    uint256[] memory prices = reportedPrices[asset];
    // ⚠️ If prices.length can be 1-3 and reporters are not decentralized,
    //    a single colluding/compromised reporter controls the median
    return _computeMedian(prices);
}

// ALSO CHECK: who can call submitPrice()?
// - Is it permissioned to a small set of addresses?
// - Are those addresses EOAs (single point of compromise)?
// - Is there a time-weighted average that limits single-block manipulation?
```

**Specific risk pattern for stablecoins:** Protocols that use a home-rolled oracle to track a collateral asset's price (BTCB, LSTs, RWAs) and then allow minting against that collateral are at maximum risk — if the oracle can be poisoned, the mint function becomes a free money printer.

**Report condition:** Oracle quorum can be satisfied by ≤ 3 reporters that are EOAs or controlled by a single team → High/Critical (single social-engineering or key-leak event enables unlimited mint).

### Hook 2: Protocol-Owned Oracle Signer Keys — Operational Security
*Tied to §2.1 Kill Chain B — Ostium $18–24M, oracle signing key exfiltrated*

Ostium's attacker did not exploit a smart contract bug. They obtained the private key of the oracle's `PriceUpKeep Forwarder` — an off-chain role that signs and submits price data. The on-chain contract had no way to distinguish a legitimate submission from an attacker-signed one.

**What to audit on-chain AND off-chain:**

```solidity
// ON-CHAIN: check how oracle submitter addresses are managed
function setOracleSigner(address newSigner) external onlyOwner {
    // ⚠️ If this is an EOA (no multisig), one key leak = full oracle compromise
    oracleSigner = newSigner;
}

// BETTER: require multisig + time-lock for oracle signer rotation
// AND: implement maximum price deviation per block
uint256 public constant MAX_PRICE_DEVIATION_BPS = 500; // 5% per update
function submitPrice(uint256 newPrice, bytes calldata sig) external {
    require(_verify(sig, newPrice), "bad sig");
    uint256 current = latestPrice;
    require(
        newPrice >= current * (10000 - MAX_PRICE_DEVIATION_BPS) / 10000 &&
        newPrice <= current * (10000 + MAX_PRICE_DEVIATION_BPS) / 10000,
        "deviation too large"
    );
    latestPrice = newPrice;
}
```

**Off-chain checks (include in scope for operational security findings):**
- Is the oracle signer key stored in a cloud environment (AWS Secrets Manager, GCP Secret Manager) with single-person access?
- Does the protocol use a Hardware Security Module (HSM) or MPC threshold signing for oracle key custody?
- Are there circuit-breakers that pause the protocol if the oracle price deviates >X% in one block?

**Report condition:** Protocol-owned oracle signer is an EOA with no deviation circuit breaker → High (key compromise = oracle fully owned, no on-chain safeguard).

### Hook 3: Stablecoin Minting Gates — Oracle Freshness + Collateral Sanity Checks
*Tied to §1.1 42DAO BLC — minted 4.5M BLC from address(0) after oracle poisoning*

The 42DAO attack ultimately succeeded because after the oracle was poisoned, the **mint function had no secondary sanity check** — it trusted the oracle price unconditionally to authorize minting.

**Check any protocol where minting is gated on an oracle price:**

```solidity
// VULNERABLE: mint trusts oracle price with no circuit breaker
function mintStablecoin(uint256 collateralAmount) external {
    uint256 collateralPrice = oracle.getPrice(COLLATERAL_TOKEN); // ← single source
    uint256 mintAmount = collateralAmount * collateralPrice / 1e18;
    _mint(msg.sender, mintAmount);
}

// SAFER: price sanity bounds + max mint per block
uint256 public constant MIN_ACCEPTABLE_BTC_PRICE = 50_000e18; // $50K floor
uint256 public constant MAX_ACCEPTABLE_BTC_PRICE = 200_000e18; // $200K ceiling
uint256 public mintedThisBlock;
uint256 public constant MAX_MINT_PER_BLOCK = 100_000e18; // $100K cap

function mintStablecoin(uint256 collateralAmount) external {
    uint256 collateralPrice = oracle.getPrice(COLLATERAL_TOKEN);
    require(collateralPrice >= MIN_ACCEPTABLE_BTC_PRICE, "price below floor");
    require(collateralPrice <= MAX_ACCEPTABLE_BTC_PRICE, "price above ceiling");

    uint256 mintAmount = collateralAmount * collateralPrice / 1e18;
    mintedThisBlock += mintAmount;
    require(mintedThisBlock <= MAX_MINT_PER_BLOCK, "mint cap exceeded");
    _mint(msg.sender, mintAmount);
}
```

**Report condition:** Stablecoin mint function accepts any oracle price without hardcoded sanity bounds or per-block mint caps → High/Critical depending on collateral TVL.
