## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Aztec RollupProcessor `escapeHatch()` — Full Technical Autopsy

SlowMist and Protos published a complete technical breakdown of the June 17–18 Aztec second exploit. The June 20 briefing listed it as a table entry; here is the full kill chain:

**Vulnerable function:** `RollupProcessor.escapeHatch()`

Designed as a backup withdrawal mechanism for users when normal rollup operations fail, the function lacked **all three expected authorization safeguards**:
- No `onlyOwner` modifier
- No rollup-provider address validation
- No signature verification

**Precise attack sequence:**

```
1. Attacker identifies rollupSize == 0 in contract state
   (zero-size rollup is a valid degenerate state that passes TurboVerifier)

2. Crafts malicious proofData with attacker-controlled public inputs:
      publicOutput → attacker EOA (0x6952…8e97f)
      outputOwner  → attacker EOA
      assetId      → ETH / DAI / renBTC

3. Calls escapeHatch() — no access control, anyone can call

4. TurboVerifier accepts proof (mathematically valid for rollupSize=0)

5. processDepositsAndWithdrawals() executes withdrawals using proofData
   public inputs directly — no cross-reference with deposits mapping,
   no on-chain ownership proof required

6. Single transaction drains:
      1,158 ETH  +  150,000 DAI  +  0.4696 renBTC  ≈ $2.21M
```

**Root cause classification:**
- `access-control` — `escapeHatch()` callable by any EOA; no access gates
- `logic-error` — `processDepositsAndWithdrawals()` trusts proof public inputs as authoritative ownership proof; valid ZK proof ≠ valid withdrawal claim
- `unverified-contract` — immutable, deprecated, unpausable; the only fix required pre-deprecation hardening

**The generalizable pattern — "Trusted Emergency Exit":**

The ZK verifier correctly validated the proof. The exploit did not require an invalid proof — it required the *downstream handler* to blindly trust the proof's public inputs without an independent on-chain ownership check.

This is a distinct and underappreciated class:

> **Verifier validates proof ≠ Contract validates beneficiary**

Any ZK rollup or bridge with an emergency/escape/forced-exit function is a candidate. The invariant to check:

```
Does the emergency path independently verify that msg.sender
OR the claimed outputOwner has a valid deposit receipt or
balance proof — independent of the ZK proof itself?
```

Auditors routinely verify the happy-path proof verification flow and skip access controls on rarely-used fallback paths. The degenerate state condition (`rollupSize == 0`, empty merkle roots) that unlocks the escape path under adversarial inputs is almost never tested.

### 2.2 Spoof-Proof-Public-Inputs: The Wider ZK Attack Surface

The Aztec case is now the second June 2026 ZK incident (the first was the June 14 Aztec Connect L1/L2 state discrepancy) that hinges on the same core assumption violation: **the ZK layer validates mathematical soundness; it does not validate economic legitimacy of the claimed outputs**.

Pattern map for hunters — where this surfaces in live codebases:

| Contract pattern | What to look for | Risk |
|-----------------|-----------------|------|
| `escapeHatch(bytes proofData)` | Missing access control + no deposits cross-check | Critical |
| `processWithdrawals(Output[] outputs)` populated from public inputs | No mapping-based ownership verification | High |
| `rollupSize == 0` degenerate path | Separate code path that may skip normal auth checks | High |
| Off-chain proof generator + on-chain executor | Executor trusts proof outputs as ownership proof | High |

**Grep to run against any ZK rollup target:**

```bash
# Find emergency/escape functions without access modifiers
grep -n "function.*[Ee]scape\|function.*[Ee]mergency\|function.*[Ff]orced" \
    contracts/**/*.sol | grep -v "onlyOwner\|onlyAdmin\|require("

# Find withdrawal logic that reads from proof public inputs
grep -n "proofData\|publicOutput\|outputOwner\|assetId" \
    contracts/**/*.sol | grep -n "transfer\|withdraw\|send"
```

## 3. 🟢 Advisories / CVEs

### 3.1 CVE-2026-48907 — Joomla JCE Editor — CVSS 10.0 — CISA KEV (June 16)

| Field | Detail |
|-------|--------|
| **Affected** | Joomla Content Editor (JCE) ≤ 2.9.99.4 (both Core free and Pro editions below 2.9.82) |
| **Fixed in** | JCE 2.9.99.5 (early June 2026); additional hardening in 2.9.99.6 |
| **CVSS v4.0** | **10.0 Critical** — CWE-284 Improper Access Control |
| **CISA KEV** | Added June 16, 2026 — **actively exploited in the wild** |
| **Attack type** | Unauthenticated RCE via arbitrary PHP file upload |

**Attack flow (two HTTP requests):**
1. POST to JCE profile management → creates malicious editor profile with permissive upload handler
2. POST to upload handler → uploads PHP webshell; GET to uploaded path = RCE inside Joomla web root

**Why this matters to crypto bug hunters:**

Many DeFi governance portals, community wikis, and documentation sites run Joomla with JCE for rich-text editing. A compromised web server can expose:
- `.env` files containing hot wallet private keys or API secrets
- Admin credentials for multisig relayer dashboards
- Discord/Telegram webhook tokens used for on-chain governance signals
- Developer SSH keys cached in web directories
- Infura/Alchemy API keys embedded in front-end config

CISA confirmed active exploitation. Any Joomla+JCE instance a protocol runs should be treated as a live critical finding pending patch.

**Patch:** Update JCE to ≥ 2.9.99.6 immediately. Temporary mitigation: disable file upload capability in JCE global configuration.

## 4. 🎯 Hunt Hooks

**Hook A — ZK rollup emergency-exit path audit**
For any ZK rollup or bridge target with an `escapeHatch()`, `emergencyWithdraw()`, `forcedExit()`, or equivalent backup path: (1) Confirm the function has proper access control (`onlyOwner`, time-lock, or operator allowlist). (2) Confirm the downstream withdrawal logic cross-references an on-chain deposits/balances mapping — not just the ZK proof's public inputs. (3) Test degenerate states: call with `rollupSize == 0`, empty merkle root, or zero-length proof. If the verifier accepts and the handler executes without ownership verification, this is a critical `access-control` + `logic-error` finding on virtually any live bug bounty program.

**Hook B — Deprecated / immutable contracts with residual funds**
Continuing from June 20's abandoned-contract pattern: the Aztec `escapeHatch()` shows that an *intended* emergency pathway becomes the attack vector when access controls are missing. When scanning a target's deployment history for old contracts, check specifically for any functions labeled "escape", "emergency", "fallback", or "recovery" in older deployed (but non-paused) versions. These are high-probability live targets if the contract holds >$50K and has no pause authority.

**Hook C — Joomla JCE on governance/docs subdomains**
Check any subdomain your target uses for governance, documentation, or community portals. Quick fingerprint: `curl -s https://<target>/administrator/components/com_jce/` returns 200 on vulnerable installs. Tools like Wappalyzer or BuiltWith can detect Joomla remotely. If JCE < 2.9.99.6 is present on a site within the bug bounty scope, this is a CVSS-10 unauthenticated RCE that can yield admin key material. Verify scope inclusion, then report immediately before weaponizing.
