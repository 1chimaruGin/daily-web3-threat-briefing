## 1. 🔴 Active Exploits (last 24–48h)

### 1.1 Altura — ~$39M | Gold-Vault Closure / Fraudulent DVN Verifier Setup

| Field | Detail |
|---|---|
| **Protocol** | Altura ("The Yield Engine") — gold-backed multi-strategy yield |
| **Chain** | Multi-chain; fund movement routed through Tron |
| **rekt.news post** | July 2, 2026 |
| **$ at risk** | ~$39M (raised/deposited into vault) |
| **Bug classes** | key-management, bridge-dvn, unverified-contract |
| **Status** | Vault closed; depositors locked; funds not returned |

**What happened:** Altura raised $39 million for a gold-backed yield strategy targeting 20% APY, marketed by a team presenting ex-Fidelity and PwC credentials. When rekt.news investigated, the picture was stark:

1. **Biased DVN verifier:** The protocol's cross-chain verification relied on a single DVN (Decentralized Verifier Network) operator — and that operator was a project directly tied to the COO. This created a 1-of-1 conflicted verifier arrangement: the team controlled both the protocol and the entity confirming transactions were valid.

2. **Dashboard that "admitted it verified nothing":** An internal verification dashboard that was publicly visible explicitly acknowledged it was not performing meaningful verification. This was visible to any user who looked.

3. **Fund movement via Tron:** Rather than keeping assets on-chain in verifiable smart contracts, Altura routed depositor funds through Tron — a chain commonly used to obscure fund flows in exit scams due to its high throughput and lower on-chain scrutiny.

4. **Vault closure without withdrawal:** The vault was silently closed and depositors were left waiting for withdrawals, with no public timeline or recovered funds as of the rekt.news post date.

**Root cause class:** This is a **key-management / bridge-DVN conflict-of-interest** failure. No smart-contract bug was needed — the team controlled the verification infrastructure and moved funds directly. The single-DVN configuration is the same structural flaw that contributed to the KelpDAO $292M incident (April 2026, covered in earlier briefings), but here the operator was not a third-party compromised by an attacker — it was the team itself.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 "Vibe-Coded" AI-Generated Solidity: Formally a CVE Driver (CSA Research Note, 2026)

**Published:** Cloud Security Alliance (CSA) Labs research note, 2026

This is the week's most-discussed new finding in the security community. Two CSA research notes formalize what practitioners have observed anecdotally: AI-generated code is now **measurably driving the CVE surge**.

**Key data from CSA Labs:**
- CVEs attributed to AI-generated code: **6 in January 2026 → 15 in February → 35 in March** (roughly doubling each month)
- Empirical study of Fortune 50 enterprises: AI-assisted developers commit at **3–4× the rate** of peers but introduce security findings at **10× the rate**
- A February 2026 academic study of 1,000+ AI-generated Solidity contracts (ChatGPT, Gemini, Copilot) found **frequent severe flaws**: reentrancy, unprotected withdrawal, missing access controls

**The canonical on-chain case study: Moonwell, ~$1.78M (Base + Optimism, February 2026)**

Moonwell's governance proposal MIP-X43 executed a Chainlink OEV wrapper configuration that **co-authored code from Claude Opus 4.6 got wrong**: the cbETH oracle derivation used only the raw `cbETH/ETH` exchange rate instead of `cbETH/ETH × ETH/USD`. This caused cbETH to be priced at **~$1.12 instead of ~$2,200**.

Liquidation bots and opportunistic borrowers immediately exploited the mispricing, draining ~$1.78M in bad debt before the governance process could patch the misconfiguration (on-chain governance has a timelock, so fixing it required a new proposal + voting period). The code commits co-authored by Claude Opus 4.6 were flagged publicly by auditor Pashov as the first documented "vibe-coded" production smart contract hack.

**Why this matters for hunters right now:** Protocols that used AI coding assistants (GitHub Copilot, Claude, Cursor) in 2025–2026 deployments are statistically more likely to carry subtle logic errors in:
- Oracle price derivation (multi-step: exchange rate × base price)
- Interest rate model parameters
- Reward accounting (rounding direction)
- Admin role assignment timing in proxy constructors

This is a **new bug class in practice** — not a new vulnerability type, but a new prevalence driver. Treat "AI co-authored contracts" as a risk multiplier when triaging targets.

## 4. 🎯 Hunt Hooks

### Hook 1: Flag Protocols Using Team-Affiliated or Single DVN Verifiers
*Tied to Altura $39M (§1.1) and KelpDAO $292M (April 2026)*

Two confirmed 2026 incidents — one a team-executed exit ($39M), one a Lazarus Group exploit ($292M) — share the same structural root cause: a **single, unaudited, or team-controlled DVN verifier**.

**What to check in your targets today:**
1. **Identify all cross-chain message/bridge calls.** Look for `ILayerZeroEndpoint.send()`, `IRouterClient.ccipSend()`, or equivalent.
2. **Read the DVN configuration.** For LayerZero V2: call `IMessageLibManager.getConfig(eid, address, ConfigType.ULN)` on the endpoint. Decode the returned `UlnConfig` struct. Check `requiredDVNCount` and `optionalDVNCount`. If `requiredDVNCount == 1` and there's no optional fallback, the protocol is single-DVN.
3. **OSINT the DVN operator.** Is the DVN address a known reputable operator (Google Cloud, Nethermind, LayerZero Labs' default)? Or is it an obscure address? Cross-reference team addresses, treasury multisigs, and deployer wallets.
4. **Check for self-hosted DVNs.** If the DVN is a custom contract, read its `verify()` logic. Does it have meaningful external consensus requirements, or is it a thin wrapper that the team can approve unilaterally?

**Bug class:** bridge-dvn, key-management | **Ceiling:** Critical (bridge TVL at risk)

### Hook 2: Audit Multi-Leg Oracle Compositions in Lending Protocols
*Tied to LendFi $5.2M (§1.2) + Moonwell $1.78M (§2.1)*

Two oracle-manipulation incidents in 2026 share a subtle root cause: **failing to combine both legs of a price derivation**. The Moonwell case (`cbETH/ETH` alone instead of `cbETH/ETH × ETH/USD`) is the clearest example, but the same error appears in:
- LST/LRT pricing (`stETH/ETH` without ETH/USD)
- Bridged/wrapped asset pricing (`WBTC/BTC` without `BTC/USD`)
- LP share pricing (using only one of the pool's two reserve values)

**Concrete check:**
```
# Pseudocode — verify oracle composition in your target's codebase:
# Search for oracle calls near division operators:
#   grep -r "latestRoundData\|getPrice\|getAssetPrice" contracts/
# For each returned price, trace: is it the USD-denominated final price,
# or an intermediate ratio that still needs a second multiplication?

# Specific pattern to flag:
#   price = priceFeed.latestRoundData()  // cbETH/ETH ratio
#   collateralValue = amount * price      // ← MISSING: * ethUsdPrice
#   # This returns value in ETH, not USD — under-prices most collateral
```

**Signal:** Any collateral asset that has a non-USD intermediate oracle (ratio to ETH, BTC, or another asset) and where the protocol accepts it for USD-denominated borrowing is worth examining.

**Bug class:** oracle-manipulation, logic-error | **Ceiling:** Critical

### Hook 3: Independence Day Coverage-Gap Targeting
*Operational awareness — July 4 US holiday*

July 4 is a US federal holiday. Most North America-based DeFi security teams, incident responders, and protocol war rooms are understaffed today. Historically, attackers time complex exploits to periods of reduced monitoring.

**As a hunter, this creates an asymmetric opportunity:**
- Protocols without automated pause mechanisms (circuit breakers, anomaly detection) are more exposed today
- Any exploit staged today will get slower on-chain response from teams
- Front-running protection on governance actions is weaker when fewer people are watching

**What to prioritize today:**
1. Protocols with **no automated pausing** (`pause()` function requires manual governance, not bot-triggered)
2. Protocols that launched in the **last 30 days** on Base, Optimism, or Arbitrum (less established monitoring)
3. Any protocol whose **security contact is a single US-based Discord/Telegram handle** — check if they've been active in the last 12h

**Bug class:** logic-error, access-control | **Ceiling:** Critical (any class of critical finding has higher chance of reaching triage on a holiday)
