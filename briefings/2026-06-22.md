### 1.1 Namada MASP — IBC Transfer Logic Exploit — ~$600K *(catch-up, Jun 19)*

> **Not yet covered in this series. First treatment here.**

| Field | Detail |
|-------|--------|
| **Protocol** | Namada — Multi-Asset Shielded Pool (MASP) |
| **Chain** | Cosmos / Namada L1 |
| **Date** | June 19, 2026 |
| **Loss** | ~$600K (ATOM, USDC, OSMO, TIA, NYM and other IBC assets) |
| **Bug class** | `logic-error`, `bridge-dvn` (IBC cross-chain transfer handling) |
| **Status** | Namada confirmed; root cause under investigation; no full post-mortem yet |

**What happened:**

An attacker swept shielded IBC assets out of Namada's Multi-Asset Shielded Pool, zeroing balances for ATOM, USDC, OSMO, TIA, and NYM. The exploit appears to have targeted a logic flaw in how Namada's MASP processes IBC transfer receipts — the specifics remain under NDA pending formal disclosure. What the investigation *has* confirmed is that the shielded pool's internal ledger did not correctly reconcile IBC-inbound asset entries, allowing the attacker to exit with funds whose provenance the pool could not fully verify.

**The detection failure — stale indexer:**

The exploit went undetected for an extended period because Namada's off-chain indexer had fallen behind the live chain state. Explorers continued displaying non-zero MASP balances while live RPC queries against validator nodes showed zero holdings. The discrepancy surfaced only when independent analysts compared explorer data against live RPC responses.

This is a structural warning for the entire privacy-chain space: **off-chain indexers that cache shielded-pool state are a single point of truth failure**. When the indexer lags — due to node sync issues, validator updates, or deliberate manipulation — the first-line monitoring that protocols and users rely on goes blind.

**Recovery:**

Namada's development team coordinated with Cosmos-ecosystem validators. The multi-asset nature of the drain (many small-cap IBC assets) limits immediate freeze options, unlike EVM chains with Tether/USDC blacklisting. Current recovery prospects are unclear.

### 2.1 Thin-Liquidity VWAP Oracle Poisoning — BlockSec YieldBlox Post-Mortem

BlockSec published a full incident analysis of the **YieldBlox DAO** exploit (February 22, 2026, Stellar / Blend V2, $10.2M drained) that is now widely circulated as the canonical reference for the **thin-liquidity VWAP oracle attack**. Even though the incident pre-dates this series, the analysis is new enough and technically rich enough to warrant a full technique treatment here — any lending protocol on any chain that prices collateral from a DEX-based VWAP oracle is exposed to this class.

**Kill chain (from BlockSec / Halborn / QuillAudits post-mortems):**

```
Target prerequisite: USTRY/USDC on Stellar DEX (SDEX)
  → < $1 in hourly volume; virtually zero market depth

Step 1: Attacker clears thin order book
  → Small sell sweep removes all legitimate limit orders

Step 2: Insert abnormal trade
  → Single large order moves USTRY/USDC price
     from ~$1.06 → ~$106+ (100× nominal value)

Step 3: Reflector oracle pulls VWAP from SDEX
  → Periodic update ingests manipulated price as current VWAP

Step 4: Borrow against inflated collateral
  → Deposit 13,003 USTRY (nominal value ~$13K; oracle value ~$1.38M)
  → Borrow 1,000,196 USDC  (TX 1)
  → Deposit additional 140,000 USTRY
  → Borrow 61,249,278 XLM  (TX 2)

Step 5: Exit — bridge or swap XLM to stable
  → Before Stellar T1 validators could coordinate a freeze
     attacker moved ~$2.7M; validators froze ~$7.5M (48M XLM)
```

**Root cause (BlockSec classification):**

The pool's oracle relied exclusively on Reflector's SDEX VWAP. Reflector is designed for normal liquid markets. The core invariant it assumes — that clearing the order book and inserting a single large trade does not represent "the market price" — was false for USTRY/USDC. There was no:
- Liquidity threshold check before accepting a price update
- Circuit-breaker (max % price move per period)
- Secondary oracle source to cross-reference
- Time-weighted smoothing window long enough to outlast a single block's trade

**The 5-defense pattern (from DEV Community / QuillAudits synthesis):**

| Defense | Implementation |
|---------|---------------|
| 1. Liquidity floor | Reject oracle update if market depth < $X in last N blocks |
| 2. Price deviation cap | Revert if new price deviates > Y% from TWAP-30m |
| 3. Multi-source aggregation | Require ≥ 2 independent oracle sources to agree within Z% |
| 4. Cooldown window | Enforce minimum time between large price changes |
| 5. Collateral liquidity check | Require collateral asset has ≥ $M on-chain liquidity before allowing borrowing against it |

## 4. 🎯 Hunt Hooks

**Hook A — DEX-priced collateral with thin liquidity (VWAP oracle poisoning)**

For any lending protocol that prices collateral assets via a DEX-sourced VWAP:
1. Identify all accepted collateral tokens → for each, check 30-day average DEX volume.
2. **Flag any token with < $100K/day on-chain volume** feeding a VWAP oracle — these are trivially manipulable.
3. Check if there's a liquidity floor or price-deviation circuit-breaker: `grep -rn "liquidityCheck\|minLiquidity\|priceDeviationMax\|twapDelta" contracts/`. Absence = `oracle-manipulation` finding candidate.
4. On Stellar / Cosmos DEX-priced pools specifically, the Reflector oracle pattern (SDEX VWAP, no floor) should be treated as a known-bad configuration until explicitly hardened.

**Hook B — IBC transfer logic in privacy chains / shielded pools**

Namada's MASP exploit points to under-audited territory: IBC integration inside shielded/privacy pools. Similar chains to review: Penumbra, Zcash-bridge integrations, Aztec's planned CCTP bridge. Key questions to ask in any target:
- Does the shielded pool correctly handle IBC `MsgRecvPacket` for every asset type it accepts?
- Is there a 1:1 on-chain proof that an IBC inbound transfer added exactly the right note commitment to the shielded tree?
- What happens if IBC channel times out mid-shielding? Is state rolled back atomically or can notes be double-committed?

**Hook C — Stale indexer vs. live state: a monitoring meta-vulnerability**

The Namada detection failure is a signal to check your own tooling. For any protocol you're watching:
- Compare `eth_call / tendermint_abciQuery` live-node balance queries against what the protocol's own explorer/API reports.
- Any divergence > dust (e.g., explorer shows 100K USDC; live node shows 0) is an incident in progress or a data integrity bug worth reporting.
- For protocols with shielded/privacy components, stale indexers are especially dangerous because the note commitments are opaque — discrepancies can be impossible to detect without direct RPC queries.
