## 1. 🔴 Active Exploits (last 24h)

### 1.1 Cascade — $1.34M CLS Vault Drain (Arbitrum)

**Date:** July 16, 2026  
**Chain:** Arbitrum (funds cross-chained to Solana → Ethereum)  
**Amount:** $1,340,000 USDC  
**Bug class:** `access-control` / `logic-error` (root cause not yet officially disclosed)  
**Status:** Hack confirmed by Cascade team. No trading live yet (protocol was pre-launch). No post-mortem. Funds laundered.  
**Sources:** [CryptoTimes](https://www.cryptotimes.io/2026/07/16/polychain-backed-cascade-hacked-for-1-34m-in-locked-user-funds/) | [CryptoNews.net](https://cryptonews.net/news/security/33158435/) | [Intellectia.ai](https://intellectia.ai/news/crypto/cascade-platform-suffers-13-million-hack)

**Protocol profile:** Cascade is a "neo-brokerage" startup that raised $15M (Polychain Capital + Variant) in December 2025. It aims to offer 24/7 perpetual trading on crypto, commodities, forex, and tokenized pre-IPO equities (OpenAI, SpaceX, Stripe). It had not yet launched publicly — users deposited USDC into the CLS (Concentrated Liquidity Strategy) vault to earn reward points ahead of the go-live date.

**Why this is significant:** Depositors had their funds *locked* in the CLS vault and could not withdraw. The vault was designed to underwrite orderbook depth and liquidations post-launch; those mechanics meant the deposited USDC had no user-accessible exit before trading went live. The attacker exploited this captive-funds structure before launch hardening was complete.

**Fund flow (laundering path):**
1. $1.34M USDC drained from CLS vault on **Arbitrum**
2. Bridged to a fresh Solana address via a cross-chain bridge
3. Routed from Solana back to **Ethereum** via **Relay Protocol**, swapped from USDC into **DAI**
4. The USDC→DAI conversion is deliberate: Circle can blacklist USDC, Maker/DAI cannot — standard post-exploit hardening move seen in multiple 2026 thefts

**Root cause (preliminary):** Not officially disclosed at time of writing. Given the vault held pre-launch locked deposits with no user withdrawal path, candidates include: (a) missing access control on vault withdrawal/drain functions gated only to an authorized role, (b) a logic error in the vault's accounting that didn't track liabilities against the locked pool. No flash loan signature in the tx path. A post-mortem has been promised.

## 2. 🟡 New Disclosed Findings / Techniques

### 2.1 Pre-Launch / Pre-Trading Vault Pattern — High-Value Locked Deposits as Exploit Targets

**Sources:** [CryptoTimes — Cascade](https://www.cryptotimes.io/2026/07/16/polychain-backed-cascade-hacked-for-1-34m-in-locked-user-funds/) | [CryptoNews.net — Cascade warnings](https://cryptonews.net/news/security/33158435/)  
**Bug class:** `access-control` / `logic-error`

The Cascade incident highlights a structural vulnerability class that recurs in pre-launch DeFi protocols: **captive-deposit vaults with asymmetric access**.

The pattern:
- Protocol solicits user deposits pre-launch (points programs, pre-trading rewards, early allocations)
- Deposits are locked with no user withdrawal path until launch
- Launch gating creates a window where: (a) full contract functionality hasn't been stress-tested, (b) access controls on withdrawal/drain paths may be developer-gated but under-hardened, (c) no live monitoring or circuit-breakers are active because the protocol isn't "live yet"
- Result: a concentrated pool of real assets (USDC, ETH) secured by pre-production code

Three prior 2026 cases with similar pre-launch or pre-trading fund structures: Altura (deposited into unverified yield strategy before launch verification), ResupplyFi (pre-liquidity vault), Summer.fi Lazy Summer (NAV manipulation on newly deployed ark). The common thread: **funds accepted before the protocol is launch-hardened**.

**Key questions to ask about any pre-launch vault:**
1. Can any admin/operator key execute a batch withdrawal without a timelock?
2. Is there a `rescue()` or `emergencyWithdraw()` function callable before launch — what's its access control?
3. Does the vault's collateral tracking distinguish "user-claimable" from "protocol-controlled" assets, or does one function drain the entire balance?

## 4. 🎯 Hunt Hooks

### Hook 1: Pre-Launch Locked Vaults — Drain Path Audit
*Tied to §1.1 Cascade $1.34M + §2.1 captive-deposit pattern*

Any protocol with an active points program or pre-launch deposit vault is a live target right now. The Cascade attacker exploited a captive pool before launch hardening.

**What to grep for in any pre-launch protocol scope:**

```solidity
// RED FLAGS — functions that move the full vault balance without timelocks:

// Pattern 1: Emergency/rescue function callable by a single admin key
function emergencyWithdraw(address to, uint256 amount) external onlyOwner {
    IERC20(token).safeTransfer(to, amount);
}
// Ask: is onlyOwner a single EOA or a multisig? Is there a timelock?

// Pattern 2: "sweep" or "rescue" on a pre-launch contract
function sweepTokens(address token, address to) external onlyAdmin {
    IERC20(token).safeTransfer(to, IERC20(token).balanceOf(address(this)));
}
// A single compromised admin key = total drain

// Pattern 3: Batch operations on locked vault without per-user accounting
function executeBatch(bytes[] calldata calls) external onlyKeeper {
    for (uint i = 0; i < calls.length; i++) {
        address(this).call(calls[i]);  // arbitrary self-call without value bounds
    }
}
```

**Specific search steps:**
1. Find any `rescue`, `sweep`, `emergencyWithdraw`, `drain`, `migrate` functions in vault contracts
2. Check their access control: single EOA? untimelocked multisig? any EOA that's also a hot deployer?
3. Check if vault ERC-20 `balanceOf` exceeds any per-user accounting ceiling (if yes, there's a "dark balance" drainable by admin without affecting individual records)
4. Check if the contract is unverified on Arbiscan (Cascade note: users had warned pre-hack that the contract wasn't verified)

**Bug class:** `access-control` | **Ceiling:** Critical

Three Arbitrum perp protocols drained in 7 days means attackers are actively scanning this sector. This is the week to run exhaustive recon on any Arbitrum perp DEX in your scope.

**Targeted checklist (build on yesterday's oracle hook):**

```bash
# 1. Check oracle signer registration — is it a hot EOA or MPC/HSM?

# 2. Check if PriceUpKeep / forwarder can be called by any EOA (not just registered Chainlink keeper)

# 3. Check vault share-price calculations — can a donated asset inflate the share price?
# (Summer.fi / ResupplyFi pattern: ERC-4626 with external totalAssets() that counts
#  un-deposited token balances)
grep -r "totalAssets\|totalSupply\|convertToShares\|previewDeposit" . --include="*.sol" \
  | grep -v "override"
# If totalAssets() calls balanceOf(address(this)) on ANY token, donation attack is possible

# 4. Check for atomic open+update+close in a single multicall or batch
grep -r "multicall\|executeBatch\|aggregate" . --include="*.sol"
# Then check if oracle update AND trade execution can both appear in the same call
```

**Report condition:** Oracle updater + trade settlement callable in the same atomic transaction with no price-deviation limit against a Chainlink reference feed → Critical.

### Hook 3: PETS 2026 Wallet-Extension Research — Address Correlation via RPC Calls
*Tied to §3 wallet-extension tracking study (arXiv July 7, presenting July 20–25)*

A study of 85 browser-extension wallets found that their RPC call patterns, injected `window.ethereum` fingerprints, and cross-site message traffic can de-anonymize users by linking multiple addresses to the same identity — without accessing private keys.

**Bug bounty angle:** Many wallet extension programs include privacy violations in scope (e.g., leaking user address to third-party analytics, de-anonymizing cross-address activity). With the PETS 2026 presentation on July 20–25, PoC techniques will become public. Now is the window to:

1. Check if your target wallet extension sends full address lists to a centralized analytics endpoint on page load
2. Test whether injecting `window.ethereum.request({method: 'eth_accounts'})` from a third-party iframe returns the user's active address without explicit permission (cross-site RPC leakage)
3. Examine the wallet's network requests in DevTools for any beacon/analytics calls that include raw addresses or address fragments

**Bug class:** privacy, `key-management` scope (address-identity linkage) | **Expected severity:** Medium–High depending on program scope
